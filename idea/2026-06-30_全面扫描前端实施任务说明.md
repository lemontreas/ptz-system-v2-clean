## 1. 前端当前要达到的结果

前端应让用户完成下面这条闭环：

1. 先选择或拍摄一张全景图/单图。
2. 在这张图片上绘制外层排查范围和唯一目标区。
3. 选择扫描频段并启动全面扫描。
4. 在同一张任务图片上显示扫描计划、实际完成点、阶段和设备信号。
5. stop 后显示“正在停止”，直到后端进入真正终态。
6. 完整成功轮次结束后可显示当前白名单和未入选原因；白名单精调不阻塞本阶段前端联调。

这样做的原因是：全面扫描现在以任务锁定图片的像素坐标为权威。像素负责范围、路径和展示；云台角度只是后端执行坐标。前端如果继续用旧角度范围或切换图片后复用旧像素，会造成路径与画面错位。

当前可先完成“扫描过程 + 原始证据 + AP/Client 关系”闭环。Todo 26-28 尚未完成时，前端可以暂时隐藏白名单增强功能，或把现有白名单明确标注为当前基线结果，不应因此阻塞实机效果联调。

## 2. 字段字典

> 本节是前端实现时的字段语义基线。字段即使名称相近，也不能互相替代。

### 2.1 启动请求字段

| 字段 | 类型 | 是否必填 | 作用 | 前端注意事项 |
|---|---|---:|---|---|
| `project_name` | string | 否 | 本次全面扫描项目名称，用于任务列表、历史记录和用户识别 | 只作名称，不参与扫描算法 |
| `mode` | `"panorama"` / `"single"` | 是 | 指定本任务使用全景PMAP还是单图标定完成像素转角度 | 图片、范围和结果必须始终使用同一mode |
| `image_url` | string | 否 | 显式指定本任务使用的图片；优先于后端自动选择的最新图片 | 只能传当前mode对应的合法图片；任务启动后不可替换 |
| `wifi_mode` | `"2.4"` / `"5"` / `null` | 否 | 锁定本任务扫描2.4GHz、5GHz或双频配置 | 它只选择频段，不代表设备类型或天线类型；更换频段必须新建任务 |
| `target_ranges` | array | 是 | 用户在任务图片上绘制的真实目标区集合 | 当前数组长度必须严格等于1；不能传多个区域 |
| `target_ranges[].x_range` | `[number, number]` | 是 | 目标区在原图中的水平像素范围，格式为 `[左, 右]` | 使用原图像素，不是CSS坐标；左值必须小于右值 |
| `target_ranges[].y_range` | `[number, number]` | 是 | 目标区在原图中的垂直像素范围，格式为 `[上, 下]` | 浏览器y轴向下递增；上值必须小于下值 |
| `scan_time_limit` | number | 否 | 单个扫描时间窗口的时长，单位分钟 | 不等于预计完成时间；到时可能产生PARTIAL轮次 |
| `time_interval` | number | 否 | 连续扫描时，相邻窗口或轮次之间的等待时间，单位秒 | 等待期间任务仍属于运行生命周期，可被stop |
| `target_distance_m` | number | 否 | 目标距离，用于后端计算摄像头视觉点与RF天线方向的偏差补偿 | 不改变pixel_position；只影响实际云台RF执行位置和白名单判定缓冲 |

禁止前端正式页面发送的内部字段：

| 字段 | 作用 | 禁止原因 |
|---|---|---|
| `test_channel` | 内部设备回归时把全面扫描配置缩减成一个信道HT20 | 仅用于缩短内部测试；不能代表正式多信道扫描结果 |

已经废弃的全面扫描输入字段：

| 字段 | 当前行为 | 前端处理 |
|---|---|---|
| `pan_range` / `tilt_range` | 不再参与全面扫描新路径 | 不发送 |
| `work_ranges` | 旧角度目标区，不再参与新路径 | 不发送 |
| `pan_step` / `tilt_step` | 后端不再接受前端步径控制 | 不发送 |
| `dwell_time` | 三阶段停留时长由后端配置决定 | 不发送，只展示后端返回值 |

### 2.2 Redis外层范围字段

这些字段由前端通过项目现有配置保存接口写入 `gimbal:default_config`，不是全面扫描启动请求体字段。

| 字段 | 类型 | 作用 | 与目标区的区别 |
|---|---|---|---|
| `work_x_range` | `[number, number]` | 当前任务图片上的外层排查范围水平像素边界 | 它是用于观察目标区周围环境的大范围，不是目标区 |
| `work_y_range` | `[number, number]` | 当前任务图片上的外层排查范围垂直像素边界 | 它必须完整包含唯一目标区 |

前端注意：

- 字段名中的 `work` 是历史命名，不应把它显示成“真实目标区”。
- 用户切换图片或mode后，旧值仍可能留在Redis；前端必须要求重新确认。
- 后端会校验范围是否超出当前图片尺寸，但前端也应先行校验并给出可理解提示。

### 2.3 图像上下文字段

图像上下文由后端解析并锁定。前端通常只需保存能标识该上下文的字段，不需要自行构造PMAP或相机内参。

| 字段 | 作用 | 前端用途 |
|---|---|---|
| `mode` | 标识全景或单图转换链路 | 决定使用哪张底图及如何组织页面 |
| `image_url` | 任务锁定图片地址 | 作为扫描计划和结果叠加的唯一底图 |
| `width` / `height` | 原图像素尺寸 | 把显示坐标换算回原图像素，并校验范围 |
| `pmap_path` | 后端全景PMAP文件路径 | 仅供后端转换和测试诊断；前端不读取本地路径 |
| `session_json` | 后端全景拍摄会话元数据路径 | 仅供后端查找来源图内参和拍摄角度 |
| `capture_pan` / `capture_tilt` | 单图拍摄时的云台角度 | 可在调试详情展示，不应当作所有像素的角度 |
| `intrinsics` | 单图相机内参 `fx/fy/cx/cy` | 由后端使用；前端不得自行按这些字段重复实现角度转换 |

### 2.4 启动响应及扫描计划字段

| 字段 | 作用 | 前端正确用法 |
|---|---|---|
| `queued` | 表示启动请求已进入后端队列 | 只能显示“已受理/等待执行”，不能显示“正在扫描第一个点” |
| `config.project_id` | 当前项目唯一标识 | 用于历史、结果和任务详情关联 |
| `config.scan_plan` | 本次扫描的逻辑点位计划 | 用于启动前后预览和预计工作量，不代表实际完成点 |
| `config.estimated_points` | 计划逻辑点总数 | 只作估算；PMAP空洞、姿态去重、移动失败或stop会导致实际数量不同 |
| `config.estimated_time_minutes` | 根据点数、配置数量和dwell估算的分钟数 | 不能用作倒计时承诺 |
| `config.dwell_time_source` | 停留时长来源，当前通常为 `config` | 提示用户参数由设备端控制 |
| `config.dwell_times` | 粗扫、细扫、偏差区每配置停留时长 | 只展示，不允许前端回传修改 |
| `config.internal_test` | 内部单信道测试标记 | 正式前端不应出现；内部工具必须醒目标注“非正式结果” |
| `warning` | 后端针对时长或测试模式给出的提示 | 原样或等价展示给操作者 |

`scan_plan[]` 子项：

| 字段 | 作用 | 前端正确用法 |
|---|---|---|
| `phase` | 计划所属阶段：`coarse` / `fine` / `deviation_a` | 用于分组和图例，不用于判断任务是否结束 |
| `coordinate_space` | 当前计划坐标空间，全面扫描新链路为 `pixel` | 只有为pixel时才叠加到任务图片 |
| `x_range` / `y_range` | 该阶段覆盖的像素范围 | 用于画计划范围框 |
| `core_point_count` | 粗扫或细扫内部格心数量 | 只表示计划核心点 |
| `outer_probe_point_count` | 粗扫外扩探测点数量 | 与内部格心分开显示，当前通常为8 |
| `layers` | 偏差区层数 | 当前为3层 |
| `points_per_layer` | 偏差区每层计划点数 | 当前为每层8点 |
| `point_count` | 当前阶段计划总点数 | 不等于实际完成数 |
| `config_source` | 偏差区配置来源说明 | `target_seen_macs` 表示只使用目标区出现MAC对应配置 |

### 2.5 状态字段

全面扫描生命周期必须读取 `status.full_scan`。

| 字段 | 作用 | 是否可判断终态 |
|---|---|---:|
| `status.state` | 顶层PTZ运动状态 | 否 |
| `status.full_scan.state` | 全面扫描生命周期：`queued/running/stopping/stopped/done/error` | 是，需结合terminal |
| `status.full_scan.terminal` | worker是否已完成清理并进入真正终态 | 是，最关键字段 |
| `status.full_scan.active` | 前端运行展示兼容标记；stop 接收后立即为 `false` | 可用于立即结束“运行中”展示，但不能作为终态或新任务启动依据 |
| `status.full_scan.phase` | 当前执行阶段 | 否，只作进度展示 |
| `status.full_scan.scan_id` | 当前任务唯一ID | 用于关联日志和防止旧任务结果串入，不由前端生成 |
| `status.full_scan.stop_requested` | 后端是否已收到停止请求 | 只用于显示“正在停止” |
| `status.full_scan.reason` | 结束、停止或错误原因 | 用于结果提示和诊断 |
| `status.full_scan.current_point` | 当前处理进度位置 | 用于进度显示，不保证每次轮询连续递增 |
| `status.full_scan.total_points` | worker当前确认的总点数 | 偏差区是否执行取决于候选配置，因此可能在运行中更新 |
| `status.full_scan.round` / `round_id` | 当前轮次序号和轮次标识 | 用于区分连续扫描中的不同轮次 |
| `status.full_scan.remaining_seconds` | 当前时间窗口剩余秒数 | 为窗口剩余时间，不是整个任务完成倒计时 |

正确终态条件：

```text
full_scan.terminal == true
且 full_scan.state 属于 stopped / done / error
```

### 2.6 实时点位结果字段

| 字段 | 作用 | 前端正确用法 |
|---|---|---|
| `round_index` / `round_id` | 点位所属扫描轮次 | 防止跨轮次混画 |
| `phase` | 点位实际完成阶段 | 用于图例和筛选，不判断生命周期 |
| `position.pan/tilt` | 逻辑像素经后端权威映射得到的视觉执行角度 | 详情和诊断使用；不能由前端再次映射成像素 |
| `pixel_position.x/y` | 点位在任务锁定图片中的权威逻辑像素 | 前端画点必须使用它 |
| `rf_position` | 天线偏差补偿后的实际RF云台位置 | 只作调试；不要替代pixel_position |
| `point_type` | 粗扫点类型：`outer_probe` / `grid` | 区分外扩探测点与内部格心 |
| `layer` | 偏差区层号1/2/3 | 用不同颜色或透明度表示外扩层 |
| `side` | 偏差点所属边：left/right/top/bottom | 用于调试点位分配 |
| `narrow_side` | 该偏差点是否来自窄边轮换策略 | 布尔值，仅作路径解释 |
| `point_started_at` / `point_finished_at` | 点位采集开始和完成时间 | 用于耗时和时序展示 |
| `scan_config_count` | 该点实际使用的信道配置数量 | 内部单信道测试为1；正式任务通常更多 |
| `mac_count` | 当前点位结果中的记录数量 | 可能包含marker，真实设备数必须先排除synthetic记录 |
| `macs` | 当前点位观察到的MAC证据字典 | 设备详情和RSSI展示的数据来源 |

`macs[mac]` 常用字段：

| 字段 | 作用 | 前端注意事项 |
|---|---|---|
| `rssi_avg` | 当前点位该MAC的平均RSSI | 不同采样数量下不能只凭微小差值下绝对结论 |
| `rssi_samples` / `sample_count` | RSSI样本数量 | 用于解释数据可信度 |
| `channel` | 当前证据采用的信道 | 必须结合 `channel_source/channel_confidence` 解释，不能只看数字 |
| `bandwidth` | 扫描接收带宽 | 当前通常为HT20，不等于AP宣告带宽 |
| `declared_bandwidth` | AP Beacon 中可解析出的 HT/VHT 宣告带宽 | 允许为null；EHT-only或缺少对应Operation IE时不得伪造 |
| `channel_source` | 信道来源 | AP常见为`ds_parameter_set`、`ht_operation`或`beacon_capture_config` |
| `channel_confidence` | 当前配置证据等级 | AP为`declared/inferred`；不要与Client连接关系等级混用 |
| `type` | `ap` / `client` / `null` | null表示证据不足，不应强行归类 |
| `subtype` | 当前可能为 `starlink` / `null` | starlink是AP子类型，不是独立type |
| `ssid` | AP名称 | Beacon 明确隐藏时为保留值 `_wildcard_`；未知或非 AP 为 `null` |
| `synthetic` | 是否为后端生成的非设备记录 | true时不得加入设备列表 |
| `role` | 记录角色；marker为 `scan_point_marker` | marker只表示点位实际被扫描 |
| `evidence_role` | `fixed_point` / `path` | 固定点与移动路径使用不同图例；路径证据不参与正式白名单或普通最强位置 |
| `sampling_complete` | 本条证据是否完整采样 | false必须显示为部分数据，不能伪装成正式完成点 |
| `completion_reason` | 完成或部分结束原因 | 用于解释manual_stop、time_limit及配置优先部分采样 |
| `observed_best_config` | 当前任务观察到的最佳配置 | 主要用于Client配置说明，不等同于AP宣告配置 |
| `current_observed_ap` | Client本轮最近观察到通信的AP对象 | 读取`current_observed_ap.bssid`；它是最近观察事实，不是路由器实时在线状态 |
| `observed_aps` | Client本轮与各AP的Data关系对象数组 | 保留BSSID、首末时间、上下行帧数等本轮历史 |
| `relationship_status` | `confirmed/inferred/uncertain` | 只解释Client–AP关系；是否显示“已观察到连接”以此字段和Data证据为准 |

### 2.7 轮次结果字段

| 字段 | 作用 | 前端正确用法 |
|---|---|---|
| `round_status` | `RUNNING` / `SUCCESS` / `PARTIAL` | 只有SUCCESS轮次生成正式白名单 |

| 字段 | 作用 | 前端正确用法 |
|---|---|---|

  - `panorama`
  - `single`
- 支持选择当前图片；未显式选择时，可以让后端按模式使用最新图片。
- 显式选择图片时向启动接口传 `image_url`。
- 增加频段选择：
  - `wifi_mode="2.4"`
  - `wifi_mode="5"`
  - 不传、`null` 或空字符串表示双频。
- 不再发送旧字段：
  - `pan_range`
  - `tilt_range`
  - `work_ranges`
  - `pan_step`
  - `tilt_step`
  - `dwell_time`

为什么：

- `mode` 决定像素使用全景 PMAP 还是单图标定参数转换。
- `wifi_mode` 在任务开始时锁定，避免任务过程中更换频段导致证据不可比较。
- 旧角度范围已退出全面扫描新路径，继续发送会制造“前端以为生效、后端实际忽略”的假配置。

### 3.2 按图片像素保存外层范围

前端要做：

- 用户在当前图片上绘制外层排查范围。
- 通过项目现有的云台默认配置保存接口，把范围写为：
  - `work_x_range: [x_min, x_max]`
  - `work_y_range: [y_min, y_max]`
- 图片或模式变化后，必须要求用户重新确认或重新绘制外层范围。
- 发送前将范围限制在当前图片尺寸内。
- 不要把 CSS 显示坐标直接提交；必须先换算回图片原始像素。

为什么：

- 后端只从 `gimbal:default_config` 读取这两个外层像素字段。
- 浏览器中的缩放、留白、响应式布局会改变显示坐标，但不会改变原图坐标。
- 切换图片后沿用旧范围会把同一组数字解释到另一套像素坐标系中。

### 3.3 只提交一个目标区

前端要做：

- 启动请求使用：

```json
{
  "mode": "panorama",
  "image_url": "/api/v1/camera/images/example.jpg",
  "wifi_mode": "5",
  "target_ranges": [
    {
      "x_range": [700, 1200],
      "y_range": [300, 500]
    }
  ]
}
```

- `target_ranges` 必须是数组，但当前长度必须严格等于 `1`。
- 目标区必须完全位于 `work_x_range/work_y_range` 内。
- 禁止提交空区域、倒序范围、负数、NaN、Infinity或超出图片尺寸的值。
- 用户尝试绘制第二个目标区时，前端直接阻止，并提示“当前版本仅支持单目标区”。

为什么：

- 数组结构为未来接口扩展保留，但当前后端只完成单目标区闭环。
- 若前端提前开放多目标区，会立即引入点位共享、区域归属、白名单择优和复核合并等尚未实现的语义。

### 3.4 锁定任务图片

前端要做：

- 启动成功后保存本次任务使用的：
  - `mode`
  - `image_url`
  - 图片宽高
  - 外层范围
  - 目标区
  - `scan_id`
- 任务运行期间，即使用户拍了新图，也继续在原任务图片上显示本任务结果。
- 如果用户切换到新图片，应视为查看操作，不得把运行中任务的像素结果直接叠加到新图片。
- 新图片、新范围或新频段必须启动新任务。

为什么：

- 后端在任务开始时锁定 `Full-scan image context`。
- 运行途中切换坐标底图，会让同一个 `pixel_position` 指向错误位置。

### 3.5 显示像素扫描计划

前端要做：

- 读取启动响应中的 `scan_plan`。
- 当 `coordinate_space="pixel"` 时，按像素范围展示：
  - `coarse`：12-24 个内部格心 + 8 个外扩点；
  - `fine`：20-26 个目标区格心；
  - `deviation_a`：3 层 × 8 点的计划值。
- `scan_plan` 是逻辑计划，不是最终完成结果。
- 不要因为计划点数存在，就提前在结果图上标成“已扫描”。

为什么：

- PMAP 空洞、硬件限位、相同执行姿态去重、移动失败或 stop 都可能减少实际完成点数。
- 计划和实际结果混在一起，会让用户误以为未执行点已经有采样证据。

### 3.6 在图片上叠加实际扫描点

前端要做：

- 轮询全面扫描实时结果。
- 每个实际完成点读取：
  - `pixel_position.x/y`：画面展示位置；
  - `position.pan/tilt`：后端执行角度，仅用于详情和调试；
  - `phase`：`coarse` / `fine` / `deviation_a`；
  - `point_type`：粗扫的 `outer_probe` / `grid`；
  - 偏差点的 `layer` / `side` / `narrow_side`。
- 绘制时始终使用 `pixel_position`，不要再把 `position.pan/tilt` 线性映射回图片。
- 建议用不同样式区分：
  - 粗扫外扩点；
  - 粗扫内部格心；
  - 细扫格心；
  - 偏差区不同层。

为什么：

- `pixel_position` 是任务图片上的权威逻辑位置。
- 全景像素与云台角度不是简单线性关系，前端自行换算会绕过 PMAP并产生错位。

### 3.7 正确处理点位 marker

前端要做：

- 当 MAC 记录满足以下任一条件时，把它当作扫描点 marker，而不是设备：
  - `synthetic=true`
  - `role="scan_point_marker"`
  - MAC 为 `ff:ff:ff:ff:ff:ff`
- marker 可用于显示“这个点实际扫过但未发现设备”。
- marker 不进入设备列表、MAC数量、白名单候选和设备详情。

为什么：

- marker 是路径可视化证据，不是真实WiFi设备。
- 如果把marker当设备，会污染设备数量、筛选和用户判断。

### 3.8 保持两阶段停止交互

前端要做：

- stop接口返回成功后显示“正在停止”。
- 禁用重复stop和新的start。
- 继续轮询 `/api/v1/ptz/status`。
- 只有满足以下条件才显示真正结束：

```text
status.full_scan.terminal == true
且 state 为 stopped / done / error
```

- 不要为了停止全面扫描内部抓包而额外调用 `/api/v1/capture/stop`。
- 不要用顶层 `status.state="IDLE"` 判断扫描已结束。

为什么：

- stop返回只代表请求已接收，worker仍需停止移动、取消抓包并清理任务。
- 额外调用独立抓包stop可能干扰其他抓包生命周期。

### 3.9 白名单和拒绝原因展示

前端要做：

- 本阶段可作为可选模块，不阻塞扫描过程、证据点位和AP/Client关系联调。
- 仅在完整成功轮次后读取白名单。
- 展示：
  - 已接受MAC；
  - `accept_reasons` / `accept_reason_codes`；
  - `rejected_macs`；
  - `failed_reasons` / `reason_codes` / `metrics`。
- 明确区分：
  - 用户绘制的真实目标区；
  - 后端白名单判定缓冲区。
- 不要把判定缓冲区画成更大的目标区。

为什么：

- 缓冲区只用于目标边缘的证据容差，不改变用户选择范围或扫描路径。
- 前端若把它显示为目标区，会让用户误以为后端擅自扩大了扫描范围。

## 5. P2：Todo 25 已完成，现在接入

AP/Client 主链路已使用真实设备完成定向验证。前端要做：

- AP展示：
  - `channel`
  - `bandwidth`（扫描带宽，当前为HT20）
  - `declared_bandwidth`
  - `channel_source`
  - `channel_confidence`
- Client展示：
  - `observed_best_config`
  - `current_observed_ap`
  - `observed_aps`
  - `relationship_status`
- 只有Data帧关系才能显示“观察到与该AP通信”。
- Probe、管理帧或同信道只能显示为发现线索，不能显示为已连接。
- `current_observed_ap` 是对象，不是单独字符串；显示和比较BSSID时读取其 `bssid`。
- `observed_aps` 是本次 `scan_id` 内的关系历史：一个AP可对应多个Client，一个Client也可先后观察到多个AP。
- 同一Client在本轮从AP-A切换到AP-B后，`current_observed_ap` 取最近Data帧对应的AP，旧AP继续保留在`observed_aps`。
- AP反查Client时，前端可按 `observed_aps[].bssid` 聚合；当前后端不额外输出AP侧 `connected_clients`。
- `relationship_status` 解释连接关系可信度；AP自身的 `channel_source/channel_confidence` 解释信道来源，两组语义不得混用。
- 当AP为 `channel_source="beacon_capture_config"` 时，表示AP身份可由Beacon确认，但信道值仍是按捕获配置推断。
- `current_observed_ap` 只表示本轮最近观察事实，没有Data更新时不能当作实时在线状态。
- iPhone等设备可能按SSID使用不同私有MAC；MAC变化后必须作为新的Client身份展示，不能自行合并为同一物理设备。
- 前端按 MAC+BSSID 持久化历史关系；后端只返回当前任务观察事实。

数据读取边界：

- 产品前端统一通过全面扫描HTTP状态/结果接口读取，不直接访问Redis内部key。
- `full_scan:{scan_id}:relationships` 是后端进程间快照与设备测试证据，不是前端正式数据接口。
- 前端必须按自己持有的 `scan_id/round_id` 展示，不能把不同任务的关系混画。

为什么：

- AP宣告带宽与扫描接收带宽不是一个概念。
- 客户端可能只在某信道被观察到，但没有足够证据证明它与某AP通信。
- 历史关系属于前端产品数据，不应被当前任务短暂观察覆盖。
