# 全面扫描单信道内部测试说明

## 目的

全面扫描正式任务需要遍历单频 13 个或双频 26 个 HT20 配置，设备回归耗时较长。内部单信道测试模式只把任务配置缩减为一个信道，点位生成、云台移动、粗扫、细扫、偏差区、抓包通知、Redis结果、stop和 `scan_id` 语义仍按正常流程执行。

该模式适合快速检查路径与状态，不可作为正式多信道扫描、设备发现完整性或白名单最终效果的验收依据。

## 安全开关

默认关闭：

```json
"全面扫描": {
  "允许单信道内部测试": false
}
```

设备测试时推荐临时使用环境变量：

```bash
export FULL_SCAN_ALLOW_SINGLE_CHANNEL_TEST=1
```

服务需要在该环境变量下重新启动。测试完成后取消环境变量并恢复正常启动。

未开启安全开关时，携带 `test_channel` 的请求返回 `403`，不会启动任务。

## 请求方式

在正常全面扫描请求中额外加入内部字段：

```json
{
  "mode": "panorama",
  "wifi_mode": "5",
  "test_channel": 157,
  "target_ranges": [
    {
      "x_range": [700, 1200],
      "y_range": [300, 500]
    }
  ]
}
```

说明：

- `test_channel` 只接受项目正式频段列表中的信道。
- 带宽固定为 `HT20`。
- `test_channel` 存在时，本任务只使用该单个配置。
- `wifi_mode` 此时不决定配置列表；后端会按 `test_channel` 自动识别为 2.4GHz 或 5GHz任务。
- API返回 `config.internal_test`，并明确提示结果不是正式多信道结论。
- 前端产品页面不得暴露该参数。

支持信道：

```text
2.4GHz：1-13
5GHz：36、40、44、48、52、56、60、64、149、153、157、161、165
```

## 验证重点

- scan plan和实际点位数量与正常任务一致。
- capture worker每个点位只收到一个 `{channel, bandwidth="HT20"}` 配置。
- 粗扫、细扫、偏差区均保持同一测试信道。
- stop仍经过 `running -> stopping -> stopped`。
- 终态仍为 `terminal=true`，并清理 `active_scan_id`。
- 下一次未携带 `test_channel` 的正式任务恢复13或26个配置。

## 禁止事项

- 不把单信道结果当作“未发现其他信道设备”的依据。
- 不用它验收正式白名单召回率。
- 不在前端正式功能中增加单信道选择。
- 不把配置文件默认值改成 `true` 后长期保留。
- 不为缩短测试而跳过点位、阶段、stop或结果写入。
