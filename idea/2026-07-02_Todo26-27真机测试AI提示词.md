## 三、Todo 27：白名单真实配置隔离

对每个候选与白名单 MAC 做逐项核对：

1. `observed_configs` 中每项必须代表网卡实际抓包配置，包含 `channel/bandwidth/rssi_avg/rssi_samples`。
2. 每个白名单 MAC 只能有一个 `decision_config`；白名单 `channel/bandwidth` 必须与该配置一致。
3. `evidence_counts.included_fixed_points` 只能来自：

```text
evidence_role=fixed_point
sampling_complete=true
实际配置等于 decision_config
```

4. 路径、marker、partial、其他配置必须进入对应排除计数，不能增加白名单硬命中数。
5. AP：
   - 有宣告主信道时，只能使用该信道的实抓证据。
   - 邻道更强也不能回退到邻道。
   - 无宣告信道时，只能从真实抓到 Beacon 的配置中推断。
   - `bandwidth` 必须保持实际扫描带宽 `HT20`；HT40/VHT80 只能出现在 `declared_bandwidth`。
6. Client：
   - 关系配置有该 Client 实抓证据时，使用关系配置。
   - 关系配置没有实抓证据时，白名单不能展示幽灵配置；应回退到该 Client 本轮实抓峰值最强配置。
   - 回退时应看到 `channel_source=observed_best_config_fallback`，但 `relationship_status/current_observed_ap` 不应被改成 uncertain。
7. 使用同一轮 Redis 原始结果运行 `rebuild_full_scan_whitelist.py --print`（先只打印，不覆盖）；在线与离线结果的成员、配置、reason 和计数必须一致。
8. 选至少一个最终白名单 MAC，用白名单返回的真实 `channel/bandwidth` 再启动定点抓包，验证能够再次捕获该 MAC。记录包数和 RSSI；这一步是“配置真实可用”的关键验收。

若现场没有同时满足 AP 宣告、AP 推断、Client 关系回退的样本，明确标记未覆盖，不得以源码测试替代。

### 4.1 状态连续性

验证基础白名单生成后：

```text
full_scan.active=true
full_scan.state=running
full_scan.terminal=false
full_scan.phase=whitelist_refinement
```

主扫描到复核之间不得出现任何一次 `active=false` 或 `terminal=true`。顶层 PTZ `state` 或 `capture.running` 不能替代 `full_scan.active/terminal`。

### 4.2 路线与采集

对至少一个具备安全矩形的白名单 MAC 验证：

1. 矩形来源只包含初始细扫、完整固定点、该 MAC 的最终真实配置。
2. 路线覆盖四边与两条对角线，共 7 段，只重复角度成本最低的一条边。
3. 起点转场标为 `approach`，不参与候选。
4. 四角不做固定 dwell。
5. 每条正式线段移动前都有 worker 确认的唯一 `segment_id`。
6. 正式线段不得使用 `3°` 中继；存在 `(0.2°,1°)` 不可靠轴向变化时应扩大矩形，仍不安全则跳过该 MAC。
7. 每个 MAC 只使用自己矩形内的证据；同配置其他 MAC 不得跨矩形获胜。
8. 线段少于 2 包不得成为候选。
9. 每条原始边/对角线仍只执行一次连续直达移动；不得为像素定位增加物理中间路标、固定停顿或额外 dwell。
10. capture worker 在线段结束摘要中返回峰值包时间；PTZ 侧只在线段内存中保存轨迹，不得出现逐包或逐轨迹点 Redis 流。
11. Pan 路线成本与安全跨度必须使用 `0°/360°` 最短环绕差；跨零度的短边不得被误判成接近 360° 的大移动。

### 4.3 替换语义

分别寻找或重复运行以覆盖：

```text
复核峰值 < 原 best_rssi + 1dB  -> 保留原位置
复核峰值 >= 原 best_rssi + 1dB -> 更新位置
```

成功替换时检查：

```text
initial_best_position / initial_best_pixel_position / initial_best_rssi 保留
best_position / best_pixel_position / best_rssi 为最终值
best_position_source=whitelist_refinement_peak_trajectory
refinement.winner.rssi_samples >= 2
refinement.winner.peak_packet_time 存在
refinement.winner.peak_angle.success=true
refinement.winner.pixel_lookup.success=true
refinement.winner.trajectory_sample_count >= 1
```

逐项核验获胜位置：

1. `winner.pixel_position` 与顶层 `best_pixel_position` 一致。
2. 使用 `winner.start_pixel/end_pixel` 验证最终像素位于该原始线段上；允许浮点误差，不得落到全景重叠区的另一条无关像素位置。
3. 将最终 `best_pixel_position` 通过任务锁定图片的像素→角度接口正向转换，结果应与 `best_position` 一致；不要使用普通全图角度→像素反查作为验收依据。
4. `winner.peak_angle.method` 应为 `trajectory_interpolation`、`clamped_start` 或 `clamped_end`；峰值早于首样本/晚于末样本时只能落到对应端点。
5. 记录 `trajectory_sample_count`、`trajectory_max_gap_ms`、`trajectory_max_query_duration_ms`、`peak_callback_delay_ms`。首轮只观察，不设置角度残差硬阈值。
6. 同一最高 RSSI 多次出现时，首版应保留第一个最高包时间；只有严格更高 RSSI 才更新峰值时间。
7. 若原像素线局部存在 PMAP 空洞，应跳过无效候选并继续搜索整线；整线无有效候选时保持初始位置并记录 `no_valid_segment_pixel`。

复核只能改变展示位置和 RSSI，不能增加或删除白名单成员。

### 4.4 Redis 与进程清理

复核结束后检查：

```text
无遗留 active refinement session
无持续增长的临时 notify/segment key
capture_worker 未异常退出或重启
无 EBUSY、Traceback、旧 scan_id 复活
```
