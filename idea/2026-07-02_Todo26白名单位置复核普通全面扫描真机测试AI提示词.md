## 六、完整状态时间线

从启动前开始，以 100–300ms 周期采集：

```text
GET /api/v1/ptz/status
Redis GET ptz:current_status
Redis GET full_scan:active_scan_id
Redis GET full_scan:latest_round
```

按时间保存，不要只记录最终状态。必须证明：

1. 启动后为 `queued/running`，`active=true`、`terminal=false`。
2. 主扫描 phase 正常推进，没有异常跳回或状态复活。
3. 基础白名单生成后进入：

```text
full_scan.active=true
full_scan.state=running
full_scan.terminal=false
full_scan.phase=whitelist_refinement
```

4. 主扫描到复核之间不得短暂出现 `active=false` 或 `terminal=true`。
5. 复核完成后正常进入：

```text
full_scan.active=false
full_scan.state=done
full_scan.terminal=true
full_scan.phase=completed
```

6. 终态后继续观察至少 5 秒，确认不复活、不回到 running。
7. `full_scan:active_scan_id` 正确清理，capture session 和 notify key 无遗留。

## 七、保存基础白名单快照

首次观察到 `phase=whitelist_refinement` 时，立即保存当时的：

```text
full_scan:round_<N>_results
full_scan:whitelist:round_<N>
full_scan:whitelist:latest_success
```

该快照作为复核前基础白名单。任务终态后再次读取相同键，比较：

- `mac_whitelist` 成员集合完全相同；
- 复核不得新增或删除白名单成员；
- `initial_best_position/initial_best_pixel_position/initial_best_rssi` 保留；
- 只允许符合条件的成员更新展示位置与 RSSI；
- 被跳过或失败的成员仍保留初始位置。

如果没有在复核开始时成功保存基础快照，成员不变验证证据不足，必须在报告中说明。

## 八、路线与采集验证

对至少一个完整复核 MAC，核对：

1. 路线仍覆盖矩形四边和两条对角线，共 7 段，只重复一条最低成本边。
2. 起点 approach 不参与候选。
3. 每条正式线段移动前有唯一 `segment_id`，worker 已确认后才开始移动。
4. 每条原始边/对角线只执行一次连续直达移动。
5. 不得因像素定位新增物理中间路标、额外固定停顿或 dwell。
6. 正式线段不使用 `3°` 中继。
7. 同一配置其他 MAC 的包不得跨矩形参与当前 MAC 候选。
8. 候选线段必须 `rssi_samples >= 2`。
9. 单包峰值只在 RSSI 严格变强时替换；并列最高 RSSI 保留第一个包时间。
10. Pan 路线成本和安全跨度使用最短环绕差，不能把跨 `0°/360°` 的短移动
    误算为接近 360°。

同时检查日志中没有新增频繁停顿、重复移动、额外信道切换或 Redis 轨迹刷写。

## 九、峰值轨迹与最终像素验收

最终至少覆盖一个：

```text
refinement.status = "updated"
best_position_source = "whitelist_refinement_peak_trajectory"
refinement.winner.rssi_samples >= 2
refinement.improvement_db >= 1.0
```

对获胜记录逐字段核对：

```text
winner.peak_packet_time
winner.peak_callback_time
winner.peak_callback_delay_ms
winner.peak_angle
winner.trajectory_sample_count
winner.trajectory_max_gap_ms
winner.trajectory_max_query_duration_ms
winner.pixel_lookup
winner.start_pixel
winner.end_pixel
winner.pixel_position
```

验收规则：

1. `peak_angle.success=true`。
2. `peak_angle.method` 只能是：

```text
trajectory_interpolation
clamped_start
clamped_end
```

3. 时间位于两个轨迹样本之间时必须是 `trajectory_interpolation`。
4. 峰值时间早于首样本时只能钳制到起点；晚于末样本时只能钳制到终点。
5. `winner.pixel_lookup.success=true`。
6. `winner.pixel_position` 与最终 `best_pixel_position` 一致。
7. `pixel_lookup.mapped_angle` 与最终 `best_position` 一致（仅允许输出取整误差）。
8. 最终像素必须位于 `start_pixel → end_pixel` 原始像素线段上。

使用以下几何检查，不要只凭肉眼判断：

```text
cross = (px-x0)*(y1-y0) - (py-y0)*(x1-x0)
dot   = (px-x0)*(x1-x0) + (py-y0)*(y1-y0)
len2  = (x1-x0)^2 + (y1-y0)^2
```

要求：

```text
abs(cross) <= 浮点容差
0 <= dot <= len2
```

9. 将最终 `best_pixel_position` 通过任务锁定图片的像素→角度正向转换，结果应
   与 `best_position` 一致。禁止用无约束全图角度→像素反查证明正确。
10. 输出中不得再出现：

```text
best_position_source=whitelist_refinement_segment_midpoint
```

11. 若像素线局部存在 PMAP 空洞，应跳过无效候选并继续查找；整线无有效候选时
    保留初始位置并记录 `no_valid_segment_pixel`。
12. 本轮只记录轨迹样本数、最大间隔和查询耗时，不自行增加角度残差门槛。

如果有完整 winner 但没有任何 `updated`（全部低于 `+1dB`），可以证明轨迹计算
链路运行，但不能完整证明最终替换分支；结论应为“部分通过/信息不足”，并建议
更换目标 MAC 后重测，不得降低 `+1dB` 阈值。

## 十二、成功、失败和信息不足

只有同时满足以下条件才能报告“通过”：

- 普通全面扫描完整跑通并进入白名单复核尾阶段；
- 至少一个 MAC 完成七段路线且至少一个成员 `status=updated`；
- 最终像素在线段上，正向转换角度与 `best_position` 一致；
- 峰值时间、轨迹插值/钳制和像素查找字段完整；
- 白名单成员集合不变；
- 主扫描到复核到终态的状态连续正确；
- API、Redis、scan_id、日志和旧流程无回归；
- 测试环境与临时配置已恢复。

以下任一情况判定“失败”：

- 仍使用整段像素中点作为最终位置；
- 最终像素不在获胜原线段上；
- 使用无约束角度→像素反查后跳到其他重叠像素；
- 峰值时间或轨迹字段缺失、时间匹配明显错误；
- 新增物理路标、停顿或 Redis 逐轨迹写入；
- 复核改变白名单成员；
- 主扫描与复核之间短暂终态、最终状态卡住或复活；
- stop、scan_id、capture session、API 或日志出现回归；
- 需要修改代码才能继续测试。

出现以下情况报告“信息不足/部分通过”，不是通过：

- 没有白名单；
- 没有安全矩形；
- 没有线段达到 2 包；
- 没有任何 `updated` 分支；
- 未保存基础白名单快照；
- 无法确认任务图片或最终像素在线性。
