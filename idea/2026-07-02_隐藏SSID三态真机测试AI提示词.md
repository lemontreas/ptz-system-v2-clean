## 二、本次契约

必须验证以下三态，不能只检查字段存在：

| 场景 | 预期 `type` | 预期 `ssid` |
|---|---|---|
| Beacon 的 SSID IE 长度为 0 | `ap` | `"_wildcard_"` |
| Beacon 的 SSID IE 内容全为 `0x00` | `ap` | `"_wildcard_"` |
| Beacon 含正常可解析名称 | `ap` | 原名称 |
| 已确认 AP，但 SSID IE 缺失或内容无法解析 | `ap` | `null` |

额外约束：

- 不新增字段。
- `subtype` 不承载隐藏 SSID；普通隐藏 AP 通常仍为 `null`，隐藏星链仍按原星链规则为 `"starlink"`。
- 任何明确隐藏的 AP 都不得继续输出空字符串。
- `null` 只表示未知或无法确定，不能被前端当作隐藏 SSID。

## 四、设备端纯逻辑三态验证

在设备实际 Python/Scapy 环境中验证以下分支：

1. `b""` → `"_wildcard_"`
2. `b"\x00" * 4` → `"_wildcard_"`
3. 正常 ASCII SSID → 原名称
4. 正常 UTF-8 中文 SSID → 原名称
5. 不可解析字节 → `None`
6. Beacon 完全缺少 SSID IE → `_extract_beacon_info(...).ssid is None`

优先运行现有专项单元测试；如测试文件未部署，可在不写入项目文件的临时命令中构造 Scapy Beacon。
构造包只验证边界逻辑，不能替代下一节真实隐藏 AP 验收。

## 五、真实隐藏 AP 空口验收

### 5.1 前置条件

- 必须有一个用户确认或空口证据确认的隐藏 AP，记录其 BSSID、信道和原始 Beacon SSID IE 字节。
- 同时选择至少一个正常可见 SSID 的 AP 作为对照。
- 如果现场找不到隐藏 AP，最终结论必须是“信息不足”，不得判定通过。

### 5.2 原始空口证据

在不竞争或破坏正式 worker 的前提下，捕获隐藏 AP Beacon，并保存以下证据：

- BSSID
- 信道
- 802.11 帧类型确为 Beacon
- SSID IE 确实存在且长度为 0，或内容全为 `0x00`
- 对照 AP 的正常 SSID IE 内容

不得只凭前端显示“没有名称”推断它是隐藏 AP。

### 5.3 业务链路验证

使用与目标信道匹配的最小安全扫描路径验证真实输出。若需开启单信道/最小点位内部测试，
必须先获得用户许可，并在测试结束后恢复原配置和服务状态。

至少核对：

1. `GET /api/v1/ptz/full_area_scan/result` 或当前实际承载 AP 身份结果的正式接口。
2. 对应轮次的 Redis `full_scan:results`。
3. 与同一 `scan_id` 对齐的服务日志和摘要。

隐藏 AP 必须满足：

```json
{
  "type": "ap",
  "ssid": "_wildcard_"
}
```

并确认：

- `subtype` 仍遵循原规则，没有被改为隐藏 SSID 标记。
- 同一轮和后续回填没有把 `"_wildcard_"` 覆盖回 `""` 或 `null`。
- 正常 AP 的 SSID 未被改写。
- 结果中不存在将“SSID 未知”误标为 `"_wildcard_"` 的证据。

## 七、成功与失败判定

只有同时满足以下条件才可报告“通过”：

- 设备端纯逻辑三态全部通过。
- 有真实隐藏 AP 的原始 Beacon 证据。
- 该真实隐藏 AP 在正式业务结果中输出 `ssid="_wildcard_"`。
- 正常 AP 名称保持不变。
- 缺失或不可解析 SSID 保持 `null`，没有误报隐藏。
- `type/subtype`、API、Redis、scan_id、停止语义和日志无回归。
- 测试环境已恢复。

以下任一情况必须报告“失败”：

- 明确隐藏仍输出 `""` 或 `null`。
- 缺失/解析失败被输出为 `"_wildcard_"`。
- 正常 SSID 被改写。
- `subtype` 被隐藏标记挤占。
- API 与 Redis 结果不一致。
- 服务、扫描状态、停止语义或旧身份识别出现回归。

没有真实隐藏 AP、无法取得原始 Beacon 证据，或业务结果没有捕获目标 BSSID时，报告“信息不足”，不是通过。
