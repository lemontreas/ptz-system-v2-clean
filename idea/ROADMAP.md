## 二、软件功能规划

### S1. 全景图采样点优化算法
- **触发场景**：多点扫描 / 全面扫描，云台在路径上移动时
- **核心目标**：从移动路径点中筛选最优停留点，最小化拼接次数
- **输入**：云台移动的完整路径点集合
- **输出**：推荐在哪些点位停止并采样
- **拼接策略**：为应对室内近景的“节点视差(Parallax)”和微步长的“旋转歧义”，采用 `pano`（高质量球面，适合单行/远景）+ `scans`（兜底仿射平面，适合密集矩阵）的自动降级双轨制。
- **畸变处理**：`pano` 模式自带出色的球面展开平整功能；`scans` 模式仅做二维叠图，不强求三维广角去畸变。
- 【待明确】摄像头视角（FOV）大小，影响采样点间距计算

### S2. 摄像头与天线的垂直偏差修正
- **问题**：摄像头光轴与天线在垂直方向存在固定偏差；水平方向对齐无需修正
- **修正目标**：确保 pan/tilt 角度与图像像素位置的映射准确
- **附加**：全景图平整操作本身也可能引入偏差，需在修正中一并考虑
- 【待明确】偏差是固定值（硬件装配决定）还是可由用户校准输入

### S3. 扫描改为定时重复 + 历史数据存储

#### S3-A. 多点扫描重复机制

**重复逻辑**：
- 启动后持续循环执行，直到收到停止指令
- 每轮扫描完成后等待 `repeat_interval` 秒再开始下一轮
- `repeat_interval` 为配置常量，当前仅后端修改（不对外暴露API）

**MAC 来源 — 两种模式（启动时由参数 `mode` 指定）**：

模式A：`auto`（来自初始扫描）
- 首次启动前需要有初始扫描结果（`multi_scan:target_macs`）
- 每隔 `mac_refresh_rounds`（可配置常量）轮，重新执行一次初始扫描
- MAC 列表更新策略：**增删并行**
  - 新发现的 MAC → 追加到列表
  - 连续 `MAC_MISS_THRESHOLD`（默认 3）次刷新扫描都未出现的 MAC → 从列表中移除
  - 每个 MAC 内部维护 `_miss_count` 计数器：出现则归零，未出现则 +1
  - 理由：偶然抓到一次就消失的 MAC 不应永久占用扫描时间；真正存在的设备会在后续刷新中重新被发现

模式B：`manual`（用户自定义）
- 用户启动时提交 MAC 列表，信道/带宽为可选字段
- 若未提供信道/带宽 → 系统在第一轮扫描前自动探测：
  - 遍历 `1.json` 中所有 (channel, bandwidth) 组合（共69个）
  - 监听每个组合 N 秒，记录能捕获到该 MAC 的所有有效配置
  - 找到多个配置 → 全部保存，均参与多点扫描
  - 本次未找到 → 该 MAC 本轮跳过，下一轮重试（不删除）
- MAC 列表固定不变，不触发初始扫描，不做周期性刷新
- 实现上复用初始扫描的底层逻辑，只过滤目标 MAC

**校准**：每轮开始时去限位点校准一次，轮内各点位不再重复校准

**API 变更**（`web_server.py`）：
- `multi_point_scan/start` 新增参数：
  - `mode`: `"auto"` | `"manual"`
  - `manual_macs`: `[{"mac": "xx:xx", "channel": 6, "bandwidth": "HT20"}, ...]`（manual模式）
  - `repeat_interval`: 轮次间隔秒数（暂由后端配置常量控制，此参数预留）
- `multi_point_scan/set_interval`：预留接口，运行中动态修改间隔（暂不实现）

**ptz_control.py 结构变化**：
```
收到 start_multi_point_scan 命令
  ├── 记录初始扫描位置（用户手动移到的位置）
  └── while 未收到停止标志:
        ├── 去限位点校准（每轮一次，消除累积误差）
        ├── (auto模式 且 本轮需刷新) → 回到初始扫描位置 → discover_macs → 增删并行更新MAC列表
        ├── (manual模式 且 第一轮 且 有未探测MAC) → 遍历66个配置探测有效信道
        ├── run_multi_point_scan_blocking()
        ├── 记录轮次编号 + 时间戳到 Redis
        └── sleep(repeat_interval) 期间分段检测停止标志
```

**每轮流程顺序（重要）**：
1. 校准 → 去限位点消除累积误差
2. 回到初始扫描位置（仅刷新轮次）→ 确保刷新时天线朝向与首次初始扫描一致
3. 刷新 MAC（仅刷新轮次）→ discover_macs，增删并行
4. 正式扫描 → 遍历路径点位

**MAC 列表跨项目行为**：
- 停止多点扫描后，`multi_scan:target_macs` 仍保留在 Redis 中
- 如果直接再启动 `start_multi_point_scan`（auto 模式），会沿用上一轮的 MAC 列表
- 只有重新执行 `start_initial_scan` 才会清掉旧列表并重新扫描

### S4. 卫星图联动与空间映射

- 【待明确】卫星图来源（离线图片 / 在线地图 API）

### S5. 时间轴功能
- **交互**：用户拖动时间轴，查看不同时间节点的数据快照
- **随时间变化的内容**：
  - 各点位 RSSI 热力数据
  - 对应时刻的全景拼接图
- **设备标注**：对每个被采集设备标注首次出现时间、历史出现时间段

### S6. 多点位数据融合与测向
- **场景**：不同设备不同点位同时采集 / 同一设备不同点位长时间采集
- **融合目标**：合并不同点位的采集数据，获得目标设备的多方向信号样本
- **最终能力**：通过多方向信号交叉验证，实现对目标设备的测向定位
- 【待明确】测向算法方向（RSSI加权三角定位 / 最大功率方向法 / 其他）

### S7. 全向天线并行采集（距离参考基线）

- **背景**：当前系统仅使用一张定向天线网卡采集 RSSI，无法判断目标设备的远近。定向天线增益高、"看得远"，但缺少参考基线。
- **方案**：新增第二张 USB 网卡，连接全向天线，在多点扫描和全面扫描期间与定向天线**同信道、同时间窗口并行抓包**。
- **核心价值**：
  - 定向天线能收到但全向天线收不到 → 目标大概率较远
  - 两者都能收到但全向 RSSI 明显弱于定向 → 目标在定向天线增益方向上、距离中等
  - 两者 RSSI 接近 → 目标较近或在全向天线覆盖范围内
  - **后端不做远近判断**，只输出双路 RSSI 数据，判断逻辑由分析端/前端自行实现
- **硬件要求**：
  - 第二张 USB WiFi 网卡，支持 Monitor 模式
  - 全向天线（替换网卡自带天线或外接）
- **软件改动范围**：
  - `capture_worker.py`：新增全向网卡环境变量、Monitor 初始化、双卡并行 sniff 逻辑
  - `ptz_control.py`：不改（命令格式不变）
  - `web_server.py`：不改（透传数据）
  - 输出数据结构：每条扫描结果新增 `omni_rssi_avg` / `omni_sample_count` 字段
- **向后兼容**：全向网卡环境变量为空时，行为与改动前完全一致

### S8. 客户端扫描

- **核心目标**：在用户指定的方向范围内逐格扫描，找出信号最强的客户端设备（STA）。
- **关联分析**：列出扫描到的客户端与其关联/尝试关联的 AP 的对应关系（AP BSSID + SSID）。
- **RSSI 采样约束**：只记录 **TA（Transmitter Address）= 客户端 MAC** 的帧的 RSSI，确保测量的是客户端的信号强度，而非 AP 的信号强度。
- **客户端帧识别**：Probe Request / Association Request / Reassociation Request / Data 帧（To DS=1, From DS=0）。
- **扩展扫描防误判**：实际扫描范围在用户选区基础上四周各扩展 `CLIENT_SCAN_GUARD_STEPS` 步，防止边界格子因旁瓣信号产生误判。扫完后自动计算每个客户端的峰值格子是否在用户选区内（`confidence: high/medium/low`）。
- **一次性任务**：扫完所有格子后自动结束，不循环，不存历史，结果写入 Redis 临时 Key。
- **架构**：独立扫描任务，复用 `full_area_scan` 的云台移动框架，新增客户端帧过滤层。
- **详细设计**：见 `DESIGN_S8.md`

### 方向 A：识别能力扩展

| ID | 想法 | 复杂度 | 备注 |

### 方向 B：空间定位精度

| ID | 想法 | 复杂度 | 备注 |
