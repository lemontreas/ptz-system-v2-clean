# 全面扫描 — 当前算法问题分析与优化方向

> 本文档整理当前三段式全面扫描算法（粗扫 / 细扫 / 偏差区）在实际场景中已知存在或潜在存在的设计缺陷，作为后续优化迭代的依据。

---

## 一、已发现的问题

---

### 问题 1：工作区过大时偏差区失效

**触发条件**：单个 work_range 面积占 precheck_range 面积的比例过高（约 > 70%）时。

**现象**：

偏差区的计算逻辑是将 work_range 向外扩展 9°，然后取与 precheck_range 的交集：

```python
guard_pan = [
    max(pre_pan_min, pan_min - 9°),
    min(pre_pan_max, pan_max + 9°),
]
if guard_pan[0] >= guard_pan[1]:
    return None  # 偏差区被完全裁掉
```

当 work_range 很大（接近 precheck_range 边界）时，外扩 9° 后立刻被截断，偏差区实际范围极窄甚至为零。

**连锁影响**：

1. 偏差区点位极少甚至为零 → 策略A 无法执行
2. 粗扫阶段几乎所有点都落在 work_range 内 → `other_list`（工作区外RSSI数据）为空
3. 白名单筛选中的核心逻辑依赖工作区内/外 RSSI 差值对比，`other_list` 为空时这个判断直接跳过 → **所有 MAC 都被当作"高置信"通过，筛选形同虚设**

**当前代码位置**：
- `_expand_work_range_for_full_scan_guard()` — L1201
- `_collect_deviation_target_macs()` — L1666
- `_build_full_scan_whitelist_payload()` — L1601

**根本原因**：算法假设 work_range 远小于 precheck_range，没有处理两者接近的退化情况。

---

### 问题 2：白名单筛选未利用历史先验信道信息

**触发条件**：全面扫描循环运行多轮后，前几轮已积累了白名单（已知设备的信道/带宽配置）。

**现象**：

每一轮无论是否有历史白名单，粗扫和细扫的每个点位都要执行完整的全信道遍历（`INITIAL_SCAN_CONFIGS`，覆盖 2.4G + 5G 全部信道），每个点位耗时约为：

```
点位时间 ≈ 信道数量 × dwell_time
# 例如 66 个信道 × 0.3s = 约 20s / 点位
```

已知设备的信道完全被忽略，每轮都从头全量扫描。

**优化空间**：

对于白名单中已知信道的 MAC，在细扫阶段只需扫其已知信道即可验证（无需全信道），可将单点耗时大幅压缩。

**当前代码位置**：
- 细扫发命令处 — L4041（`discover_macs_for_full_scan`，configs 始终用 `INITIAL_SCAN_CONFIGS`）
- 粗扫发命令处 — L3867（同上）

---

### 问题 3：多个 work_ranges 的步径由最小区域决定，可能不合理

**触发条件**：同时存在多个 work_range，且各区域大小差异悬殊。

**现象**：

`_calc_fine_scan_step(work_ranges)` 的逻辑是遍历所有 work_range，取计算结果最小的步径：

```python
best_step = min(best_step, step)  # 取所有区域中最小的步径
```

结果：一个面积很小的区域（比如 5°×5°）会把步径拉到 2°，导致其他面积很大的区域也用 2° 步径细扫，产生大量不必要的点位，耗时倍增。

**当前代码位置**：
- `_calc_fine_scan_step()` — L1458

**更合理的方案**：每个 work_range 使用各自的步径独立生成路径，而不是共用最小步径。

---

### 问题 4：工作区很小时仍执行全 precheck_range 粗扫，效率低

**触发条件**：work_range 很小（面积 < 100°²，约 10°×10° 以内）。

**现象**：

粗扫始终覆盖整个 precheck_range（可能达到 60°×40° 甚至更大），产生大量与小工作区无关的扫描点位，占用了宝贵的时间窗口。

而对于小工作区，真正有价值的是提高工作区内的扫描分辨率（增加细扫次数），而不是在大范围内反复粗扫。

**当前代码位置**：
- `_build_coarse_scan_path()` — L1531（固定覆盖全 precheck_range）
- `_calc_fine_scan_count()` — L1479（已有面积判断但未联动粗扫范围）

## 二、问题影响汇总

| # | 问题 | 严重程度 | 影响范围 | 对应方向 |
|---|---|---|---|
| 1 | 工作区过大导致偏差区失效 + 白名单筛选失效 | ⭐⭐⭐ 高 | 白名单正确性 | 方向A |
| 2 | 未利用历史信道信息，全量信道重复扫描 | ⭐⭐ 中 | 扫描速度 | 方向B |
| 3 | 多 work_range 步径由最小区域决定 | ⭐⭐ 中 | 点位数量/效率 | 方向C |
| 4 | 小工作区时粗扫范围过大 | — | — | **不实现**，已有机制覆盖 |

## 三、优化方向设计

---

### 方向 A：按偏差区实际可用面积分档，切换筛选策略（对应问题 1）

#### A.1 判别指标：偏差区实际可用面积比（`dev_area_ratio`）

不使用单纯覆盖率（work/precheck面积比），原因：
- 覆盖率无法反映 work_range 是否贴近 precheck 边界（反例：覆盖率 38% 但四边各只剩 9° 余量，偏差区为零）
- 硬件限制导致某一方向（如垂直）天然较窄，不应因此降级策略

**计算方式**：

```python
# 理论偏差区：work_range 向四周各扩 GUARD_DEG(9°)，不限边界
theoretical_pan  = work_pan_span  + 2 * GUARD_DEG
theoretical_tilt = work_tilt_span + 2 * GUARD_DEG
theoretical_dev_area = theoretical_pan * theoretical_tilt - work_pan_span * work_tilt_span

# 实际偏差区：扩展后取与 precheck_range 的交集，再减去工作区本身
actual_pan_min  = max(precheck_pan_min,  work_pan_min  - GUARD_DEG)
actual_pan_max  = min(precheck_pan_max,  work_pan_max  + GUARD_DEG)
actual_tilt_min = max(precheck_tilt_min, work_tilt_min - GUARD_DEG)
actual_tilt_max = min(precheck_tilt_max, work_tilt_max + GUARD_DEG)
actual_expanded_area = (actual_pan_max - actual_pan_min) * (actual_tilt_max - actual_tilt_min)
actual_dev_area = actual_expanded_area - work_pan_span * work_tilt_span

dev_area_ratio = actual_dev_area / theoretical_dev_area  # 0.0 ~ 1.0
```

> 垂直范围天然窄时（如 tilt 只有 15°），若 pan 方向偏差区仍然宽裕，面积比不会跌至大范围模式，不会误触策略切换。

**多个 work_ranges 时的策略决策**：

对每个 work_range 分别计算 `dev_area_ratio`，取**最小值**作为本轮统一档位。

```python
round_dev_area_ratio = min(
    _calc_dev_area_ratio(wr, precheck_range)
    for wr in work_ranges
)
# 用 round_dev_area_ratio 决定本轮所有区域统一走哪个档位
```

> **设计决策（思路一）**：统一取最差区域的比值，避免同一轮内不同区域策略不一致导致筛选逻辑复杂化。
> 代价是：某个大区域会拉低整体档位，小区域可能被迫走宽松/大范围模式。实测场景中多个差异悬殊的 work_range 不常见，后续如有必要再迭代为 per-range 独立策略。

#### A.2 三档策略

| 档位 | 触发条件 | 偏差区处理 | 筛选逻辑 |
|---|---|---|---|
| **正常模式** | `dev_area_ratio ≥ 0.7` | 策略A（外扩边缘）→ 按需策略B | 当前逻辑：工作区RSSI - 区域外RSSI ≥ `min_work_delta`（默认4dB） |
| **宽松模式** | `0.3 ≤ dev_area_ratio < 0.7` | 策略A（外扩边缘）→ 按需策略B | 放宽阈值：RSSI差值 ≥ `min_work_delta × 0.5`（约2dB） |
| **大范围模式** | `dev_area_ratio < 0.3` | **跳过策略A，直接执行策略B** | 切换为 RSSI 分布分析（详见 A.3） |

> 两个阈值（0.7 / 0.3）建议放入 `config.json` 可配置，不硬编码。

#### A.3 大范围模式的筛选逻辑

内外RSSI差值对比数据不可靠，改为以下两步判断：

**Step 1：执行策略B，能拿外部数据算一点（补充验证）**

策略B在整个 precheck_range 内定向侦察（锁定目标MAC已知信道），尽量收集工作区外的RSSI数据。

| 情况 | 处理 |
|---|---|
| 策略B有外部数据 | 计算 `work_best - outside_best`，只要 ≥ 0（工作区不弱于外部）即"补充验证通过" |
| 策略B无外部数据 | 跳过这一层，不参与评分 |

**Step 2：工作区内RSSI分布分析（必做）**

| 判断维度 | 含义 | 通过条件 |
|---|---|---|
| ① 峰值位置 | 信号最强点是否在工作区内部 | 峰值点距工作区任意边界 > 工作区短边的 15%（边缘保护） |
| ② 信号梯度 | RSSI分布是否有明显方向性（非均匀平铺） | 工作区内各点RSSI标准差 > 梯度阈值（建议初始值 3dBm，待实测校准） |

**Step 3：综合判断**

```
有外部数据：补充验证通过 + 分布分析 ≥ 2项通过  →  纳入白名单
无外部数据：分布分析全部 3 项通过              →  纳入白名单
否则                                          →  标为"待确认"，不纳入白名单
```

### 方向 B：冷启动 / 温启动两档，按白名单先验信道信息提速（对应问题 2）

#### B.1 两档模式定义

| 模式 | 触发条件 | 粗扫行为 | 细扫行为 |
|---|---|---|---|
| **冷启动** | 历史白名单 MAC 数量 < `warm_start_threshold`（默认 **10**） | 全信道遍历（当前逻辑） | 全信道遍历（当前逻辑） |
| **温启动** | 历史白名单 MAC 数量 **≥** `warm_start_threshold` | 全信道遍历（不变，保留新MAC发现能力） | 优先扫白名单已知信道；剩余信道每 2 个点位扫一次 |

> `warm_start_threshold` 写入 `config.json` 可配置，默认值 **10**。
> 粗扫保持全信道遍历不变，保证每轮仍能发现新出现的设备。

#### B.2 温启动细扫执行逻辑

```
每个细扫点位（point_idx 从 0 开始）：
    step 1 → 扫白名单已知信道（必做）
             configs = 去重后的白名单信道/带宽配置列表
    step 2 → if point_idx % 2 == 0:
                 追加扫剩余信道（INITIAL_SCAN_CONFIGS 中不在白名单内的部分）
             else:
                 跳过剩余信道
```

**效果估算**（以 66 个总信道、白名单含 10 个已知信道为例）：

| | 冷启动 | 温启动 |
|---|---|---|
| 每个点位扫信道数 | 66 个 | 奇数点：10 个；偶数点：66 个 |
| 平均每点位信道数 | 66 | 38 |
| 理论提速比 | 1× | ≈ 1.7× |

#### B.3 白名单有效性判断

温启动使用的白名单来自上一个成功轮次（`full_scan:whitelist:latest_success`）。

有效性判断：
- 白名单 key 存在且 MAC 数量 ≥ `warm_start_threshold` → 温启动
- 否则（白名单不存在 / MAC 不足 / Redis 读取失败）→ 冷启动（安全回退）

> 暂不引入"白名单过期"机制，由 Redis TTL 和轮次覆盖自然淘汰旧数据。

### 方向 C：多 work_ranges 各自用独立步径（对应问题 3）

#### C.1 问题

当前 `_calc_fine_scan_step(work_ranges)` 取所有区域中最小的步径，所有区域共用。一个 5°×5° 的小区域会把步径拉到 2°，导致 50°×50° 的大区域也用 2° 细扫，产生大量无效点位。

#### C.2 修改方案

`_build_fine_scan_path()` 改为每个 work_range 单独计算步径，独立生成路径，最后合并去重：

```python
def _build_fine_scan_path(work_ranges, pan_offset=0.0, tilt_offset=0.0):
    seen = set()
    all_points = []
    for item in work_ranges:
        # 每个区域各自算步径
        span_h = abs(item["pan_range"][1]  - item["pan_range"][0])
        span_v = abs(item["tilt_range"][1] - item["tilt_range"][0])
        step = clamp(round(min(span_h, span_v) / 3), FINE_STEP_MIN, FINE_STEP_MAX)
        # 用该区域自己的步径生成蛇形路径
        points = _build_single_range_fine_path(item, step, pan_offset, tilt_offset)
        for pt in points:
            if pt not in seen:
                seen.add(pt)
                all_points.append(pt)
    return all_points
```

#### C.3 与方向 A 的关系

方向C影响的是**路径生成**（每个区域用多细的步径），方向A影响的是**筛选逻辑**（哪个档位）。两者正交，不冲突：
- 方向A的策略档位由 `round_dev_area_ratio`（取最小值）统一决定
- 方向C的步径由每个 work_range 自身大小独立决定
- 两个维度各算各的，互不影响

### 方向 D：小工作区时缩减粗扫范围（对应问题 4）——**不实现**

**结论**：工作区范围是用户定义的客观区域，粗扫覆盖整个 precheck_range 是正确行为，不应因工作区小而缩减，否则会丢失工作区外的对比数据，反而破坏方向A的筛选逻辑。

**已有机制已覆盖**：
- `_calc_fine_scan_count()` 中 `area < 100 → base += 1`，工作区小时自动多做几次细扫
- `_calc_fine_scan_step()` 中工作区小时步径也会计算得更小

无需额外修改，此方向关闭。

## 四、待确认的开放问题

1. **方向A三档阈值**（dev_area_ratio: 0.3 / 0.7）建议首次实测后根据设备实际 precheck_range 数据校准。
2. **方向A分布分析参数**（待实测校准）：
   - 边缘保护角度：建议工作区短边的 15%，是否合适？
   - 梯度阈值（RSSI标准差）：建议初始值 3dBm。
3. **方向B温启动阈值**：暂定 10，后续可根据实测场景调整。
4. **各优化方向的实施顺序**：方向A（正确性）→ 方向B（速度）→ 方向C（效率，后续迭代）。
