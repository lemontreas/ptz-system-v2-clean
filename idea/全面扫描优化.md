# 全面扫描逻辑优化方案

> 本文档为全面扫描算法的逻辑设计说明，面向 AI 实现者。请严格按照本文档的逻辑实现，不要在未说明的情况下自行补充或更改扫描策略。

---

## 一、术语定义

| 术语 | 说明 |
|---|---|
| **工作区** | 前端传入的粗筛范围，即天线的可视工作区，是一个较大的角度范围（pan_min ~ pan_max，tilt_min ~ tilt_max） |
| **目标区** | 用户在工作区内标注的关注范围，是工作区的子集，代表用户实际关心的空间区域 |
| **偏差区** | 工作区向外扩展的边缘缓冲区域，默认扩展 9°，用于检测处于目标区与工作区边缘的设备，防止误判 |
| **步径** | 云台每次移动的角度步长，单位为度（°） |
| **dwell_time** | 网卡在某个信道配置上的停留采集时间，单位为秒（s） |
| **old MAC** | 工作区粗扫阶段已经出现过的 MAC 地址 |
| **new MAC** | 工作区粗扫未出现、但目标区细扫阶段新出现的 MAC 地址 |
| **置信度** | 对某个 MAC 地址是否处于目标区内的综合判断可信程度 |

---

## 二、信道扫描策略（固定）

- 遍历 **2.4G + 5G 全部信道**，每个信道使用 **HT20 带宽**进行监听
- 使用 HT20 遍历所有信道可以完整覆盖所有设备：无论目标设备使用 HT20/HT40/HT80，其 Beacon 帧和 Probe Response 均在**主信道（Primary Channel）**发送，HT20 全信道遍历不会漏掉任何设备
- **不使用 HT40 或更宽带宽进行扫描**，避免信道重叠导致漏检

---

## 三、一次全面扫描的完整流程

一次全面扫描由以下三个阶段顺序执行：

```
[阶段一] 工作区粗扫
       ↓
[阶段二] 目标区细扫（多次）
       ↓
[阶段三] 偏差区扫描（策略A，必做；策略B，按需触发）
```

---

### 阶段一：工作区粗扫

**目标：** 以最快速度建立工作区内的无线信号概览，得到 old MAC 列表及其信道配置。

**参数：**
- 步径：**9°**（固定，快速覆盖）
- dwell_time：**0.25s ~ 0.5s / 信道**（跨轮次动态调整）
- 执行次数：**每次全面扫描只执行一次**

**起始点偏移策略（交错采样）：**

工作区粗扫的起始点在每次全面扫描时不固定，而是按照交错采样规则偏移，目的是累积多次扫描后尽可能覆盖工作区内更多的角度点位，等效提升整体分辨率。

规则如下：
- 将步径 S 等分为 N 份（推荐 N=3，即每次偏移 S/3 = 3°）
- 每次全面扫描的起始点按以下顺序循环偏移：

| 扫描轮次 | 水平起始偏移 | 垂直起始偏移 |
|:---:|:---:|:---:|
| 第 1 次 | 0° | 0° |
| 第 2 次 | 3° | 0° |
| 第 3 次 | 6° | 0° |
| 第 4 次 | 0° | 3° |
| 第 5 次 | 3° | 3° |
| 第 6 次 | 6° | 3° |
| 第 7 次 | 0° | 6° |
| 第 8 次 | 3° | 6° |
| 第 9 次 | 6° | 6° |
| 第 10 次 | 0° | 0°（循环重置） |

经过 9 次全面扫描后，工作区的每个角度格子都被覆盖，实际分辨率从 9° 提升至 3°。

**第 9 次之后循环重置（不继续细分）：**
- 9 次后起始点重置回 (0°, 0°)，重新开始下一轮 9 次交错循环
- **原因：** 工作区设备状态随时间变化（设备开关机、移动、切换信道），继续细分只会积累越来越旧的数据；重置可以保证地图数据的时效性，用新数据持续刷新旧格子
- 更高分辨率的工作由**阶段二目标区细扫**承担，工作区粗扫只需保持"概览级"覆盖即可

**输出：**
- old MAC 列表（含每个 MAC 的信道+带宽配置、各角度 RSSI 数据）

---

### 阶段二：目标区细扫

**目标：** 对用户标注的目标区进行精细扫描，得到目标区内设备的详细 RSSI 数据，并发现 new MAC。

**参数：**
- 步径：**动态计算，范围 [2°, 9°]**（根据目标区大小一次性确定，任务内不变）
- dwell_time：**0.5s ~ 1s / 信道**（跨轮次动态调整）
- 执行次数：**动态调整，范围 [2, 5] 次**

#### 步径动态计算规则

```
N = 3  # 期望每个方向覆盖的点位数（固定值）

span_h = 目标区水平角度跨度
span_v = 目标区垂直角度跨度

raw_step = min(span_h / N, span_v / N)  # 取较小方向，保证两边都有覆盖
step = clamp(round(raw_step), 2°, 9°)
```

**边界情况处理：**
- 若某一方向 span < 2°（小于 step 下限）：该方向**固定在中心，不移动**，只移动另一方向
- 若两个方向 span 均 < 2°：**直接在中心点扫描，不移动云台**（目标区极小情况）

#### 次数动态调整规则

```
base = 3（默认）
if 目标区角度面积（span_h × span_v）> 900°²（即约 30°×30°以上）:  base -= 1
if 目标区角度面积 < 100°²（即约 10°×10°以下）:  base += 1
if white_list MAC 数量少 或 本轮 new MAC 数量多:  base += 1
if 上轮扫描数据稳定（RSSI 波动小，无新 MAC）:  base -= 1
次数 = clamp(base, 2, 5)
```

#### 多次细扫偏移规则

每次细扫的起始点在目标区左上角基础上叠加偏移量，偏移量以 `step / 2` 为单位，保证多次扫描后覆盖点位尽可能互补：

| 第几次 | 水平偏移 | 垂直偏移 |
|:---:|:---:|:---:|
| 第 1 次 | 0 | 0 |
| 第 2 次 | step/2 | 0 |
| 第 3 次 | 0 | step/2 |
| 第 4 次 | step/2 | step/2 |
| 第 5 次 | step/4 | step/4 |

- 偏移后超出目标区边界的路径点**直接跳过**，只扫目标区内的点
- 4 次后目标区内等效分辨率从 `step` 提升至 `step/2`，第 5 次进一步填补中心空隙

**扫描路径形状根据目标区形状自适应：**
- 目标区为窄长条（某一方向 span < 2°）：该方向固定中心，只移动另一方向
- 其他形状：按蛇形路径遍历所有点位

**输出：**
- 目标区内所有 MAC 的 RSSI 分布数据
- **new MAC 列表**（含信道+带宽配置）
- 置信度模糊的 old MAC 列表（边缘情况，待偏差区确认）

---

### 阶段三：偏差区扫描

**目标：** 对无法直接判断的 MAC（new MAC 和置信度低的 old MAC）进行定向侦察，辅助最终筛选决策。

**偏差区扫描不是无差别全扫，而是始终针对具体目标 MAC 进行定向侦察。**

#### 3.1 哪些 MAC 需要进入偏差区扫描？

| MAC 类型 | 判断条件 | 处理方式 |
|---|---|---|
| **可直接纳入白名单的 old MAC** | 在目标区多次出现、RSSI 强且稳定、方向性明确（某角度最强） | 直接纳入白名单，跳过偏差区 |
| **可直接排除的 old MAC** | 在目标区信号极弱、方向明确指向工作区外 | 直接排除，跳过偏差区 |
| **置信度低的 old MAC** | 信号强度处于边缘、在目标区和目标区外均有出现、强度差异不明显 | **进入偏差区扫描** |
| **new MAC** | 工作区粗扫未出现、目标区细扫新出现 | **进入偏差区扫描** |

#### 3.2 策略 A：9° 偏差区侦察（必做）

**范围：** 工作区向外扩展 9°（水平和垂直四个方向均扩展）

**扫描逻辑：**
- 将所有需要侦察的目标 MAC 收集为一个列表，每个 MAC 匹配其已知信道+带宽配置
- **合并扫描：在偏差区内每个角度点位停留时，依次切换各目标 MAC 对应的信道，分别采集 dwell_time**
  - 即：每个角度点位 = 所有目标 MAC 的信道各扫一次，不是每个 MAC 单独走一遍偏差区
  - 好处：多个 MAC 合并到同一轮角度遍历，总时间不随 MAC 数量成倍增长
- 每个角度点位单个信道的 dwell_time：0.5s ~ 1s

**判断逻辑：**
- 若目标 MAC 在偏差区出现且 RSSI 可接受 → 确认该设备在目标区边缘，根据 RSSI 强弱决定纳入或排除
- 若目标 MAC 在偏差区未出现 → 触发策略 B

#### 3.3 策略 B：工作区范围定向侦察（按需触发）

**触发条件：** 策略 A 结束后，仍有 MAC 在 9° 偏差区内未找到

**范围：** 整个工作区（与阶段一相同的角度范围）

**扫描逻辑：**
- 取策略 A 结束后仍未找到的目标 MAC 列表
- **合并扫描：在整个工作区内每个角度点位停留时，依次切换各目标 MAC 对应的信道，分别采集 dwell_time**，不对每个 MAC 单独走一遍工作区
- 步径 9°，每个角度点位单个信道 dwell_time：0.5s ~ 1s

**策略 B 时间可控的原因：**
- 信道维度从"遍历全部信道"压缩为"只扫 1 个目标信道"，时间量级大幅降低
- 多个 MAC 合并到同一轮角度遍历，总时间不随 MAC 数量成倍增长
- 实质上是一次"针对多个目标 MAC 的定向全区侦察"，而非重复粗扫

**判断逻辑：**
- 若在工作区内找到该 MAC → 记录其角度区域和 RSSI，综合判断
- 若在工作区内仍未找到该 MAC → 判定为工作区外的远端设备，将其标记为"疑似外部干扰"，不纳入白名单

## 四、MAC 白名单筛选策略（置信度评估）

每个 MAC 地址最终会被归入以下三类之一：

| 分类 | 条件 | 处理 |
|---|---|---|
| **目标区内设备（高置信）** | 在目标区多次出现、RSSI 强且稳定、偏差区不出现或明显弱于目标区 | 纳入白名单 |
| **边缘模糊设备（低置信）** | 偏差区与目标区信号强度接近，无法明确判断位置 | 标记为"待确认"，不纳入白名单，进入下一轮联动再判断 |
| **外部设备（排除）** | 在工作区和偏差区均未出现，或信号明显更强在偏差区方向 | 排除 |

## 五、跨轮次联动优化

每次全面扫描结束后，其结果将用于优化下一次全面扫描的参数：

| 数据 | 用途 |
|---|---|
| 白名单 MAC + 信道配置 | 下一轮目标区细扫和偏差区扫描可优先扫描这些信道 |
| 上轮各阶段 RSSI 稳定性 | 判断目标区细扫次数（稳定则减少，波动则增加） |
| 工作区粗扫起始点偏移记录 | 按交错采样序列自动推进到下一个起始点 |
| "待确认"MAC 列表 | 下一轮优先在偏差区对这些 MAC 做重点确认 |

各阶段的 dwell_time 均有给定范围，应根据上一轮扫描的实际数据情况在范围内动态调整，不得超出上下限。**目标区细扫步径在任务开始时根据目标区大小一次性计算确定，任务运行期间不跨轮次变化。**

> **注意：** 白名单数据应设置有效期机制。若某 MAC 连续 N 轮全面扫描均未出现，应将其从白名单降权或移除，避免过期数据干扰判断。N 值建议根据实际场景设定，默认可为 3~5 轮。

## 六、总体流程图

```
开始一次全面扫描
        │
        ▼
[阶段一] 工作区粗扫
  步径9°（固定），dwell=0.25s~0.5s，遍历所有信道
  起始点按交错采样偏移
  → 输出 old MAC 列表
        │
        ▼
[阶段二] 目标区细扫（2~5次，跨轮次动态调整）
  步径=[2°~9°]（任务内固定），dwell=0.5s~1s（跨轮动态）
  多次细扫按偏移表错开路径
  → 输出 new MAC 列表 + 低置信 old MAC 列表
        │
        ▼
[阶段三] 偏差区扫描
  ┌─────────────────────────────────────┐
  │ 策略A（必做）：9°偏差区            │
  │ 锁定目标MAC信道，步径9°，dwell=0.5s~1s │
  │                                     │
  │   ┌── A 能找到 → 综合判断纳入/排除  │
  │   └── A 找不到 → 触发策略B          │
  │                                     │
  │ 策略B（按需）：整个工作区           │
  │ 锁定目标MAC信道，步径9°，dwell=0.5s~1s │
  │                                     │
  │   ┌── B 能找到 → 综合判断纳入/排除  │
  │   └── B 找不到 → 标记为外部干扰排除 │
  └─────────────────────────────────────┘
        │
        ▼
生成本轮白名单（MAC + 信道配置 + RSSI 数据）
        │
        ▼
与下一轮全面扫描联动（优化参数）
```

## 七、关键约束与注意事项

1. **工作区粗扫每次全面扫描只做一次**，不重复执行
2. **工作区粗扫步径固定为 9°**，不跨轮次变化
3. **目标区细扫步径在任务开始时一次性计算确定**（基于目标区大小），任务运行期间不跨轮次变化
4. **偏差区扫描针对的是具体 MAC，不是无差别全扫**，必须锁定目标 MAC 的信道
5. **策略 B 不是重复粗扫**，是对特定 MAC 的定向侦察，两者逻辑不同
6. **dwell_time 与信道数量共同决定每个角度点位的停留时间**，实现时需注意总耗时估算
7. **云台移动时间是物理限制**，不纳入算法逻辑控制，但实现时需在时间预算中留出余量
8. **白名单应有过期机制**，避免旧数据污染新一轮判断
9. **前端接口输出格式不变**，所有优化仅在后端扫描逻辑内进行，对前端透明
