# 前端对接常见问题与注意事项

> 本文档总结了前端在对接后端 API 时，**使用 AI 辅助开发特别容易踩到的坑**，以及已排查到的状态卡死、loading 关不掉等问题的根本原因。请在改任何前端代码前优先阅读本文件。

---

## 🔴 问题一：响应数据没有解包（最常见、最致命）

### 现象
- 点击"开始全面扫描"后，任务在后端实际正常启动
- 但前端的 loading 转圈、弹窗等状态没有按预期关闭
- 控制台报错：`TypeError: Cannot read properties of undefined (reading 'project_id')`

### 根本原因
后端所有接口的返回都有一层统一包装：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {
    "queued": true,
    "config": { "project_id": "..." }
  }
}
```

文档里所有的 JSON 示例（如 `{ "queued": true, "config": {...} }`）描述的是 **`data` 字段里的内容**，不是直接的响应 Body。

### 错误写法 vs 正确写法

```javascript
const res = await axios.post('/api/v1/ptz/full_area_scan/start', body)

// ❌ 错误 - AI 最常犯的错误
const queued = res.data.queued           // undefined
const projectId = res.data.config.project_id  // TypeError: 崩溃!

// ✅ 正确 - 多解一层 .data
const queued = res.data.data.queued
const projectId = res.data.data.config.project_id
```

```javascript
// fetch 写法
const json = await fetch('/api/v1/ptz/status').then(r => r.json())

// ❌ 错误
const fsState = json.full_scan?.state

// ✅ 正确
const fsState = json.data.full_scan?.state
```

### 影响范围
所有接口，包括：
- `POST /api/v1/ptz/full_area_scan/start`
- `POST /api/v1/ptz/location_scan/start`
- `POST /api/v1/ptz/capture_at_best`
- `GET /api/v1/ptz/status`
- `POST .../stop` 等所有接口

## 🔴 问题二：轮询 status 时未处理子对象不存在的情况

### 现象
- 任务刚点击启动后，轮询状态接口立刻报 JS 错误
- 控制台报：`TypeError: Cannot read properties of null (reading 'state')`
- 页面崩溃，后续轮询中断，loading 永远不会关掉

### 根本原因
全面扫描启动后，后端 Worker 需要几秒时间完成初始化/校准，才会把 `full_scan` 写入 Redis。在此之前，`GET /api/v1/ptz/status` 的响应里**根本没有 `full_scan` 这个字段**。

```json
// 任务刚启动时，status 可能长这样（没有 full_scan / location_scan）：
{
  "code": 0,
  "data": {
    "state": "IDLE",
    "position": { "pan": 0, "tilt": 0 }
  }
}
```

### 修复方法：所有状态读取必须用可选链

```javascript
const status = res.data.data   // 先解包

// ✅ 安全
const fsState = status?.full_scan?.state
const lsPhase = status?.location_scan?.phase
const isCapturing = status?.capture?.running === true

// ❌ 危险 - 对象不存在时直接崩溃
const fsState = status.full_scan.state
```

### 正确的启动后轮询逻辑

```javascript
// 任务刚启动，full_scan 可能为 undefined，需要容忍这种情况
function checkFullScanStarted(status) {
  const fsState = status?.full_scan?.state
  // undefined / 'queued' / 'running' 都说明任务正在进行
  return fsState === 'queued' || fsState === 'running' || fsState === undefined
}

// 任务已彻底结束的判断
function isFullScanTerminal(status) {
  const fs = status?.full_scan
  if (!fs) return false  // 子对象不存在 = 任务还没开始，不算结束
  return fs.terminal === true ||
    ['stopped', 'done', 'error'].includes(fs.state)
}
```

## 🟡 问题三：混淆顶层 `state`（大写）和子对象 `state`（小写）

### 现象
- 全面扫描明明在跑，判断逻辑却认为"没有任务在运行"
- 停止任务后，前端不知道何时才应该显示"已停止"

### 根本原因
`status` 对象里有**两套完全不同**的 `state` 字段：

| 字段路径 | 大小写 | 含义 |
|----------|--------|------|
| `status.state` | **大写** | 云台当前物理动作（`"IDLE"` / `"MOVING"` / `"FULL_AREA_SCANNING"`） |
| `status.full_scan.state` | **小写** | 全面扫描任务的生命周期（`"running"` / `"stopping"` / `"stopped"`) |
| `status.location_scan.state` | **小写** | 定位扫描任务的生命周期（`"running"` / `"stopping"` / `"stopped"`） |

### 正确的判断方式

```javascript
// ❌ 错误：顶层 state 是云台动作，不是扫描任务状态
if (status.state === 'running') { showProgress() }    // 永远不会匹配
if (status.state === 'RUNNING') { showProgress() }    // 也是错的，这个值根本不存在

// ✅ 正确：用子对象的 state 判断任务生命周期
if (status.full_scan?.state === 'running') { showProgress() }
if (status.location_scan?.state === 'stopping') { showStopping() }
```

## 🟡 问题四：`state="IDLE"` 不代表任务已停止或抓包已结束

### 现象
- 前端看到顶层 `state="IDLE"` 就认为任务结束，提前把 loading 关掉
- 实际上任务还在后台运行，或抓包还在进行

### 根本原因
顶层 `state="IDLE"` 只表示**云台当前没有在物理移动**，它不能反映：
- 全面扫描任务是否结束（可能云台停着，但扫描还在跑）
- 抓包是否结束（云台停了，但 Scapy 还在抓包）

### 正确的判断方式

```javascript
// 判断全面扫描是否结束
// ✅ 看 full_scan.terminal
const isFullScanDone = status?.full_scan?.terminal === true

// 判断定位扫描是否结束
// ✅ 看 location_scan.terminal
const isLocationScanDone = status?.location_scan?.terminal === true

// 判断抓包是否结束
// ✅ 看 capture.running
const isCaptureRunning = status?.capture?.running === true

// ❌ 错误：用顶层 IDLE 判断
if (status.state === 'IDLE') hideLoading()  // 太早了，可能任务还在！
```

## 🟡 问题五：点击 stop 后立即认为任务已停止

### 现象
- 点击停止按钮后，前端立刻开放了"重新开始"的按钮
- 用户马上点击开始，导致两个任务状态交叠

### 根本原因
`POST /api/v1/ptz/full_area_scan/stop` 成功返回，只代表**停止请求已被后端接收**，不代表任务已经停止。后端 Worker 还需要：
1. 等待当前云台移动完成
2. 等待当前信道抓包任务在最近可中断点退出
3. 写入终止状态到 Redis

### 正确的 stop 交互流程

```javascript
async function handleStopFullScan() {
  // 1. 发送停止请求
  await axios.post('/api/v1/ptz/full_area_scan/stop')

  // 2. 立即显示"正在停止"，禁用 stop 按钮（不要关 loading）
  setScanStatus('stopping')
  setStopBtnDisabled(true)

  // 3. 继续轮询，等 terminal=true 才真正结束
  // 不要在这里 setLoading(false) !!!
}

// 轮询回调里：
function onPollStatus(status) {
  const fs = status?.full_scan
  if (fs?.terminal === true) {
    // ✅ 这才是真正结束
    setLoading(false)
    setStartBtnEnabled(true)
  } else if (fs?.state === 'stopping') {
    // 正在停止，保持"正在收尾"的 UI
    setScanStatus('正在停止...')
  }
}
```

## 🟡 问题六：用 `phase` 判断任务是否结束

### 现象
- 定位扫描结束后，前端没有进入"已停止"状态
- 或前端在 `phase="idle"` 时就认为任务结束，但任务实际还在运行

### 根本原因
`phase` 只表示当前的**执行阶段**（`coarse`/`fine`/`capturing`...），不表示任务的生命周期。判断生命周期应使用 `state` 和 `terminal`：

```javascript
// ❌ 错误
if (status.location_scan?.phase === 'idle') setDone(true)

// ✅ 正确
if (status.location_scan?.terminal === true) setDone(true)
// 或
if (['stopped', 'done', 'error'].includes(status.location_scan?.state)) setDone(true)
```

## 🟢 推荐：使用统一状态解析函数（避免上述所有问题）

把以下函数放入项目的工具文件，所有扫描状态判断都通过它来做：

```typescript
/**
 * 统一解析扫描任务生命周期
 * 适用于 full_scan / location_scan 子对象
 */
function getScanLifecycle(scan?: {
  state?: string;
  phase?: string;
  active?: boolean;
  stop_requested?: boolean;
  terminal?: boolean;
}) {
  // 安全兼容：没有 state 时用 active 字段推断（旧接口兼容）
  const state = scan?.state ?? (scan?.active ? 'running' : 'idle')
  const terminal = scan?.terminal === true ||
    ['stopped', 'done', 'error', 'idle'].includes(state)

  return {
    state,
    phase: scan?.phase,
    isActive: ['queued', 'running', 'stopping'].includes(state), // 任务在跑
    isStopping: state === 'stopping' || scan?.stop_requested === true, // 收尾中
    isTerminal: terminal,    // 彻底结束（可以允许重新开始）
    canStart: terminal,      // 是否允许启动新任务
    canStop: state === 'queued' || state === 'running', // 是否可以发 stop
  }
}

// 标准使用方式：
const rawStatus = res.data.data               // ← 先解包！
const fs = getScanLifecycle(rawStatus?.full_scan)
const ls = getScanLifecycle(rawStatus?.location_scan)

// UI 控制
if (fs.isStopping) showBanner('全面扫描正在停止...')
if (fs.isTerminal) { hideLoading(); enableStartButton() }
if (fs.isActive)   showProgress(rawStatus?.full_scan?.phase)
if (fs.canStop)    showStopButton()
if (!fs.canStart)  disableStartButton()
```

## 快速检查清单

在改完前端代码后，逐条自检：

- [ ] 所有接口响应是否通过 `.data.data` 解包到业务数据？
- [ ] 轮询 status 时，`full_scan` / `location_scan` 的访问是否加了可选链 `?.`？
- [ ] 判断任务是否结束，是否用的是 `terminal === true`，而不是 `state === 'IDLE'`？
- [ ] 顶层 `state`（大写）和子对象 `state`（小写）是否没有混用？
- [ ] stop 按钮点击后，是否在等 `terminal=true` 才开放"重新开始"，而不是在 stop 接口返回后立刻开放？
- [ ] 判断抓包是否结束，是否用的是 `capture.running === false`，而不是 `state === 'IDLE'`？
- [ ] `phase` 字段是否只用于显示进度，没有用来判断任务的生命周期？
