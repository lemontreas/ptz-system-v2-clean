# 前端对接文档：项目化扫描 + 历史时间轴

## 1. 目标

本文档用于前端对接当前后端实现的两类能力：

1. 实时扫描控制与实时结果展示
2. 历史项目查看与时间轴回放

本文档严格以当前代码实际行为为准，不描述尚未实现的能力。

---

## 2. 当前业务模型

当前系统采用两层模型：

- `project`：一次完整扫描项目
- `snapshot`：项目中的每轮历史快照

当前约束：

- 一个项目只对应一种扫描模式
- 支持的项目扫描模式：
  - `multi_point`
  - `full_area`
- 项目开始后一直运行，直到用户停止
- 停止当前项目后，才能开启下一个项目
- 时间轴查看以 `project_id` 为范围，不跨项目混查

不使用 `task` 层。

---

## 3. 前端页面建议

建议前端拆成两个页面域：

1. 实时扫描页
   - 启动项目
   - 查看 PTZ 当前状态
   - 查看当前扫描结果
   - 停止当前项目

2. 历史回放页
   - 查看项目列表
   - 进入某个项目
   - 查看该项目的时间轴节点
   - 拖动时间轴查看某个时间点最近快照

注意：

- 实时页主要读 Redis 实时接口
- 历史页主要读 SQLite 历史接口
- 不要把“实时结果接口”和“历史时间轴接口”混用

---

## 4. 通用响应格式

所有接口统一返回：

```json
{
  "code": 0,
  "data": {},
  "msg": "ok"
}
```

失败时：

```json
{
  "code": 2002,
  "data": {},
  "msg": "已有运行中的项目: xxx，请先停止当前项目"
}
```

前端判断规则：

- `code === 0` 表示成功
- `code !== 0` 表示失败
- 错误提示统一使用 `msg`

## 5.2 初始 MAC 扫描（多点扫描前置步骤）

> 这是多点扫描（auto 模式）的**必要前置步骤**。
> 在当前云台朝向位置遍历所有信道/带宽配置，发现周围的 WiFi 设备。
> manual 模式可跳过此步骤。

### 5.2.1 启动初始扫描

`POST /api/v1/ptz/initial_scan/start`

请求体（均为可选）：

```json
{
  "dwell_time": 3.0
}
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `dwell_time` | float | 3.0 | 每个信道/带宽配置的监听时长（秒），范围 1-10 |

成功返回：

```json
{
  "code": 0,
  "data": {
    "queued": true,
    "message": "初始MAC扫描已启动",
    "config": {
      "dwell_time": 3.0,
      "configs_count": 69
    }
  },
  "msg": "ok"
}
```

重复调用返回（409）：

```json
{
  "code": -1,
  "msg": "初始扫描正在进行中，请等待完成"
}
```

前端注意：
- 启动前应先将云台移到目标观测方向（用户有意选择的位置）
- 扫描期间云台会自动校准再回到原位，前端无需干预
- 扫描耗时约 `69 × dwell_time` 秒（默认约 3.5 分钟）
- **不要重复调用**，后端已有防重入保护

### 5.2.2 停止初始扫描

`POST /api/v1/ptz/initial_scan/stop`

无请求体。

### 5.2.3 获取初始扫描结果

`GET /api/v1/ptz/initial_scan/result`

成功返回（有结果）：

```json
{
  "code": 0,
  "data": {
    "has_result": true,
    "mac_count": 156,
    "macs": {
      "14:51:7e:69:9e:d4": {
        "channel": 48,
        "bandwidth": "HT20",
        "rssi_avg": -61.2,
        "rssi_samples": 10,
        "omni_rssi_avg": -78.5,
        "omni_rssi_samples": 6
      }
    },
    "initial_position": {
      "pan": 90.0,
      "tilt": 0.0
    }
  },
  "msg": "ok"
}
```

无结果返回：

```json
{
  "code": 0,
  "data": {
    "has_result": false
  },
  "msg": "ok"
}
```

前端轮询建议：
- 启动初始扫描后，每 3-5 秒轮询此接口
- 当 `has_result` 为 `true` 且 `mac_count > 0` 时，表示扫描完成
- 可结合 `GET /api/v1/ptz/status` 判断状态（`initial_scan.active` 字段）

## 5.3 启动多点扫描项目

> **与初始扫描的关系**：
> - `auto` 模式：必须先完成初始扫描（5.2），否则返回 400 错误
> - `manual` 模式：不需要初始扫描，直接指定 MAC 即可

### 接口

`POST /api/v1/ptz/multi_point_scan/start`

### 请求体

```json
{
  "project_name": "仓库东侧多点排查",
  "mode": "manual",
  "manual_macs": [
    {
      "mac": "14:51:7e:69:9e:d4",
      "channel": 48,
      "bandwidth": "HT20"
    }
  ],
  "pan_range": [0, 90],
  "tilt_range": [-20, 20],
  "pan_step": 15.0,
  "tilt_step": 20.0,
  "dwell_time": 4.0,
  "extend_time": 3.0
}
```

### 关键字段

- `project_name`
  - 可选
  - 若不传，后端自动生成
- `mode`
  - `auto` 或 `manual`
- `manual_macs`
  - `manual` 模式必填

### 成功返回

前端必须读取：

- `data.config.project_id`
- `data.config.project_name`

示例：

```json
{
  "code": 0,
  "data": {
    "queued": true,
    "message": "多点扫描已启动（manual 模式，持续重复）",
    "config": {
      "project_id": "multi_point_1741770000_ab12cd",
      "project_name": "仓库东侧多点排查",
      "mode": "manual",
      "manual_macs_count": 1,
      "pan_range": [0, 90],
      "tilt_range": [-20, 20],
      "pan_step": 15.0,
      "tilt_step": 20.0,
      "estimated_points": 21
    }
  },
  "msg": "ok"
}
```

### 前端动作

- 保存 `project_id`
- 进入实时扫描页
- 开始轮询：
  - `GET /api/v1/ptz/status`
  - `GET /api/v1/ptz/multi_point_scan/result`
  - `GET /api/v1/history/projects/current`

## 5.4 停止多点扫描项目

### 接口

`POST /api/v1/ptz/multi_point_scan/stop`

### 返回

```json
{
  "code": 0,
  "data": {
    "queued": true,
    "message": "停止命令已发送",
    "project_id": "multi_point_1741770000_ab12cd"
  },
  "msg": "ok"
}
```

### 前端动作

- 收到成功后，前端可立即进入“正在停止”状态
- 当前系统通常会在几秒内停止，不会长时间卡住
- 但不要在 stop 接口返回后立刻判定“项目已结束”
- 应继续轮询 `GET /api/v1/history/projects/current`
- 直到：
  - `has_project = false`
  - 或当前项目变为 `STOPPED`

原因：

- 停止接口本身表示“停止请求已发送成功”
- 真正项目结束由 `ptz_control.py` 在扫描循环收尾后完成
- 收尾通常很快，但仍然是异步完成过程

## 5.5 启动全面扫描项目

### 接口

`POST /api/v1/ptz/full_area_scan/start`

### 请求体

旧单范围写法仍兼容，`pan_range` / `tilt_range` 表示目标工作范围：

```json
{
  "project_name": "仓库东侧全面扫描",
  "pan_range": [-20, 20],
  "tilt_range": [-10, 10],
  "dwell_time": 7.0,
  "scan_time_limit": 60,
  "time_interval": 60
}
```

多工作范围写法：

```json
{
  "project_name": "仓库东侧全面扫描",
  "work_ranges": [
    {
      "pan_range": [-20, 0],
      "tilt_range": [-10, 10]
    },
    {
      "pan_range": [5, 25],
      "tilt_range": [-8, 8]
    }
  ],
  "dwell_time": 7.0,
  "scan_time_limit": 60,
  "time_interval": 60
}
```

说明：

- 全面扫描不再接收前端步径控制，`pan_step` / `tilt_step` 不需要传。
- 后端每轮先执行 `precheck` 初筛粗扫，再按 `work_ranges` 顺序执行 `work` 工作区细扫。
- 初筛范围从 Redis `gimbal:default_config.work_pan_range` / `work_tilt_range` 读取。
- 如果 Redis 未提供 `work_pan_range` / `work_tilt_range`，初筛范围回退到 Redis `pan_range` / `tilt_range`。
- `scan_time_limit` 可选，单位分钟，表示一个扫描窗口持续多久。
- `time_interval` 可选，单位秒，表示一次扫描窗口/完整轮结束后等待多久再开始下一次。
- `scan_time_limit` / `time_interval` 可放在 JSON body，也可放 query string；body 优先级更高。

时间参数组合：

| 参数组合 | 行为 |
|----------|------|
| 都不传 | 完整执行一轮全面扫描后自动停止 |
| 仅 `scan_time_limit` | 在指定分钟数内连续扫描；时间到则停止，未完成轮次标记为 `PARTIAL` |
| 仅 `time_interval` | 每次完整执行一轮，等待指定秒数后继续下一轮，直到手动停止 |
| 两者都传 | 每个窗口按 `scan_time_limit` 持续扫描，窗口结束后等待 `time_interval` 秒，再进入下一窗口 |

query 示例：

```http
POST /api/v1/ptz/full_area_scan/start?scan_time_limit=60&time_interval=60
```

### 成功返回

前端必须读取：

- `data.config.project_id`
- `data.config.project_name`

可选读取：

- `data.config.precheck_range`：本轮使用的初筛范围
- `data.config.work_ranges`：归一化后的工作范围列表
- `data.config.scan_plan`：后端自动计算的阶段、步径、预计点位数

返回示例：

```json
{
  "queued": true,
  "message": "全面扫描已启动",
  "config": {
    "project_id": "proj_1710000000",
    "project_name": "仓库东侧全面扫描",
    "precheck_range": {
      "pan_range": [-39.8, 38.05],
      "tilt_range": [-18.75, 19.19],
      "source": "redis_gimbal_work",
      "fallback_source": "redis_gimbal"
    },
    "work_ranges": [
      {
        "pan_range": [-20.0, 0.0],
        "tilt_range": [-10.0, 10.0]
      }
    ],
    "scan_plan": [
      {
        "phase": "precheck",
        "area_index": null,
        "pan_range": [-39.8, 38.05],
        "tilt_range": [-18.75, 19.19],
        "pan_step": 15.57,
        "tilt_step": 7.59,
        "point_count": 42
      },
      {
        "phase": "work",
        "area_index": 0,
        "pan_range": [-20.0, 0.0],
        "tilt_range": [-10.0, 10.0],
        "pan_step": 2.5,
        "tilt_step": 2.5,
        "point_count": 81
      }
    ],
    "dwell_time": 7.0,
    "scan_time_limit": 60,
    "scan_time_limit_unit": "minutes",
    "time_interval": 60,
    "time_interval_unit": "seconds",
    "estimated_points": 123,
    "estimated_time_minutes": 947
  },
  "warning": "全面扫描本次扫描窗口将持续 60 分钟，窗口结束后间隔 60 秒再继续"
```

## 5.6 停止全面扫描项目

### 接口

`POST /api/v1/ptz/full_area_scan/stop`

### 前端动作

与多点扫描停止相同：

- 接口接收后立即把 `status.full_scan.active` 改为 `false`
- 先收到“已发送停止命令”
- 此时 `state="stopping"`、`terminal=false`，表示后台正在快速清理
- 再轮询当前项目状态；正常应在 2 秒内进入 `terminal=true`
- 新任务仍以 `terminal` 和后端 `active_scan_id` 门禁为准，不能只凭 `active=false` 立即启动

## 5.7 获取 PTZ 实时状态

### 接口

`GET /api/v1/ptz/status`

### 用途

- 实时页状态栏
- 当前扫描轮次与当前点位显示
- 当前云台位置显示

### `state` 字段枚举

| 值 | 含义 | 说明 |
|----|------|------|
| `IDLE` | 空闲 | 无任何扫描任务 |
| `CALIBRATING` | **限位点校准中** | 云台正在移动到 (0°, 0°) 消除累积误差，每轮扫描开始前自动触发 |
| `INITIAL_SCANNING` | 初始扫描中 | 正在执行初始 MAC 发现扫描 |
| `MULTI_SCANNING` | 多点扫描中 | 正在执行多点精细扫描 |
| `FULL_AREA_SCANNING` | 全面扫描中 | 正在执行全区域覆盖扫描 |
| `CLIENT_SCANNING` | 客户端扫描中 | S8 客户端扫描任务进行中，`client_scan` 附加字段含进度 |
| `AUTO_SCANNING` | 旧栅格扫描中 | 旧版自动栅格扫描（智能扫描内部使用） |
| `MOVING` | 移动中 | 云台正在执行单次移动指令 |

### `CALIBRATING` 状态附加字段

当 `state == "CALIBRATING"` 时，返回额外字段：

```json
{
  "state": "CALIBRATING",
  "position": {"pan": 45.0, "tilt": 0.0},
  "calibration": {
    "active": true,
    "context": "初始扫描开始前",
    "target": {"pan": 0.0, "tilt": 0.0}
  }
}
```

- `context`：本次校准的触发场景（用于显示提示文案）
- `target`：目标位置，固定为 `(0, 0)`

### `FULL_AREA_SCANNING` 状态附加字段

当 `state == "FULL_AREA_SCANNING"` 时，返回额外字段：

```json
{
  "state": "FULL_AREA_SCANNING",
  "position": {"pan": -20.0, "tilt": -10.0},
  "full_scan": {
    "active": true,
    "round": 1,
    "round_id": "full_1710000000",
    "total_points": 123,
    "current_point": 7,
    "phase": "work",
    "area_index": 0,
    "scan_time_limit": 60,
    "scan_time_limit_unit": "minutes",
    "time_interval": 60,
    "time_interval_unit": "seconds",
    "window_index": 1,
    "window_started_at": 1710000000.0,
    "window_deadline_at": 1710003600.0,
    "remaining_seconds": 3512
  }
}
```

| 字段 | 说明 |
|------|------|
| `round` | 当前全面扫描轮次序号 |
| `round_id` | 当前轮唯一 ID |
| `total_points` | 当前轮预计总点位数，包含初筛和所有工作范围 |
| `current_point` | 当前正在处理的点位序号 |
| `phase` | `"precheck"` 或 `"work"` |
| `area_index` | 工作范围下标；初筛阶段为 `null` |
| `scan_time_limit` | 当前扫描窗口时长，单位见 `scan_time_limit_unit`；未配置为 `null` |
| `time_interval` | 窗口/轮次间隔，单位见 `time_interval_unit`；未配置为 `null` |
| `window_index` | 当前扫描窗口序号，从 1 开始 |
| `remaining_seconds` | 当前限时窗口剩余秒数；未配置 `scan_time_limit` 时为 `null` |

当处于窗口间隔等待时，`full_scan.phase` 为 `waiting_interval`：

```json
{
  "state": "FULL_AREA_SCANNING",
  "full_scan": {
    "active": true,
    "phase": "waiting_interval",
    "rounds_completed": 3,
    "time_interval": 60,
    "time_interval_unit": "seconds",
    "next_window_in_seconds": 42
  }
}
```

### `CLIENT_SCANNING` 状态附加字段

当 `state == "CLIENT_SCANNING"` 时，返回额外字段：

```json
{
  "state": "CLIENT_SCANNING",
  "position": {"pan": 100.0, "tilt": -5.02},
  "client_scan": {
    "active": true,
    "current_point": 6,
    "total_points": 16,
    "completed_points": 5,
    "in_target": true
  }
}
```

| 字段 | 说明 |
|------|------|
| `current_point` | 当前正在处理的格子序号（移动中时比 `completed_points` 大1） |
| `total_points` | 本次扫描的总格子数（含 guard zone） |
| `completed_points` | 已完成采集的格子数 |
| `in_target` | 当前格子是否在用户指定的选区内 |

### 前端建议

轮询间隔建议：

- 扫描中 / 校准中：1 秒
- 非扫描中：2 到 3 秒

> **注意**：`CALIBRATING` 持续约 15-30 秒（云台物理移动时间）。前端应在 UI 上展示"正在校准"状态，避免用户误以为系统卡死。

## 5.9 获取全面扫描实时结果

### 接口

`GET /api/v1/ptz/full_area_scan/result`

### 用途

- 仅用于实时页
- 返回当前最新一轮全面扫描结果

### 当前返回关键字段

- `latest_round`
- `round_index`
- `round_id`
- `round_status`
- `stop_reason`
- `expected_points`
- `completed_points`
- `round_started_at`
- `round_finished_at`
- `scan_time_limit`
- `time_interval`
- `window_index`
- `remaining_seconds`
- `whitelist_key`
- `whitelist_count`
- `points_count`
- `total_mac_detections`
- `results`

`round_status` 枚举：

| 值 | 含义 |
|----|------|
| `RUNNING` | 当前轮正在扫描 |
| `SUCCESS` | 当前轮完整扫完 |
| `PARTIAL` | 当前轮未完整扫完，通常是手动停止、时间到或异常中断 |

`stop_reason` 常见值：

| 值 | 含义 |
|----|------|
| `completed` | 当前轮完整完成 |
| `time_limit` | `scan_time_limit` 到时 |
| `manual_stop` | 用户手动停止 |
| `incomplete` | 未完整扫完，但没有明确停止原因 |

返回外层示例：

```json
{
  "has_result": true,
  "source_results_key": "full_scan:round_2_results",
  "latest_round": 2,
  "round_index": 2,
  "round_id": "full_1710000000",
  "round_status": "PARTIAL",
  "stop_reason": "time_limit",
```

## 7. 当前快照数据含义

## 7.0 全向天线数据说明（S7 新增）

当后端启用了全向天线网卡（环境变量 `CAPTURE_OMNI_INTERFACE` 非空）时，所有扫描结果中会新增全向天线的 RSSI 数据字段。

新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `omni_rssi_avg` | float / null | 全向天线 RSSI 均值，null 表示未收到信号 |
| `omni_sample_count` | int | 全向天线采集到的样本数，0 表示未收到 |

- `omni_rssi_avg`: null
- `omni_sample_count`: 0
- `omni_status`: "disabled"

- 这些字段为可选参考数据，前端可选择性展示
- 定向 RSSI 仍然是主数据，全向 RSSI 仅作为距离参考基线
- 后端不做远近判断，前端/分析端可根据两路 RSSI 差值自行推断

## 8. 前端推荐调度流程

## 8.1 启动项目

1. 用户选择模式
2. 前端先调用 `GET /api/v1/history/projects/current`
3. 若已有运行项目，则禁止再次启动
4. 若无运行项目，则调用对应启动接口
5. 从返回里取 `project_id`
6. 页面进入实时扫描态

---

## 8.2 实时扫描中

前端每 1 秒轮询：

- `GET /api/v1/ptz/status`
- 多点模式：`GET /api/v1/ptz/multi_point_scan/result`
- 全面模式：`GET /api/v1/ptz/full_area_scan/result`
- `GET /api/v1/history/projects/current`

当项目运行时间较长时，可每 5 到 10 秒刷新一次：

- `GET /api/v1/history/projects/<project_id>/snapshots`

作用：

- 获取已经持久化完成的新轮次
- 给时间轴增加新节点

---

## 8.3 停止项目

1. 前端点击停止
2. 根据当前项目 `scan_type` 调对应 stop 接口
3. 前端立即显示“正在停止”
4. 按 1 秒轮询 `GET /api/v1/history/projects/current`
5. 当项目不再存在于 current 中，或状态已结束时，切换到历史查看态

---

## 8.4 查看历史项目

1. 调 `GET /api/v1/history/projects`
2. 用户选择某个项目
3. 调 `GET /api/v1/history/projects/<project_id>/snapshots`
4. 渲染时间轴
5. 用户拖动时间轴时调：
   - `GET /api/v1/history/projects/<project_id>/snapshot?timestamp=...`
6. 根据返回的 `snapshot.grid_data` + `snapshot.render_meta` 渲染

---

## 9. 前端必须注意的约束

1. 当前系统同时只允许一个运行中的项目。

3. 不要把实时接口结果直接当历史接口使用。

5. 当前快照已支持信号历史回放，但 `panorama_path` 和 `thumbnail_path` 还未完整接入全景 worker。
   当前历史页应优先围绕：
   - 点位
   - RSSI
   - MAC 分布
   来展示。

## 12. S7 全向天线变更摘要（2026-04-01）

本次后端新增了全向天线并行采集能力。以下是前端需要关注的变更点。

### 12.1 哪些接口的返回数据有变化

| 接口 | 变化位置 | 说明 |
|------|----------|------|
| `GET /api/v1/ptz/multi_point_scan/result` | `scan_results[]` 每条记录 | 新增 3 个字段 |
| `GET /api/v1/ptz/full_area_scan/result` | `macs{}` 每个 MAC 记录 | 新增全向 RSSI 字段，以及 `type/subtype/ssid` |
| `GET /api/v1/history/projects/<id>/snapshot` | `grid_data` 内部 | 同上，历史快照自动包含 |

### 12.2 多点扫描新增字段

每条 `scan_results` 记录新增：

```json
{
  "channel": 48,
  "bandwidth": "HT20",
  "target_mac": "14:51:7e:69:9e:d4",
  "rssi_avg": -61.2,
  "sample_count": 10,
  "status": "success",
  "omni_rssi_avg": -78.5,
  "omni_sample_count": 6,
  "omni_status": "success"
}
```

| 新增字段 | 类型 | 说明 |
|----------|------|------|
| `omni_rssi_avg` | float / null | 全向天线 RSSI 均值。null = 未收到信号或未启用 |
| `omni_sample_count` | int | 全向天线样本数。0 = 未收到 |
| `omni_status` | string | `"success"` / `"no_signal"` / `"channel_switch_failed"` / `"disabled"` |

### 12.3 全面扫描新增字段

每个 MAC 记录新增：

```json
{
  "channel": 6,
  "bandwidth": "HT20",
  "rssi_avg": -58.3,
  "rssi_samples": 15,
  "omni_rssi_avg": -72.1,
  "omni_rssi_samples": 8,
  "type": "ap",
  "subtype": "starlink",
  "ssid": "STARLINK-XXXX"
}
```

| 新增字段 | 类型 | 说明 |
|----------|------|------|
| `omni_rssi_avg` | float / null | 全向天线 RSSI 均值。null = 未收到或未启用 |
| `omni_rssi_samples` | int | 全向天线样本数。0 = 未收到 |
| `type` | string / null | 主类型：`"ap"` / `"client"` / `null`。星链不再作为独立 type |
| `subtype` | string / null | 子类型：星链 AP 为 `"starlink"`，其他为 `null` |
| `ssid` | string / null | AP 的 SSID；Beacon 明确隐藏时为保留值 `"_wildcard_"`，非 AP、未抓到 SSID IE 或无法解析时为 `null` |

> 设备身份缓存由 `capture_worker` 在程序生命周期内累积，程序重启后自然清空。后续点位或轮次识别出的 AP/SSID/星链信息会回填到当前轮已有 MAC 结果。

### 12.4 全面扫描两阶段点位字段

每个点位记录新增：

| 字段 | 类型 | 说明 |
|------|------|------|
| `round_index` | int | 全面扫描轮次序号 |
| `round_id` | string | 本轮唯一 ID，用于历史快照和图片目录关联 |
| `phase` | string | `"precheck"` 初筛粗扫 / `"work"` 工作区细扫 |
| `area_index` | int / null | `work_ranges` 下标；初筛阶段为 `null` |
| `scan_range` | object | 当前阶段使用的 pan/tilt 范围 |
| `scan_step` | object | 后端自动计算的 pan/tilt 步径 |

后续目标区域判断算法应基于同一 `round_id` 内的 `precheck` 与 `work` 数据做对比。

### 12.5 前端处理建议

1. 这些字段是**向后兼容**的新增字段，不影响已有逻辑
2. 前端可以选择性展示，比如：
   - 在点位详情弹窗中增加"全向 RSSI"一行
   - 在表格中增加一列
   - 用颜色/图标标注"仅定向可见"（全向为 null）的设备
3. `omni_status` 为 `"disabled"` 时表示后端未启用全向天线，前端可隐藏相关 UI
4. 判断远近的逻辑由前端/分析端自行实现，后端只提供原始双路数据

`discover_macs` 的结果（存储在 `multi_scan:target_macs`）中，每个 MAC 也新增了：

```json

```

这部分数据主要供后端内部使用，前端一般不直接读取，但如果有展示初始扫描结果的需求，注意新增字段即可。

## 13. auto 模式 MAC 列表行为说明（2026-04-01）

### 13.1 MAC 淘汰机制

auto 模式下，后端每隔 5 轮（`MULTI_SCAN_MAC_REFRESH_ROUNDS`）重新执行一次初始扫描刷新 MAC 列表。

刷新策略为**增删并行**：
- 新发现的 MAC → 追加到列表
- 连续 3 次（`MULTI_SCAN_MAC_MISS_THRESHOLD`）刷新扫描都未出现的 MAC → 从列表中移除

前端影响：
- 实时结果中的 MAC 数量可能随轮次变化（有新增也有淘汰）
- 前端展示实时结果时，应以当前返回的数据为准，不要缓存旧的 MAC 列表
- 被淘汰的 MAC 的历史数据仍保留在快照中，不会被删除

### 13.2 MAC 列表跨项目行为

- 停止多点扫描后，MAC 列表仍保留在 Redis 中
- 直接再启动 auto 模式 → 沿用上一轮 MAC 列表（不重新做初始扫描）
- 重新执行 `start_initial_scan` → 清掉旧列表并重新扫描

前端影响：
- 如果用户想"从头开始"，前端应引导用户先做初始扫描再启动多点扫描
- 如果用户想"继续上次"，可以直接启动多点扫描

### 13.3 刷新扫描位置

每次刷新 MAC 时，云台会自动回到首次初始扫描时的位置（用户手动移到的位置），确保每次刷新条件一致。前端无需关心此细节。

## 14. S8 客户端扫描接口（2026-04-09）

### 功能说明

客户端扫描（Client Scan）用于在用户指定的区域内逐格扫描，识别**正在发包的客户端设备（STA）**，并分析其关联的 AP 信息（BSSID、SSID）。

与多点扫描、全面扫描不同，客户端扫描：
- **单次任务，扫完自动结束，无需循环**
- **不写历史（无 project/snapshot）**，结果直接存入 Redis 临时 Key
- **自动扩展扫描范围**：在用户选区之外向四周各扩展 `CLIENT_SCAN_GUARD_STEPS`（默认1）步，防止边界误判

---

### 14.1 启动客户端扫描

**POST** `/api/v1/ptz/client_scan/start`

#### 请求参数（JSON Body）

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `pan_range` | `[float, float]` | ✅ | — | 用户指定的水平扫描范围 `[min, max]` |
| `tilt_range` | `[float, float]` | ✅ | — | 用户指定的垂直扫描范围 `[min, max]` |
| `pan_step` | `float` | ❌ | `10.0` | 水平步进（度） |
| `tilt_step` | `float` | ❌ | `10.0` | 垂直步进（度） |
| `channel` | `int` | ❌ | 不填 | 指定信道时固定单信道扫描；**不填则自动全频段跳扫**（遍历所有信道/带宽组合） |
| `bandwidth` | `string` | ❌ | `"HT20"` | 带宽，仅在指定 `channel` 时有效，可选 `"HT20"` / `"HT40+"` / `"HT40-"` |
| `dwell_time` | `float` | ❌ | 后端常量（3s）| 每个信道配置的驻留采集时长（1.0~30.0 秒） |

#### 响应（成功）

```json
{
  "code": 0,
  "data": {
    "queued": true,
    "message": "客户端扫描已启动",
    "config": {
      "pan_range": [90.0, 180.0],
      "tilt_range": [-10.0, 10.0],
      "pan_step": 10.0,
      "tilt_step": 10.0,
      "channel": 6,
      "bandwidth": "HT20",
      "dwell_time": null
    },
    "note": "扫描范围将自动向外扩展 guard_steps 步以防边界误判；扫完自动结束，无需手动停止"
  },
  "msg": "ok"
}
```

#### 错误码

| app_code | 含义 |
|----------|------|
| `1001` | `pan_range` 缺失或格式错误 |
| `1002` | `tilt_range` 缺失或格式错误 |
| `1003` | 参数类型错误 |
| `1004` | `dwell_time` 超出范围（1.0~30.0） |
| `1006` | 范围 min >= max |
| `2001` | 客户端扫描正在进行中（防重入） |

### 14.2 停止客户端扫描

**POST** `/api/v1/ptz/client_scan/stop`

无需请求体。

停止机制为**最终一致性**（非立刻中断）：
- 后端先写 Redis 标志位 `client_scan:stop`，当前点采集完毕后将在进入下一个点位前检查并退出
- 最大延迟 ≈ 当前点的剩余 `dwell_time`（默认 5s），如有 WiFi 包到达则更快

#### 响应

```json
{
  "code": 0,
  "data": {
    "queued": true,
    "message": "停止命令已发送"
  },
  "msg": "ok"
}
```

#### `clients` 字典字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `best_rssi` | `float` | 该客户端在所有点位中最强的 RSSI（dBm），按此降序排列 |
| `pan` | `float` | 信号最强时云台的水平角度 |
| `tilt` | `float` | 信号最强时云台的垂直角度 |
| `in_target` | `bool` | 最强信号点是否在用户选区（pan/tilt range）内 |
| `confidence` | `string` | **`"high"`** / **`"medium"`** / **`"low"`**，见下方说明 |
| `ap_bssid` | `string\|null` | 关联的 AP MAC 地址（来自 Assoc Req / Reassoc Req / Data(To DS=1) / Auth / Disassoc / Deauth 帧的 addr1），未知时为 `null` |
| `ap_ssid` | `string\|null` | 关联的 AP SSID。来源见 `ap_ssid_source`；未知时为 `null` |
| `ap_ssid_source` | `string\|null` | SSID 来源：`"assoc_req"`（来自关联请求帧）/ `"beacon"`（来自 AP Beacon 帧补全）/ `null`（未知） |
| `probe_ssids` | `list[string]` | Probe Request 中携带的 SSID 列表（可能有多个，含空字符串表示广播 Probe） |
| `status` | `string` | `"associated"`（已关联 AP）或 `"probing"`（只发 Probe Request）|
| `sample_count` | `int` | 定向天线采样到的帧数 |
| `omni_rssi_avg` | `float\|null` | 全向天线 RSSI 均值（未启用全向天线时为 `null`） |
| `ap_is_starlink` | `bool\|null` | 关联 AP 是否为星链设备（三特征全中判定）；`ap_bssid` 为 null 时此字段也为 `null` |

#### `confidence` 置信度说明

| 值 | 含义 | 建议前端展示 |
|---|---|---|
| `"high"` | 最强信号点在用户选区内，客户端很可能就在选区范围内 | 绿色/高亮 |
| `"medium"` | 最强信号点在扩展区（guard zone），但选区内也采集到过该客户端信号 | 黄色/提示 |
| `"low"` | 最强信号点在扩展区，选区内完全未采集到该客户端信号，可能是边界外干扰 | 灰色/低亮 |

> **前端建议**：优先展示 `confidence="high"` 的客户端；`"low"` 的可折叠或标注"可能在选区外"。

```
用户框选区域 → 填写信道/步径
    ↓
POST /client_scan/start
    ↓
轮询 GET /client_scan/status（每秒一次）
    → progress.current_point / completed_points / total_points 更新进度条
    → ptz_state == "CLIENT_SCANNING" 时显示云台移动状态
```

## 15. S9 星链设备识别接口

```json
    }
```

**前端建议**：
- 客户端列表中，`ap_is_starlink=true` 的条目可标注"星链用户"标签
- 独立查询 `/starlink/detected` 可展示当前环境中所有检测到的星链 AP，不依赖是否有客户端
- `detected_at` 可用于显示"X 分钟前检测到"等提示

### 15.3 典型前端调用流程

```
运行任意扫描（client_scan / multi_point / full_area）
    ↓ （后台自动采集 Beacon 并识别，无需额外命令）
扫描结束后（或扫描过程中任意时刻）
    ↓
GET /api/v1/starlink/detected
    → count > 0：展示星链设备列表（BSSID / SSID / 信道 / 检测时间）
    → count = 0：提示"当前环境未检测到星链设备"
```

**与其他扫描结果交叉匹配（前端自行处理）：**

```
拿到 /starlink/detected 的 BSSID 列表（Set）
    ↓
遍历 client_scan/result 的 clients：
    → 若 client.ap_bssid 在星链 BSSID Set 中 → 标注"星链用户"
遍历 multi_point/full_area 的 MAC 列表：
    → 若某条 MAC 在星链 BSSID Set 中 → 标注"星链 AP"
```

> **跨会话持久化说明**：
> - `starlink:detected_bssids` TTL 为 24 小时（Redis Key）
> - 后端同时维护 `starlink_whitelist.json` 持久化白名单文件（与脚本同目录）
> - 程序重启时自动将白名单预加载入 Redis，**无需重新扫描即可恢复上次识别结果**
> - 白名单只增不减（Starlink 设备不会"取消认定"），如需手动清空：
>   ```bash
>   redis-cli del starlink:detected_bssids
>   # 并删除 /home/ultiwill/ptz_capture_system/starlink_whitelist.json
>   ```
