## ⚠️ 必读：HTTP 响应格式与前端对接规范

> **本节是前端对接最容易踩坑的地方，AI 辅助开发时尤其要先读这里。**

### 所有接口统一响应包装

后端所有 API 的真实响应结构如下，**不是文档中的示例 JSON，而是再套了一层**：

```json
{
  "code": 0,
  "msg": "ok",
  "data": {
    // ← 文档中所有接口响应示例里的字段，都在这里
  }
}
```

| 字段 | 含义 |
|------|------|
| `code` | `0` = 成功；非 `0` = 业务错误（具体含义见各接口错误表） |
| `msg` | `"ok"` 或错误描述字符串 |
| `data` | 真正的业务数据，文档中所有示例 JSON 均指此处 |

**前端正确的取值方式：**

```javascript
// 以 Axios 为例（axios 本身还会额外套一层 .data）：
const res = await axios.post('/api/v1/ptz/full_area_scan/start', body)
const queued = res.data.data.queued          // ✅ 正确
const config = res.data.data.config          // ✅ 正确
// const queued = res.data.queued            // ❌ 错误，拿到 undefined
// const queued = res.queued                 // ❌ 错误，拿到 undefined

// 以 fetch 为例：
const json = await res.json()
const queued = json.data.queued              // ✅ 正确
```

> **如果不解包到 `.data`，读取 `config.project_id` 等嵌套属性时会得到 `TypeError: Cannot read properties of undefined`，导致后续 UI 状态（loading 关闭、弹窗关闭等）永远不会执行。**

### 轮询 `/api/v1/ptz/status` 时的注意事项

#### 1. `full_scan` / `location_scan` 子对象在任务刚启动时可能不存在

任务发出后的前几秒，Worker 正在初始化/校准，Redis 中的状态尚未更新，此时 `GET /api/v1/ptz/status` 的响应中 **`full_scan` 和 `location_scan` 这两个 Key 可能根本不存在**。

**必须使用可选链（Optional Chaining）访问，否则会报错：**

```javascript
const status = res.data.data  // 解包后的状态对象

// ✅ 安全访问
const fsState = status?.full_scan?.state        // 可能是 undefined
const lsPhase = status?.location_scan?.phase    // 可能是 undefined

// ❌ 危险访问（full_scan 不存在时直接崩溃）
const fsState = status.full_scan.state
```

#### 2. 顶层 `state` 和子对象 `state` 是两个完全不同的字段

| 字段路径 | 大小写 | 含义 | 示例值 |
|----------|--------|------|--------|
| `status.state` | **大写** | 云台当前动作状态 | `"IDLE"` / `"MOVING"` / `"FULL_AREA_SCANNING"` |
| `status.full_scan.state` | **小写** | 全面扫描任务生命周期 | `"running"` / `"stopping"` / `"stopped"` |
| `status.location_scan.state` | **小写** | 定位扫描任务生命周期 | `"running"` / `"stopping"` / `"stopped"` |

```javascript
// ❌ 错误：用顶层 state 判断任务生命周期
if (status.state === 'running') { ... }           // 这是云台动作，不是任务状态

// ✅ 正确：用子对象 state 判断任务生命周期
if (status.full_scan?.state === 'running') { ... }
```

#### 3. `state="IDLE"` 不代表所有任务都已停止

顶层 `state="IDLE"` 只表示**云台当前没有在移动**，与抓包任务和扫描任务是否结束无关。

```javascript
// ❌ 错误：用顶层 IDLE 判断任务结束或抓包结束
if (status.state === 'IDLE') setLoading(false)

// ✅ 正确：
// 判断全面扫描是否结束 → full_scan.terminal === true
// 判断抓包是否结束    → capture.running === false
```

### 推荐的统一状态读取函数

```typescript
function getScanLifecycle(scan?: {
  state?: string; phase?: string; scan_id?: string;
  active?: boolean; stop_requested?: boolean; terminal?: boolean;
}) {
  // state 优先；兼容旧前端：active=true → running, active=false → idle
  const state = scan?.state || (scan?.active ? 'running' : 'idle')
  const terminal = scan?.terminal === true ||
    ['stopped', 'done', 'error', 'idle'].includes(state)

  return {
    state,
    phase: scan?.phase,
    isActive: ['queued', 'running', 'stopping'].includes(state),
    isStopping: state === 'stopping' || scan?.stop_requested === true,
    isTerminal: terminal,
    canStart: terminal,
    canStop: state === 'queued' || state === 'running',
  }
}

// 使用示例：
const status = res.data.data          // 解包
const fs = getScanLifecycle(status?.full_scan)
const ls = getScanLifecycle(status?.location_scan)
```

## 2. 全面扫描启动接口

接口：

```http
POST /api/v1/ptz/full_area_scan/start
```

### 2.1 单工作范围写法

```json
{
  "project_name": "仓库东侧全面扫描",
  "pan_range": [-20, 20],
  "tilt_range": [-10, 10],
  "scan_time_limit": 60,
  "time_interval": 60
}
```

### 2.2 多工作范围写法

```json
{
  "project_name": "仓库东侧全面扫描",
  "work_ranges": [
    {
      "pan_range": [-20, 0],
      "tilt_range": [-10, 10]
    },
    {
      "pan_range": [5, 25],
      "tilt_range": [-8, 8]
    }
  ],
  "scan_time_limit": 60,
  "time_interval": 60
}
```

前端不要再向全面扫描接口传 `dwell_time`。后端会从 `config.json["全面扫描"]` 读取：

```text
粗扫每信道停留时长
细扫每信道停留时长
偏差区每信道停留时长
```

修改这些配置后需要重启后端。

### 2.3 全面扫描频段选择

启动参数 `wifi_mode` 只影响本次全面扫描任务：

| 请求值 | 本任务配置 |
|---|---|
| `"2.4"` | 信道 `1-13`，共 13 个 HT20 配置 |
| `"5"` | 信道 `36/40/44/48/52/56/60/64/149/153/157/161/165`，共 13 个 HT20 配置 |
| 缺失、`null`、`""` | 上述双频合并，共 26 个 HT20 配置 |

其他非空值返回参数错误且不会启动任务。配置在任务启动时锁定，任务内所有轮次复用；更换频段或天线后必须启动新任务。该参数不改变定位扫描、客户端扫描、独立抓包或 WiFi 连接。

```json
{
  "project_name": "仓库东侧全面扫描",
  "pan_range": [-20, 20],
  "tilt_range": [-10, 10],
  "wifi_mode": "5"
}
```

### 2.4 Query 写法也支持

`wifi_mode`、`scan_time_limit` 和 `time_interval` 也可以放在 query string：

```http
POST /api/v1/ptz/full_area_scan/start?wifi_mode=5&scan_time_limit=60&time_interval=60
```

如果 body 和 query 同时存在同名参数，**body 优先**。

### 2.5 时间参数组合语义

| 参数组合 | 行为 |
|----------|------|
| 都不传 | 完整执行一轮全面扫描后自动停止 |
| 仅 `scan_time_limit` | 在指定分钟数内连续扫描；时间到则停止，未完成轮次标记为 `PARTIAL` |
| 仅 `time_interval` | 每次完整执行一轮，等待指定秒数后继续下一轮，直到手动停止 |
| 两者都传 | 每个窗口按 `scan_time_limit` 持续扫描，窗口结束后等待 `time_interval` 秒，再进入下一窗口 |

接口：

```http
GET /api/v1/ptz/status
```

**前端感知全面扫描的主入口是 `status.full_scan`，不要只看顶层 `status.state`。**

扫描中返回示例：

```json
{
  "state": "FULL_AREA_SCANNING",
  "position": {"pan": -20.0, "tilt": -10.0},
  "full_scan": {
    "active": true,
    "state": "running",
    "scan_id": "full_1710000000000",
    "stop_requested": false,
    "terminal": false,
    "round": 1,
    "round_id": "full_1710000000",
    "total_points": 130,
    "current_point": 7,
    "phase": "fine_1",
    "area_index": 0,
    "scan_time_limit": 60,
    "scan_time_limit_unit": "minutes",
    "time_interval": 60,
    "time_interval_unit": "seconds",
    "window_index": 1,
    "window_started_at": 1710000000.0,
    "window_deadline_at": 1710003600.0,
    "remaining_seconds": 3512
  }
}
```

前端重点感知：

| 字段 | 含义 | 前端用途 |
|------|------|----------|
| `state` | **任务生命周期**：`queued` / `running` / `stopping` / `stopped` / `done` / `error` / `idle` | 判断任务是否在运行、正在停止、已结束 |
| `phase` | 当前扫描阶段：`coarse` / `fine_1` / `fine_2` / `deviation_a` / `deviation_b` / `waiting_interval` / `completed` / `stopped` | 展示当前扫描进度 |
| `scan_id` | 任务唯一标识 | 前端不需要使用，仅后端内部隔离用 |
| `stop_requested` | 是否已请求停止 | 配合 `state` 判断是否正在停止 |
| `terminal` | 是否为终态（`true` 后不会再变化） | 判断任务是否彻底结束，可以允许重新开始 |
| `round` | 当前扫描轮次 | 展示轮次 |
| `current_point` / `total_points` | 当前轮点位进度 | 展示进度条 |
| `remaining_seconds` | 当前限时窗口剩余秒数 | 展示倒计时 |

**全面扫描 `state` 生命周期：**

```
queued → running → stopping → stopped (手动停止)
queued → running → done      (正常完成)
queued → error               (启动失败)
```

| `state` | `active` | `terminal` | 前端展示 |
|---------|----------|------------|----------|
| `queued` | `true` | `false` | "排队中" |
| `running` | `true` | `false` | "扫描中"，展示 `phase` 进度 |
| `stopping` | `true` | `false` | **"正在停止"**，stop 按钮禁用 |
| `stopped` | `false` | `true` | "已停止"，允许重新开始 |
| `done` | `false` | `true` | "已完成"，允许重新开始 |
| `error` | `false` | `true` | 展示错误信息 |

窗口间隔等待时：

```json
{
  "state": "FULL_AREA_SCANNING",
  "full_scan": {
    "active": true,
    "phase": "waiting_interval",
    "rounds_completed": 3,
    "time_interval": 60,
    "time_interval_unit": "seconds",
    "next_window_in_seconds": 42
  }
}
```

前端可以用 `phase="waiting_interval"` 和 `next_window_in_seconds` 展示倒计时。

### 3.4 全面扫描白名单

接口：

```http
GET /api/v1/ptz/full_area_scan/whitelist
```

默认返回最新一个 `SUCCESS` 轮次的白名单。也可以按轮次读取：

```http
GET /api/v1/ptz/full_area_scan/whitelist?round_index=2
```

说明：

- 只有完整成功的 `SUCCESS` 轮次会生成白名单。
- `PARTIAL` 轮次不生成白名单，前端不要拿半轮数据做目标区域判断。
- 白名单按轮次保存，`source_results_key` 指向同一轮原始结果。
- 筛选逻辑按点位坐标是否落入 `work_ranges` 判断真实目标区，不按 `phase` 简单判断。
- 2026-06-26 起，白名单筛选会额外计算“白名单判定缓冲区”：它只用于筛选证据归类，不改变前端显示范围、扫描路径或原始点位。
- 默认要求真实目标区命中点数 `>=2`、白名单判定缓冲区命中点数 `>=3`、整轮命中点数 `>=4`。
- 默认要求缓冲区内最强 RSSI 至少比缓冲区外最强 RSSI 高 `2dB`；如果缓冲区外没有命中，则跳过该差值判断。
- 如果缓冲区外强点与缓冲区最强点差距在 `2dB` 内，并且覆盖至少 2 个相对/对角象限，该 MAC 会被拒绝并在 `rejected_macs[].metrics` 中记录触发象限和点位。
- 通过项含 `accept_reasons` / `accept_reason_codes`；拒绝项仍看 `rejected_macs[].failed_reasons` / `reason_codes` / `metrics`。
- 如果工作区点位存在全向 RSSI，默认要求定向 RSSI 至少比全向 RSSI 高 `5dB`。
- 如果没有全向 RSSI 数据，则跳过定向/全向差值判断，不会因为缺失全向数据过滤掉 MAC。
- 新全面扫描任务开始时会清理上一任务的 Redis 运行态轮次和白名单，历史回放请走历史项目接口。

### 4.1 启动定位扫描

接口：

```http
POST /api/v1/ptz/location_scan/start
```

请求示例：

自动探测信道/带宽：

```json
{
  "project_name": "仓库东侧定位",
  "target_macs": ["aa:bb:cc:dd:ee:ff", "11:22:33:44:55:66"],
  "pan_ranges": [[-20, 20]],
  "tilt_ranges": [[-10, 10]],
  "dwell_time": 5,
  "probe_dwell_time": 1,
  "probe_rounds_max": 2,
  "capture_time_limit": 180,
  "pcap_split_size_mb": 100,
  "time_interval": 300,
  "track_time_limit": 1440,
  "track_rssi_threshold": 5.0
}
```

固定信道/带宽：

```json
{
  "project_name": "仓库东侧定位",
  "target_macs": ["aa:bb:cc:dd:ee:ff", "11:22:33:44:55:66"],
  "pan_ranges": [[-20, 20]],
  "tilt_ranges": [[-10, 10]],
  "dwell_time": 5,
  "channel": 6,
  "bandwidth": "HT20"
}
```

多 MAC 分别指定信道/带宽：

```json
{
  "project_name": "仓库东侧定位",
  "target_macs": [
    "aa:bb:cc:dd:ee:ff",
    "11:22:33:44:55:66",
    "22:33:44:55:66:77"
  ],
  "pan_ranges": [[-20, 20]],
  "tilt_ranges": [[-10, 10]],
  "dwell_time": 5,
  "target_configs": [
    {
      "mac": "aa:bb:cc:dd:ee:ff",
      "channel": 6,
      "bandwidth": "HT20"
    },
    {
      "mac": "11:22:33:44:55:66",
      "channel": 149,
      "bandwidth": "HT40+"
    }
  ]
}
```

上例中 `22:33:44:55:66:77` 没有出现在 `target_configs` 里，后端会对它执行自动信道/带宽探测。

前端注意：

- 不再传 `pan_step` / `tilt_step`，后端自己算。
- `target_macs` 可以多个，后端会按信道/带宽分组共用点位扫描。
- 定位扫描的扩圈不会超过 Redis `gimbal:default_config.work_pan_range/work_tilt_range`；启动返回 `config.location_bound` 会给出后端实际采用的业务运行边界。
- 定位扫描不会只跟随 RSSI 强点完全偏离目标区域：中扫/细扫会把目标区域保底点追加到当前阶段点位里，`phase` 仍然显示为 `mid` / `fine`，不会新增额外 phase。
- 每个目标区域的保底点由“四角 + 中心 + 内部均匀点”组成；内部点默认 5 个。若某一轴跨度小于 1°，该轴不强行分点；两个轴都很小时只保留去重后的四角/中心。
- 传顶层 `channel` 时必须同时传顶层 `bandwidth`，否则启动接口返回 400。
- `target_configs[]` 中每一项都必须包含 `mac`、`channel`、`bandwidth`；其中 `mac` 必须存在于 `target_macs`，不能重复。
- 顶层 `channel` / `bandwidth` 不能和 `target_configs` 同时使用；需要多 MAC 不同信道/带宽时只用 `target_configs`。
- `bandwidth` 常用值为 `HT20` / `HT40+` / `HT40-`，前端需要按实际信道选择有效组合。
- `dwell_time` 是主扫描和最终确认的每点停留时长，默认 `1.0` 秒，影响 `coarse` / `mid` / `fine` / `locating` 阶段。
- `probe_dwell_time` 只影响 `channel_probe` 信道探测。
- `probe_rounds_max` 是信道探测最大轮数，避免不存在设备卡在开局。
- `capture_time_limit` 有值才抓包；不传或为 `null` 时不抓包。
- `time_interval`（**新增**）: 轮次之间的间隔时间（秒）。不传时只执行一轮；传了以后，当前轮完整结束并等待指定秒数，再进入下一轮。
- `track_time_limit`（**新增**）: 总追踪时长限制（分钟），只在传了 `time_interval` 的连续追踪场景下有意义；不传则持续追踪到用户手动停止。
- `track_rssi_threshold`（**新增**）: 快速校验允许的 RSSI 下降阈值，默认为 **5dB**。第二轮起，如果目标在上一轮最优点的 RSSI 没有明显下降，则通过 `fast_verify` 直接沿用位置并跳过大面积扫描；下降超过阈值才重新扫描定位。
- `fast_verify` 的停留时长后端固定为 6 秒，前端不需要传额外参数。

信道/带宽三种传法：

| 传法 | 请求字段 | 行为 |
|------|----------|------|
| 自动探测 | 不传 `channel`、`bandwidth`、`target_configs` | 所有 MAC 自动探测 |
| 任务级固定 | 顶层 `channel` + `bandwidth` | 所有 `target_macs` 共用同一组信道/带宽 |
| 按 MAC 指定 | `target_configs[]` | 指定 MAC 使用指定信道/带宽；未指定 MAC 自动探测 |

### 4.2 感知定位扫描进度（2026-06-08 更新）

接口：

```http
GET /api/v1/ptz/location_scan/status
GET /api/v1/ptz/status          ← 推荐同时读这个，status.location_scan 包含完整状态
```

**前端感知定位扫描推荐读 `status.location_scan`，不要只依赖 `phase` 或 `active`。**

`/api/v1/ptz/status` 中 `location_scan` 返回示例：

```json
{
  "location_scan": {
    "active": true,
    "phase": "coarse",
    "state": "running",
    "scan_id": "location_1710000000000",
    "stop_requested": false,
    "terminal": false,
    "current_point": 5,
    "total_points": 20
  }
}
```

**定位扫描 `state` 生命周期：**

```
queued → running → stopping → stopped (手动停止)
queued → running → done      (正常完成)
queued → error               (启动失败)
```

`phase` 可能取以下值（只表示阶段，不表示是否运行）：

| phase | 含义 |
|-------|------|
| `queued` | 启动接口已接受，命令已入队 |
| `channel_probe` | 信道探测：目标区域中心 + 四角，多 MAC 同时探测 |
| `fast_verify` | 第二轮起快速校验上一轮最优点 |
| `coarse` | 粗扫阶段：目标范围扩边后大步径扫描 |
| `mid` | 中扫阶段：收缩候选框 + 追加保底覆盖点 |
| `fine` | 细扫阶段：继续收缩 + 补完保底点 |
| `locating` | 最终定位：移动到每个 MAC 最强点确认 RSSI |
| `capturing` | 按 MAC 抓包中 |
| `waiting` | 连续追踪模式下等待 `time_interval` |
| `idle` | 任务未运行或已结束 |

| `state` | `active` | `terminal` | 前端展示 |
|---------|----------|------------|----------|
| `queued` | `true` | `false` | “排队中” |
| `running` | `true` | `false` | “定位中”，展示 `phase` |
| `stopping` | `true` | `false` | **”正在停止”**，stop 按钮禁用 |
| `stopped` | `false` | `true` | “已停止”，允许重新开始 |
| `done` | `false` | `true` | “已完成”，允许重新开始 |
| `error` | `false` | `true` | 展示错误信息 |

前端按钮建议：

| 状态 | stop 按钮 | start 按钮 |
|------|-----------|------------|
| `queued` / `running` | 可用 | 禁用 |
| `stopping` | **禁用**，显示”正在停止” | 禁用 |
| `stopped` / `done` / `error` / `terminal=true` | 隐藏 | 可用 |

### 4.3 获取定位扫描结果

接口：

```http
GET /api/v1/ptz/location_scan/result
```

结果按 MAC 分组，重点字段：

| 字段 | 含义 |
|------|------|
| `channel` | 该 MAC 探测到的信道 |
| `bandwidth` | 该 MAC 探测到的带宽 |
| `best_position` | 后端计算出的最优位置 |
| `best_rssi` | 最优位置定向天线 RSSI |
| `best_omni_rssi` | 最优位置全向天线 RSSI，可能为 `null` |
| `verify_rssi` | 最终定位确认定向天线 RSSI |
| `type` / `subtype` / `ssid` | MAC 类型、星链子类型和 AP SSID，含义见下一节 |
| `omni_verify_rssi` | 最终确认点的全向天线 RSSI，可能为 `null` |
| `all_points` | 定向扫描点位 RSSI 明细 |
| `omni_all_points` | 全向 RSSI 点位明细，可能为空 |
| `capture_status` / `capture_reason` | 抓包状态和原因，仅启用抓包时重点展示 |
| `pcap_files` | 如果启用抓包，该 MAC 对应的 pcap 分包列表；连续追踪时会跨轮次合并 |

注意：定位扫描的 `best_rssi` / `verify_rssi` / `all_points` 仍然只统计 `TA=目标 MAC` 的帧；只有启用 `capture_time_limit` 后生成的 `pcap_files` 会放宽过滤，只要目标 MAC 出现在 802.11 `addr1/addr2/addr3/addr4` 任意字段就保存，便于保留上下行和管理帧上下文。

### 4.4 指定点位抓包

接口：

```http
POST /api/v1/ptz/capture_at_best
```

用途：

- 前端根据自己持有的历史数据或当前页面数据，显式指定点位、MAC、信道、带宽后发起抓包。
- 后端不再读取 `location_scan:result` / `full_scan:results` 查最强点，也不依赖 Redis 历史扫描结果。
- 如果当前有正在运行的全面扫描或定位扫描，后端会按当前 `scan_id` 发出 stop。前端应先看到 `/api/v1/ptz/status` 中 `full_scan` 或 `location_scan` 进入 `state="stopping"`；worker 收尾完成后才会进入 `state="stopped"`、`terminal=true`。不要因为接口返回 `queued` 就认为原扫描已经停止。
- 后端把 `move_to_best_capture` 命令推给 PTZ 进程；PTZ 移动到指定点位后再启动抓包。
- 接口会立即返回排队成功和指定抓包信息，不等待云台移动完成，也不等待抓包完成。

请求示例：

```json
{
  "pan": 12.5,
  "tilt": -3.0,
  "mac": "aa:bb:cc:dd:ee:ff",
  "channel": 6,
  "bandwidth": "HT20",
  "capture_time_limit": 180,
  "min_free_memory_mb": 512,
  "min_free_disk_mb": 1024,
  "pcap_filename": "capture_aabbccddeeff.pcap"
}
```

参数说明：

| 参数 | 必填 | 含义 |
|------|------|------|
| `pan` | 是 | 抓包前云台要移动到的水平角度 |
| `tilt` | 是 | 抓包前云台要移动到的垂直角度 |
| `mac` | 是 | 要抓包的目标 MAC |
| `channel` | 是 | 抓包使用的信道 |
| `bandwidth` | 是 | 抓包使用的带宽，如 `HT20` / `HT40+` |
| `capture_time_limit` | 否 | 抓包时长，单位秒；不传或为 `null` 时持续抓包，直到调用 `/api/v1/capture/stop` |
| `min_free_memory_mb` | 否 | 抓包资源保护内存阈值，单位 MB；低于该值时停止或拒绝存盘 |
| `min_free_disk_mb` | 否 | 抓包资源保护磁盘阈值，单位 MB；低于该值时停止或拒绝存盘 |
| `pcap_filename` | 否 | 保存文件名；不传时后端自动生成 `capture_{mac}_{timestamp}.pcap` |

成功返回示例：

```json
{
  "queued": true,
  "message": "已发送移动+抓包命令，云台正在移动到指定点位",
  "target_position": {
    "pan": 12.5,
    "tilt": -3.0
  },
  "mac": "aa:bb:cc:dd:ee:ff",
  "channel": 6,
  "bandwidth": "HT20",
  "capture_time_limit": 180,
  "min_free_memory_mb": 512,
  "min_free_disk_mb": 1024,
  "pcap_filename": "capture_aabbccddeeff.pcap",
  "capture_target_mac_match": "any_addr"
}
```

前端交互建议：

- 按钮入口建议放在 MAC 详情、定位扫描结果、全面扫描白名单或前端历史结果列表中。
- 前端必须从自己持有的数据里带上 `pan/tilt/mac/channel/bandwidth`；后端不会再替前端查 Redis 历史结果。
- 调用成功后立即提示“已发送移动+抓包命令”，并继续轮询 `/api/v1/ptz/status`。
- `/api/v1/ptz/status` 中 `state` 是云台运动状态，不是抓包状态；`state="IDLE"` 只表示云台已经停住。
- 前端判断抓包是否仍在运行，优先看 `/api/v1/ptz/status.capture.running`；`capture_at_best` 主要用于展示指定点位抓包的上下文：
  - `capture.active=true`：指定点位抓包流程处于移动、配置网卡或实际抓包阶段。
  - `capture.running=true`：`capture_worker` 已经真正开始 Scapy 抓包。
  - `capture.phase="capturing"`：正在抓包或已经进入抓包阶段。
- `/api/v1/ptz/status` 里也可能出现 `capture_at_best.active=true`：
  - `state="MOVING"`：云台正在移动到指定点位。
  - `capture_at_best.phase="capturing"`：云台已到指定点位，抓包已交给 `capture_worker` 执行。
  - `capture_at_best.phase="stopped"`：抓包已通过停止接口结束。
- 抓包 pcap 过滤为 `capture_target_mac_match="any_addr"`，即目标 MAC 出现在 802.11 `addr1/addr2/addr3/addr4` 任意字段就保存。
- 如果传了 `min_free_memory_mb` / `min_free_disk_mb`，后端会在存盘前检查资源，并在限时抓包过程中持续检查；资源不足时会停止抓包并把原因写为 `low_memory` 或 `low_disk_space`。
- 不传 `capture_time_limit` 时，如需停止抓包，仍使用现有 `POST /api/v1/capture/stop`。

状态返回示例：

```json
{
  "state": "IDLE",
  "position": {
    "pan": 127.4,
    "tilt": 0.2
  },
  "capture": {
    "active": true,
    "running": true,
    "phase": "capturing"
  },
  "capture_running": true,
  "capture_at_best": {
    "active": true,
    "phase": "capturing",
    "mac": "4c:b7:e0:5f:45:0f",
    "channel": 2,
    "bandwidth": "HT20",
    "target_mac_match": "any_addr",
    "capture_time_limit": null,
    "pcap": "capture_4cb7e05f450f_1780560529.pcap"
  }
}
```

停止抓包：

```http
POST /api/v1/capture/stop?timeout=15
```

停止成功后，`capture.running=false`，`capture.phase="idle"`；如果当前状态仍带有 `capture_at_best`，其 `phase` 会变为 `stopped` 或 `completed`。

常见错误：

| HTTP 状态 | app_code | 含义 | 前端处理建议 |
|-----------|----------|------|--------------|
| 400 | 1001 | 缺少 `pan/tilt/mac/channel/bandwidth` 任一必填参数 | 提示补齐点位、MAC、信道和带宽 |
| 400 | 1002 | `pan/tilt/channel` 类型错误 | 检查数值类型 |
| 400 | 1003 / 1004 | `pan` 或 `tilt` 超出硬件限位 | 提示选择有效点位 |
| 400 | 1005 | `capture_time_limit` 类型错误或小于等于 0 | 检查抓包时长 |
| 400 | 1006 | `pcap_filename` 包含路径或非法名称 | 只传文件名，不传目录 |
| 400 | 1007 / 1008 | `min_free_memory_mb` 或 `min_free_disk_mb` 类型错误或小于 0 | 检查资源保护阈值 |

注意：

- 该接口不负责重新搜索目标 MAC，也不负责替前端选择最强点。
- 前端可以使用任意历史来源的点位和信道带宽，只要请求体把必填参数传全。
- 触发后会停止当前全面扫描/定位扫描。若原任务是定位扫描，前端应把 `phase="idle", status="stopped"` 视为被用户操作打断后的结束态。

## 4.5 天线偏差补偿（2026-06-12 新增）

### 4.5.1 概述

定向天线 RF 最强响应方向与前端全景图上的实际目标位置存在稳定偏差。后端通过天线偏差补偿自动修正：扫描结果中的 `position` / `best_position` 字段已经是修正后的视觉坐标，前端无需做任何额外处理。

### 4.5.2 偏差来源

偏差配置在 `config.json` 的 `"天线偏差补偿"` 块中。运行时目标距离从 Redis `gimbal:default_config.target_distance` 读取。如果该字段缺失或无效，补偿自动禁用，扫描行为等价于旧逻辑。

### 4.5.3 对前端的影响

**前端不需要改动**。核心约定：

| 字段 | 含义 | 前端是否需要改读 |
|------|------|----------------|
| `position` / `best_position` | 修正后的视觉坐标 | **不需要**，继续读这个 |
| `target_position`（capture_at_best） | 前端传入的 visual 点位 | **不需要** |
| `rf_position` / `rf_best_position` / `rf_target_position` | 原始 RF 坐标 | 仅调试，前端不需要读 |
| `antenna_bias` | 本次补偿参数 | 仅调试/展示可选 |

### 4.5.4 全面扫描结果

启用补偿时，每个点位结果新增调试字段：

```json
{
  "position": {"pan": 43.8, "tilt": 7.4},
  "rf_position": {"pan": 43.2, "tilt": 6.0},
  "antenna_bias": {"pan_bias_deg": 0.6, "tilt_bias_deg": 1.4}
}
```

- `position`：修正后的视觉坐标，前端继续读这个
- `rf_position`：原始 RF 扫描坐标，仅调试
- 未启用补偿时只有 `position`，无 `rf_position`

### 4.5.5 定位扫描结果

启用补偿时，每个 MAC 的结果新增调试字段：

```json
{
  "best_position": {"pan": 43.8, "tilt": 7.4},
  "rf_best_position": {"pan": 43.2, "tilt": 6.0},
  "antenna_bias": {"pan_bias_deg": 0.6, "tilt_bias_deg": 1.4}
}
```

- `best_position`：修正后的视觉最强点，前端继续读这个
- `rf_best_position`：原始 RF 最强点，仅调试
- 未启用补偿时只有 `best_position`，无 `rf_best_position`

### 4.5.6 capture_at_best

可选新增请求体参数 `target_distance`（单位米）：

```json
{
  "pan": 43.84,
  "tilt": 7.42,
  "mac": "aa:bb:cc:dd:ee:ff",
  "channel": 6,
  "bandwidth": "HT20",
  "target_distance": 220
}
```

| 有无 `target_distance` | 行为 |
|------------------------|------|
| 有 | 后端计算偏差，云台移动到纠偏后的 RF 点位，响应新增 `rf_target_position` + `antenna_bias` |
| 无 | 后端按前端传入 `pan/tilt` 直接移动，行为与旧逻辑完全一致 |

响应示例（有 `target_distance` 时）：

```json
{
  "target_position": {"pan": 43.84, "tilt": 7.42},
  "rf_target_position": {"pan": 43.2, "tilt": 6.0},
  "antenna_bias": {"enabled": true, "pan_bias_deg": 0.6, "tilt_bias_deg": 1.4}
}
```

### 4.5.7 WiFi 连接

可选新增请求体参数 `best_position` 和 `target_distance`：

```json
{
  "ssid": "TargetNetwork",
  "best_position": {"pan": 43.84, "tilt": 7.42},
  "target_distance": 220
}
```

| 组合 | 行为 |
|------|------|
| `best_position` + `target_distance` | 透传 `best_position` 和 `antenna_bias`，后续连接前移动到纠偏后 RF 点位 |
| 只有 `best_position` | 透传 `best_position`，不纠偏，直接按 visual 点位移动 |
| 都没有 | 不移动云台，原连接流程 |

`best_position` 必须同时有 `pan` 和 `tilt` 且为有限数值，否则忽略。

### 4.5.8 前端对接清单补充

- [ ] 全面扫描/定位扫描状态轮询不需要改动，`position` / `best_position` 仍是 visual 坐标
- [ ] `capture_at_best` 可选传 `target_distance`，不传时行为不变
- [ ] WiFi 连接可选传 `best_position` + `target_distance`
- [ ] `rf_position` / `rf_best_position` / `antenna_bias` 仅调试用，前端不需要读取
- [ ] Redis `gimbal:default_config.target_distance` 由上层业务后端写入，云台程序只读取

## 5. MAC 类型字段更新

全面扫描和定位扫描结果中，每个 MAC 可能包含：

```json
{
  "type": "ap",
  "subtype": "starlink",
  "ssid": "STARLINK-XXXX"
}
```

字段含义：

| 字段 | 含义 |
|------|------|
| `type` | 主类型：`ap` / `client` / `null` |

- 星链不再作为独立 `type`。
- 星链统一表示为 `type="ap"` + `subtype="starlink"`。

## 6. 摄像头拍照更新：HTTP 取图 + 过曝修复

本次摄像头部分包含两个更新：

1. **取图方式更新**
   - 后端 `camera_utils.py` 不再优先直接打开摄像头设备。
   - 正常情况下改为通过前端提供的 HTTP 摄像头服务获取截图。
   - 这样可以避免前端预览流、后端拍照进程同时抢占同一个摄像头设备。

2. **过曝修复**
   - 问题场景：摄像头从较暗室内转向明亮室外时，自动曝光还没调整完，直接拍照容易过曝。
   - 修复方式：`camera_worker.py` 拍照时调用 `PersistentCamera.capture_stable()`。
   - `capture_stable()` 会在拍照前短暂等待曝光稳定：
     - 每 50ms 获取一帧截图。
     - 计算灰度平均亮度。
     - 当前帧与上一帧亮度差小于阈值时，累计稳定帧数。
     - 连续 3 帧稳定后立即拍照。
     - 最多等待 500ms，超时后直接拍照兜底。

当前默认截图地址：

```text
http://127.0.0.1:8080/?action=snapshot
```

前端或摄像头服务需要保证：

| 地址 | 用途 |
|------|------|
| `http://127.0.0.1:8080/?action=snapshot` | 后端拍照取图，返回 JPEG/PNG 图片字节 |
| `http://127.0.0.1:8080/?action=stream` | 前端预览流 |

后端可通过环境变量覆盖截图地址：

```bash
CAMERA_HTTP_URL=http://127.0.0.1:8080/?action=snapshot
CAMERA_HTTP_TIMEOUT=5.0
```

### 前端如何感知摄像头更新

摄像头更新本身没有新增前端接口字段，前端主要通过以下方式感知：

1. **服务依赖**
   - 前端/摄像头服务必须先启动并可访问 `snapshot` 地址。
   - 后端拍全景图时会从这个地址取图片。

2. **过曝修复效果**
   - 全景图原始采样图在“室内暗处 → 室外亮处”切换时，应明显减少刚转过去第一张图过曝的问题。
   - 前端不需要新增字段判断，直接看最终全景图或原始采样图质量。
   - 这是拍照质量改善，不是接口结构变化。

3. **异常表现**
   - 如果 `snapshot` 服务不可用，后端会尝试回退旧的直连摄像头方式。
   - 回退方式可能更慢，也可能因为设备被占用而失败。

### 过曝修复参数

当前后端默认参数：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `max_wait_ms` | `500` | 最多等待曝光稳定 500ms |
| `stability_threshold` | `5` | 亮度变化阈值，范围 0-255 |
| `stable_frames` | `3` | 连续稳定帧数要求 |

这些参数目前在后端代码内默认配置，前端暂不需要传。

## 7. 前端如何测试

### 7.1 2026-05-27 全面扫描配置化 dwell 和三段优化怎么测

1. 启动全面扫描时只传 `pan_range/tilt_range` 或 `work_ranges[]`，不要传 `pan_step/tilt_step`，也不要传 `dwell_time`。
2. 确认启动返回里有 `scan_plan`、`dwell_time_source="config"` 和 `dwell_times`。
3. 确认 `dwell_times.coarse_dwell_time` / `fine_dwell_time` / `deviation_dwell_time` 与 `config.json["全面扫描"]` 一致。
4. 轮询 `/api/v1/ptz/status`，确认 `full_scan.phase` 会在 `coarse` / `fine_1` / `fine_2` ... / `deviation_a` / `deviation_b` / `waiting_interval` 之间变化。
5. 轮询 `/api/v1/ptz/full_area_scan/result`，确认点位结果中 `phase` 为 `coarse` / `fine` / `deviation_a` / `deviation_b`，并能看到 `round_status`、`stop_reason`、`window_index` 等字段。
6. 粗扫点位数应受上限 20 点限制（大范围自动放大步径）；细扫最多 2 轮；偏差区 B 最多 9 点且不含目标区内点。

### 7.2 2026-05-27 定位扫描连续追踪和 fast_verify 怎么测

1. 启动定位扫描时不传 `pan_step/tilt_step`。
2. 传多个 `target_macs`，确认后端能进入 `channel_probe` 并返回每个 MAC 的信道/带宽。
3. 传顶层 `channel` + `bandwidth`，确认返回的 `config.channel/config.bandwidth` 与请求一致，所有 MAC 结果使用同一组信道/带宽。
4. 传 `target_configs[]` 给部分或全部 MAC 指定不同信道/带宽，确认指定 MAC 的结果使用指定配置，未指定 MAC 仍会进入 `channel_probe` 自动探测。
5. 不传 `time_interval` 时，确认定位完成一轮后结束，不会自动进入下一轮。
6. 传 `time_interval` 时，确认一轮完成并等待指定秒数后进入下一轮；第二轮起状态中可能出现 `fast_verify`。
7. 轮询 `/api/v1/ptz/location_scan/status`，确认运行中 `phase` 出现在 `queued` / `channel_probe` / `fast_verify` / `coarse` / `mid` / `fine` / `locating` / `capturing` / `waiting` 中；任务停止后应为 `phase="idle", status="stopped"`，正常完成后应为 `phase="idle", status="done"`。
8. 等定位完成后请求 `/api/v1/ptz/location_scan/result`，确认每个 MAC 有 `best_position`、`best_rssi`、`verify_rssi`、`type/subtype/ssid`、`omni_verify_rssi`、`omni_all_points`。
9. 传 `capture_time_limit` 时，确认结果里对应 MAC 有 `capture_status` 和 `pcap_files`；连续追踪时 `pcap_files` 会跨轮次汇总。不传时不抓包。

### 7.3 2026-05-21 全面扫描白名单更新怎么测

1. 跑完整一轮全面扫描，确认该轮 `round_status="SUCCESS"`。
2. 请求 `GET /api/v1/ptz/full_area_scan/whitelist`，确认能拿到最新成功轮次白名单。
3. 请求 `GET /api/v1/ptz/full_area_scan/whitelist?round_index=N`，确认能按轮次读取白名单。
4. 请求 `GET /api/v1/ptz/full_area_scan/result?round_index=N`，确认能按轮次读取原始结果。
5. 修改 `config.json` 里的 `全面扫描筛选`，确认阈值变化会影响白名单筛选结果。

### 7.4 2026-06-04 capture_at_best 指定点位抓包怎么测

1. 前端准备一个已有点位和配置：`pan/tilt/mac/channel/bandwidth`。
2. 调用 `POST /api/v1/ptz/capture_at_best`，body 传 `pan/tilt/mac/channel/bandwidth`，可选传 `capture_time_limit` 和 `pcap_filename`。
3. 确认接口立即返回 `queued=true`、`target_position`、`mac`、`channel`、`bandwidth`、`capture_time_limit`、`pcap_filename`。
4. 轮询 `/api/v1/ptz/status`，确认先出现 `state="MOVING"` 和 `capture_at_best.active=true`，随后出现 `capture.active=true`；抓包线程真正启动后应出现 `capture.running=true`。
5. 确认云台最终停在返回的 `target_position` 附近，并且抓包已开始。
6. 注意 `state="IDLE"` 只表示云台停住；只要 `capture.running=true`，前端仍应展示抓包中。
7. 未传 `capture_time_limit` 时，调用 `POST /api/v1/capture/stop?timeout=15`，确认抓包可以正常停止；传了 `capture_time_limit` 时确认到时自动停止。

## 8. 统一状态模型（2026-06-08 新增）

### 8.1 概述

`ptz:current_status` 中的 `location_scan` 和 `full_scan` 子对象新增统一状态字段，旧字段保持兼容。

**旧前端**仍可兼容读取子对象里的 `active` / `phase`，以及顶层 `state` 作为云台运动状态；但任务生命周期不要再只靠这些字段判断。

**新前端**推荐读 `location_scan.state` / `full_scan.state` 判断任务生命周期，`active` / `phase` 保留兼容。

### 8.2 统一 state 枚举

所有扫描任务的子对象共用同一组 `state`：

| state | 含义 | `active` 值 |
|-------|------|-------------|
| `idle` | 任务未运行或已结束 | `false` |
| `queued` | 启动接口已接受，命令已入队 | `true` |
| `running` | 任务正在执行 | `true` |
| `stopping` | 停止请求已收到，worker 正在收尾 | `true` |
| `stopped` | 已停止（手动停止） | `false` |
| `done` | 已完成 | `false` |
| `error` | 启动参数错误或执行异常 | `false` |

**兼容规则**：

```text
state = running / stopping / queued  → active = true
state = stopped / done / error / idle → active = false
```

### 8.3 新增字段说明

`location_scan` 子对象：

```json
{
  "active": true,
  "phase": "coarse",
  "state": "running",
  "scan_id": "location_1710000000000",
  "stop_requested": false,
  "stop_requested_at": null,
  "terminal": false,
  "current_point": 5,
  "total_points": 20,
  "reason": null
}
```

`full_scan` 子对象：

```json
{
  "active": true,
  "phase": "coarse",
  "state": "running",
  "scan_id": "full_1710000000000",
  "stop_requested": false,
  "terminal": false,
  "round": 1,
  "current_point": 10,
  "total_points": 130,
  "remaining_seconds": 300
}
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `state` | string | 任务生命周期状态，见上表 |
| `scan_id` | string | 任务唯一标识，每次启动生成；capture 命令和 notify 携带此 ID 防止旧队列污染新任务 |
| `stop_requested` | bool | 是否已请求停止。`true` 表示 API 已收到停止请求，worker 可能还在收尾 |
| `stop_requested_at` | float/null | 停止请求时间戳 |
| `terminal` | bool | 是否为终态（stopped/done/error）。`true` 后不会再变化 |
| `reason` | string/null | 终态原因，如 `manual_stop`、`time_limit`、参数错误信息等 |

### 8.4 stop 语义变化

**旧语义**（已废弃）：stop API 直接写 `active=false`、`phase=stopped`。

**新语义**：

1. 前端调 stop API → 后端在对应扫描子对象中**立即**写 `state=stopping`、`stop_requested=true`，**`active` 保持 `true`**。
2. 前端看到对应扫描子对象 `state=stopping` → 显示"正在停止"。
3. worker 检测到停止标志并完成收尾 → 在对应扫描子对象中写 `state=stopped`、`active=false`、`terminal=true`。
4. 前端看到对应扫描子对象 `terminal=true` → 显示"已停止"。

这样前端可以区分"停止请求已收到但后台还在收尾"和"已完全停止"。

**2026-06-09 补充**：后端已补齐扫描任务和 capture_worker 之间的取消通信。

- 全面扫描 stop 后，已派发的 `discover_macs_for_full_scan` 会在收到命令、设置监听模式前后、每个信道切换前后、sniff 前后检查取消。
- 定位扫描 stop 后，`detect_channels` 和 `scan_at_point` 会检查 `location_scan:stop` / `capture:stop` / `scan_id`，旧任务会提前退出。
- capture_worker 被取消时会向对应 notify key 写 `status="stopped"`，让 PTZ worker 尽快完成收尾。
- 这不改变前端接口和字段。前端仍然只需要调用原 stop API，并轮询 `/api/v1/ptz/status`。
- stop 是"尽快停止"，不是物理瞬停。当前正在执行的云台移动、网卡 setup、信道切换或短片 sniff 可能会在最近可中断点退出。

### 8.5 前端判断建议

推荐封装统一函数：

```typescript
function getScanLifecycle(scan?: {
  state?: string; phase?: string; scan_id?: string;
  active?: boolean; stop_requested?: boolean; terminal?: boolean;
}) {
  // state 优先；兼容旧前端：active=true → running, active=false → idle
  const state = scan?.state || (scan?.active ? "running" : "idle")
  const terminal = scan?.terminal === true ||
    ["stopped", "done", "error", "idle"].includes(state)

  return {
    state,                              // 任务生命周期
    phase: scan?.phase,                 // 具体扫描阶段
    scanId: scan?.scan_id,              // 任务唯一标识（前端不需要传入）
    isActive: ["queued", "running", "stopping"].includes(state),
    isStopping: state === "stopping" || scan?.stop_requested === true,
    isTerminal: terminal,               // 任务已彻底结束，可以重新开始
    canStart: terminal,                 // 允许发起新任务
    canStop: state === "queued" || state === "running",
  }
}
```

使用示例：

```typescript
const status = await fetchPtzStatus()
const fs = getScanLifecycle(status.full_scan)
const ls = getScanLifecycle(status.location_scan)

// 全面扫描
if (fs.isStopping) showStopping()        // "正在停止"
else if (fs.isTerminal) showIdle()        // "已停止/已完成"
else if (fs.isActive) showProgress()      // 展示 phase 进度

// 定位扫描
if (ls.canStop) showStopButton()          // 可以点击停止
if (ls.canStart) showStartButton()        // 可以重新开始
```

**兼容说明**：
- `active` 是兼容字段，新前端优先使用 `state`
- `terminal=true` 是任务已收尾完成的强信号，比 `active=false` 更可靠
- `scan_id` 由后端生成，前端不要生成或传入

### 8.6 stop UX 指南

**stop API 返回成功只代表"停止请求已接收"**，不代表任务已经停止。

前端交互建议：

1. 点击 stop → 立即显示"正在停止"，禁用 stop 按钮
2. 轮询 `/api/v1/ptz/status`，看到对应子对象 `full_scan.state="stopping"` 或 `location_scan.state="stopping"` 时保持"正在停止"
3. 看到对应子对象 `terminal=true` 或子对象 `state` 进入终态（`stopped`/`done`/`error`）→ 显示"已停止"，启用 start 按钮
4. **不要连续狂点 stop**——后端按 `scan_id` 隔离，重复 stop 不会更快生效
5. 若长时间（>30s）未进入终态，可显示超时提示，但**不要自动把任务当 stopped**
6. 后端是"尽快停止"，不是物理毫秒级瞬停——云台移动、信道切换、抓包系统调用会在最近可中断点退出

### 8.7 任务级取消后的前端适配（2026-06-09 新增）

这次后端修改主要是**后端内部可靠性增强**，前端接口基本不变。前端需要调整的是状态解释和交互时机。

#### 8.7.1 全面扫描 stop

前端仍然调用：

```http
POST /api/v1/ptz/full_area_scan/stop
```

适配要求：

1. 请求成功后立即把 UI 切到"正在停止"，禁用 stop 按钮。
2. 继续轮询 `GET /api/v1/ptz/status`。
3. 看到 `full_scan.state="stopping"` 或 `full_scan.stop_requested=true` 时保持"正在停止"。
4. 只有看到 `full_scan.terminal=true`，或 `full_scan.state` 进入 `stopped` / `done` / `error`，才显示"已停止/已完成"并允许重新开始。
5. 不要因为顶层 `state="IDLE"` 就认为全面扫描已经完全停止；顶层 `state` 只代表云台当前运动状态。

后端现在会取消已派发的全扫抓包任务，所以 stop 后不应再长时间持续推进 `full_scan.phase`。但前端不要依赖"某个 phase 必须马上消失"，最终仍以 `terminal` / 子对象 `state` 为准。

#### 8.7.2 定位扫描 stop

前端仍然调用：

```http
POST /api/v1/ptz/location_scan/stop
```

适配要求：

1. 请求成功后立即把定位任务切到"正在停止"，禁用 stop 按钮。
2. 继续轮询 `/api/v1/ptz/status` 的 `location_scan` 子对象。
3. 看到 `location_scan.state="stopping"` 或 `location_scan.stop_requested=true` 时保持"正在停止"。
4. 只有看到 `location_scan.terminal=true`，或 `location_scan.state` 进入 `stopped` / `done` / `error`，才允许重新开始。
5. 不要用 `/api/v1/ptz/location_scan/status.phase` 单独判断任务是否结束；`phase` 是阶段，生命周期以 `/api/v1/ptz/status.location_scan.state` 为准。

后端现在会取消定位扫描内部的信道探测和点位抓包任务。前端不需要为了停止定位扫描内部抓包而额外调用 `/api/v1/capture/stop`；定位扫描 stop API 已经会设置对应 capture 停止标志。只有在停止独立抓包或 `capture_at_best` 持续抓包时，才调用 `/api/v1/capture/stop`。

#### 8.7.3 抓包状态和扫描状态的关系

`/api/v1/ptz/status` 里同时可能有：

```json
{
  "state": "IDLE",
  "full_scan": {"state": "stopped", "terminal": true},
  "location_scan": {"state": "idle", "terminal": true},
  "capture": {"running": true, "active": true, "phase": "capturing"}
}
```

前端解释规则：

| 判断目标 | 应读取字段 |
|----------|------------|
| 全面扫描生命周期 | `status.full_scan.state` / `status.full_scan.terminal` |
| 定位扫描生命周期 | `status.location_scan.state` / `status.location_scan.terminal` |
| 云台是否移动 | 顶层 `status.state` |
| 普通/指定点位抓包是否仍在进行 | `status.capture.running` 或 `status.capture.active` |

因此，`status.state="IDLE"` 不代表抓包已经停止；`capture.running=false` 才代表独立抓包已经停止。

#### 8.7.4 何时允许启动新任务

推荐规则：

```typescript
const canStartFullScan =
  getScanLifecycle(status.full_scan).isTerminal &&
  getScanLifecycle(status.location_scan).isTerminal &&
  status.capture?.running !== true

const canStartLocationScan =
  getScanLifecycle(status.full_scan).isTerminal &&
  getScanLifecycle(status.location_scan).isTerminal &&
  status.capture?.running !== true
```

如果前端允许"停止后立刻启动另一类任务"，至少要等当前任务的 `terminal=true`，并确认没有独立抓包仍在运行。后端已做旧命令隔离和 capture stop 标志清理，但 UI 上仍建议等终态后再开放 start，避免用户看到两个任务状态交叠。

### 8.8 scan_id 机制

每个扫描任务启动时生成唯一 `scan_id`：

- 定位扫描：`location_{timestamp_ms}`
- 全面扫描：`full_{timestamp_ms}`

存储在 Redis：

- `location_scan:active_scan_id`
- `full_scan:active_scan_id`

capture 命令（`discover_macs_for_full_scan`、`scan_at_point`、`detect_channels`）携带 `scan_id`。capture_worker 收到命令时校验 `scan_id` 是否匹配当前活跃任务，不匹配则跳过。

**作用**：防止重启、异常退出后旧队列中的命令污染新任务。

### 8.9 白名单淘汰明细（2026-06-08 新增）

全面扫描白名单结果新增 `rejected_macs` 字段，列出被筛选掉的 MAC 及原因：

```json
{
  "mac_whitelist": [...],
  "mac_count": 2,
  "rejected_macs": [
    {
      "mac": "aa:bb:cc:dd:ee:ff",
      "reason_codes": ["best_point_outside_work", "work_vs_other_delta_too_low"],
      "failed_reasons": ["最强点不在工作区", "工作区相对其他区域强度差不足"],
      "metrics": {
        "work_hit_count": 1,

  "candidate_mac_count": 3

```

| 字段 | 含义 |
|------|------|
| `reason_codes` | 稳定英文 code，前端可做国际化或筛选 |

| `metrics` | 筛选时的关键指标，前端可展示详情 |

`reason_codes` 枚举：

| code | 含义 |

| `active` | `state` | `active` 保留，新前端优先用 `state` |
| `phase` | `phase` | 不变，仍表示具体扫描阶段 |
| 顶层 `state` | `full_scan.state` / `location_scan.state` | 顶层 `state` 是云台运动状态，子对象 `state` 是任务生命周期 |

**新前端推荐**：用 `GET /api/v1/ptz/status` 作为主入口，读 `full_scan.state` / `location_scan.state` 判断任务生命周期。
