# 智能定位（T3）逻辑梳理与优化方向

> 本文档梳理当前 `start_location_scan` 命令的执行逻辑，标注现有设计的问题，并提出优化思路。

---

## 一、术语定义

| 术语 | 说明 |
|---|---|
| **目标 MAC** | 前端传入的需要定位的 MAC 地址列表 |
| **候选区域** | 前端传入的 `pan_ranges` / `tilt_ranges`，即每个目标 MAC 可能所在的初始粗略范围 |
| **信道探测** | 在已知候选范围内快速侦听，确定每个 MAC 正在使用的信道+带宽 |
| **逐级收缩** | 粗扫 → 中扫 → 细扫，每轮根据上一轮 RSSI 数据生成更小的候选框 |
| **最终确认** | 移动到每个 MAC 的最强点再采集一次，作为定位结果 |
| **expand_deg** | 扩边角度：每个候选区域在粗扫前四个方向各扩展的角度，默认 10° |

---

## 二、当前完整流程

```
输入：target_macs、pan_ranges/tilt_ranges（候选区域）
      可选：channel（固定信道）、capture_time_limit（抓包时长）
        │
        ▼
[0] 校准（去限位点，整个定位流程只做一次）
        │
        ▼
[1] 信道探测（channel probe）
   - 若前端已传 channel：跳过，直接用固定信道
   - 否则：在每个候选区域的 中心+四角（5 点）遍历全部信道
     - 使用 LOCATION_SCAN_CONFIGS，每个点 dwell = probe_dwell_time（默认 1s）
     - 最多探 probe_rounds_max 轮（默认 2 轮），找到即停
     - 探测到信道的 MAC → 记入 mac_channel_map
     - 仍未找到的 MAC → 标记 not_found，后续跳过
        │
        ▼
[2] 三轮逐级收缩扫描（scan_at_point，即定向锁定信道扫描）
   第一轮 coarse16：16° 步径，扩边后的候选区域，所有 MAC 共享点位
        │
   第二轮 mid8：8° 步径，根据第一轮 RSSI 数据生成收缩后的候选框
        │
   第三轮 fine4：4° 步径，进一步收缩候选框
   - 每轮结束后调 _build_fine_boxes：
     按 RSSI 取 Top N 强点，并要求不低于 best_rssi - 配置阈值
     过滤距离最强点过远的离群点后生成包围盒，再扩 expand_deg
     多 MAC 包围盒重叠则合并
        │
        ▼
[3] 最终定位确认（verify）
   - 每个 MAC 独立：移动到其 RSSI 最强点，锁定信道再扫一次
   - 记录 verify_rssi，更新 best_position
        │
        ▼
[4] 可选抓包（save_pcap）
   - 若传了 capture_time_limit：在最终定位点对目标 MAC 抓包
        │
        ▼
输出：location_scan:result（每个 MAC 的 best_position、best_rssi、信道等）
```

## 三、当前实现的问题与可优化点

### 3.1 信道探测效率低

**当前问题：**
- 探测点固定为"中心 + 四角"（5 点），与候选区域大小无关
- 探测点之间没有优先级，全部遍历完一轮才进入下一轮
- 每个点位使用 `LOCATION_SCAN_CONFIGS`（全信道），即使设备可能只在 2.4G 或 5G

**优化方向：**
- 探测点应按区域大小自适应：小区域（< 20°×20°）仅中心 1 点，大区域才用 5 点
- 若前次全面扫描的白名单中已有该 MAC 的信道记录，直接使用，不重新探测
- 2.4G / 5G 分两轮：先快速扫 2.4G 常用信道（1/6/11），再扫 5G

---

### 3.2 三轮收缩步径固定（16°→8°→4°），不自适应

**当前问题：**
- 步径与候选区域大小无关，始终 16°→8°→4°
- 候选区域很小（如 10°×10°）时，16° 步径只有 1~2 个点，粗扫几乎等于没做
- 候选区域很大（如 90°×40°）时，16° 步径仍可能漏掉信号峰值

**优化方向：**
- 粗扫步径根据候选区域大小动态计算，目标约 4~6 个点，范围 [8°, 20°]
- 细扫步径根据上一轮收缩后的包围盒大小动态计算，范围 [2°, 8°]
- 若收缩后包围盒 < 5°×5°，直接跳到最终确认，不再细扫

---

### 3.3 多 MAC 合并逻辑过于激进

**当前问题：**
- `_build_fine_boxes` 将重叠区域的 MAC 合并为一个包围盒共享路径
- 若两个 MAC 位置接近但不完全相同，合并后精度下降，且一个 MAC 定位完毕后另一个还没收缩

**优化方向：**
- 合并策略应可选：默认合并以节省时间，但合并阈值可配置（如 overlap > 60% 才合并）
- 合并后仍记录每个 MAC 的独立最强点，不共用 best_position

---

### 3.4 最终确认每个 MAC 独立移动，效率低

**当前问题：**
- `ls_target_macs` 中每个 MAC 单独移动到其最强点做确认
- 若多个 MAC 最强点位置接近，会反复移动到相似位置，浪费时间

**优化方向：**
- 最终确认前，对所有 MAC 的 best_pos 做聚合：将距离 < 5° 的 MAC 合并到同一确认点
- 在同一确认点一次发多个 MAC 的信道配置（如已有的 `_group_location_configs`）
- 一次采集返回后，各 MAC 分别记录自己的 verify_rssi

---

### 3.5 dwell_time 三轮统一，未区分粗/细/确认阶段

**当前问题：**
- 粗扫（16°）、细扫（4°）、最终确认均使用同一个 `ls_dwell`（默认 5s）
- 粗扫不需要那么长，5s × 每点 × 多 MAC 信道 × 大量点位 = 很慢

**优化方向（参考全面扫描的做法）：**
- 信道探测：`probe_dwell_time`（已有，默认 1s）
- 粗扫（coarse16）：`coarse_dwell_time`（建议 1~2s）
- 中扫/细扫（mid8/fine4）：`scan_dwell_time`（建议 2~3s）
- 最终确认（verify）：`verify_dwell_time`（建议 5s，精确定位）

---

### 3.6 抓包与定位串行执行，整体时间长

**当前问题：**
- 对每个 MAC 按顺序：定位确认 → 抓包 → 再下一个 MAC
- 若有 3 个 MAC 每个抓 60s，整体需要 3 × 60s = 3 分钟只在抓包

**优化方向：**
- 先完成所有 MAC 的最终确认（不抓包），输出所有 best_position
- 之后对所有 MAC 同时触发抓包（或由前端按需单独触发抓包，定位和抓包解耦）

---

### 3.7 停止响应不及时

**当前问题：**
- `_ls_stopped()` 只在点位循环开始前检查，在等待 `scan_at_point` 返回时（最长 30s+）不检查停止标志
- `_move_and_scan` 内部的 brpop 循环虽然有 `_ls_stopped()` 检查，但需要等 timeout=3s 才能响应

**优化方向：**
- 目前实现已经在 brpop 内部循环检查 `_ls_stopped()`，响应延迟 ≤ 3s，可接受
- 若需更快响应，可在发命令时也传 `stop_key`，让 capture_worker 主动中止采集

## 四、前端接口（当前不变）

### 输入命令（`start_location_scan`）

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `target_macs` | `[string]` | 必填 | 需要定位的 MAC 地址列表 |
| `pan_ranges` | `[[min,max]]` | 必填 | 每个 MAC 对应的水平候选范围 |
| `tilt_ranges` | `[[min,max]]` | 必填 | 每个 MAC 对应的垂直候选范围 |
| `dwell_time` | `float` | `5.0` | 每个点位每个信道的监听时长（秒） |
| `expand_deg` | `float` | `10.0` | 候选区域四周扩边角度（度） |
| `channel` | `int` | `null` | 固定信道（传了则跳过信道探测） |
| `bandwidth` | `string` | `"HT20"` | 固定带宽（配合 channel 使用） |
| `probe_dwell_time` | `float` | `1.0` | 信道探测每信道停留时长（秒） |
| `probe_rounds_max` | `int` | `2` | 信道探测最多轮次 |
| `capture_time_limit` | `float` | `null` | 抓包时长（秒），不传则不抓包 |
| `pcap_split_size_mb` | `float` | `100.0` | 单个 pcap 文件最大体积（MB） |

### 输出 Redis Key

| Key | 说明 |
|---|---|
| `location_scan:status` | 当前进度（phase、status、当前扫描点等） |
| `location_scan:result` | 最终结果（每个 MAC 的 best_position、best_rssi、信道等） |

### 结果结构（`location_scan:result`）

```json
{
  "ts": 1234567890.0,
  "results": {
    "aa:bb:cc:dd:ee:ff": {
      "status": "found",
      "channel": 6,
      "bandwidth": "HT20",
      "best_position": {"pan": 45.0, "tilt": -10.0},
      "best_rssi": -55.2,
      "verify_rssi": -54.8,
      "scan_points": 12,
      "all_points": {"45.0,-10.0": -55.2, "43.0,-10.0": -61.0}
    },
    "11:22:33:44:55:66": {
      "status": "not_found",
      "reason": "channel_probe_not_found"
    }
  }
}
```

**status 枚举值：**
- `found`：成功定位
- `not_found`：信道探测阶段未找到（设备可能不在候选区域或已离线）
- `no_signal`：信道找到但扫描阶段始终无信号
- `stopped`：用户手动停止

## 五、优先级建议

| 优化项 | 收益 | 改动复杂度 | 优先级 |
|---|---|---|---|
| 3.5 dwell_time 三阶段拆分 | 速度提升明显 | 低 | ⭐⭐⭐ 高 |
| 3.2 步径自适应 | 精度+速度 | 中 | ⭐⭐⭐ 高 |
| 3.1 信道探测优化（用白名单缓存） | 减少探测时间 | 低 | ⭐⭐ 中 |
| 3.4 最终确认聚合 | 减少移动次数 | 中 | ⭐⭐ 中 |
| 3.3 合并逻辑优化 | 精度提升 | 中 | ⭐ 低 |
| 3.6 抓包解耦 | 架构改善 | 高 | ⭐ 低 |

## 六、已完成的优化记录 (2026-05)

### 1. 连续追踪与无限循环定位机制
- **机制引入**：为 `start_location_scan` 接口新增 `time_interval` 和 `track_time_limit` 参数，允许在完成一轮定位和抓包后，自动休眠并开启下一轮，实现无人值守的持续信号追踪。
- **状态维护**：跨轮次在内存中维护 `mac_history`（上一轮的最佳位置、最佳信号强度、确定的信道和带宽）。

### 2. Fast Verify 快速校验 (秒级跨轮次定位)
- **问题解决**：针对目标设备可能在原地不动的情况，避免每轮都重复耗时的 16°/8°/4° 扫描。
- **优化逻辑**：如果存在上一轮历史记录，新一轮开始时优先切入 `fast_verify` 阶段，让云台花 2 秒钟直接移动到前次 `best_pos` 测 RSSI。
- **阈值判定**：引入 `track_rssi_threshold`（默认 5dB），如果当前 RSSI 相比上一轮跌幅在阈值内，直接跳过全套扫描，原地进入采集阶段；如果跌幅过大，则重新走大面积扫描找回目标。

### 3. PCAP 抓包自动分包与跨轮次累积
- **无缝追加**：利用底层 `RotatingPcapWriter` 支持 `append=True` 的特性，连续追踪多轮时写入同一个基础文件名（如 `part_001.pcap`）。
- **安全分包**：利用底层的 `max_bytes`（默认 100MB）自动切分 `part_002.pcap`，避免将设备存储写满。
- **结果汇总**：每轮最终返回给前端的 `result_payload` 中，`pcap_files` 字段会自动累计该 MAC 自第一轮开始产生的所有历史分包文件数组。

### 4. 设备类型与全向天线辅助增强
- **设备类型上报**：结果数据融合了 MAC 对应的 `type`（ap/client）、`subtype`（starlink）、`ssid`（无线名称）信息。
- **全向天线辅助**：结果数据增加了 `omni_verify_rssi` 和 `omni_all_points`，可配合定向天线数据做双源比对。

### 5. 防多径飞点与扫描上限控制
- **防飞点聚集**：细扫阶段如果出现多个强度接近但相距极远的多径信号点，现在会过滤掉距离绝对最强点 > 20° 的离群点，避免生成的候选框被拉扯得过大。
- **扫描点位兜底**：在极个别极大范围下，强制干预单框扫描点数上限（不超过 25 个点），防止扫描路径无限膨胀。

## 七、连续追踪与快速校验 (Fast Verify) 详细逻辑

为了支持“设备可能长期处于固定位置不动”的持续监听与抓包需求，引入了外层无限循环（While True）与 Fast Verify 机制。其完整执行流如下：

### 1. 第一轮（冷启动探测）
1. **输入解析**：解析 API 传入的 `time_interval` (轮次间隔) 和 `track_time_limit` (追踪总时长上限)。
2. **常规扫描**：执行完整的“信道探测 -> 粗扫 -> 中扫 -> 细扫”。
3. **定位与采集**：
   - 移动到计算出的绝对最强点 `best_pos`。
   - 若设置了 `capture_time_limit`，则在该点位停留并进行抓包，保存到 `part_001.pcap`。
4. **历史存档**：将每个找到的 MAC 对应的 `best_pos`、`best_rssi`、信道带宽等信息存入内存 `mac_history`。
5. **进入休眠**：推送当前轮次结果，进入 `time_interval` 秒的倒计时休眠（期间可被用户手动 Stop 打断）。

### 2. 第二轮及后续轮次（热启动追踪）
1. **触发 Fast Verify**：
   - 唤醒后轮次 `round_num > 1`，且 `mac_history` 中存在该设备的历史记录。
   - 云台直接转动到该设备上一轮的 `best_pos`，启动一个仅持续 **2秒** 的极简采集（跳过大面积扫描）。
2. **阈值判定**：
   - 获取 2 秒采集结果的平均 RSSI。
   - 比较：`当前 RSSI >= 上一轮 best_rssi - track_rssi_threshold (默认 5dB)`。
3. **判定成功（设备未移动）**：
   - **跳过扫描**：该 MAC 从本轮的后续“信道探测、粗/中/细扫”候选名单中剔除。
   - **直接采集**：无缝直达定位采集阶段。抓包程序会发现 `part_001.pcap` 存在，并利用 `append=True` 原地追加数据包（若超出 100MB 会自动切换到 `part_002.pcap`）。
4. **判定失败（设备已逃逸）**：
   - 清除该 MAC 在 `mac_history` 中的记录。
   - 该 MAC 重新降级为“未追踪状态”，跟随系统老老实实做本轮的粗/中/细扫描，重新锁定其新坐标。
5. **休眠与循环**：
   - 结果下发，汇总所有产生的 `pcap_files`。
   - 继续休眠 `time_interval` 秒，直至到达 `track_time_limit` 后结束整个任务。
