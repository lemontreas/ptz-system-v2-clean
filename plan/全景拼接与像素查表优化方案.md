# 全景拼接与像素查表优化方案

## Summary

本方案用于优化当前 Hugin 全景拼接后的像素映射链路。目标是在保持现有 HTTP 接口不变的前提下，让后端默认通过紧凑二进制像素映射文件进行查表，实现：

- 全景像素快速查到来源原图编号与原图像素坐标。
- 避免 CSV/SQLite 带来的生成慢、查表慢、文件大的问题。
- 测试阶段保留重叠区多候选信息，用于验证最终 owner 选择策略。
- 生产阶段优先保证生成速度、读取速度、程序可用性和文件大小控制。

## Current Stage Scope

2026-06-21 当前阶段只推进 **全景像素映射链路验证与 `.pmap` 正式路线**，暂不实施边界拼接优化。边界中心补拍仍保留在方案中，等待像素映射链路验证通过后单独评审。

本阶段明确不做：

- 不修改 `compute_shot_grid()` 的拍摄点生成逻辑。
- 不新增 `pan_min/pan_max` 边界补拍点。
- 不实现 `left_boundary` / `right_boundary` / `stitch_role`。
- 不做非对称裁剪。
- 不修改前端加灰或边界低可信 UI。
- 不因为边界补拍去改 per-shot intrinsics。

本阶段已知事实：

- 旧 CSV 偏差根因是旧 nona 改造把左上角像素坐标 `px,py` 直接传给 `SpaceTransform::transformImgCoord()`，导致约半幅水平偏差。
- 实际修复发生在 Linux 设备上：用户手动修改 Hugin `src/tools/nona.cpp` 并重新编译 nona，不是通过 Windows 工作区的 `patch_nona.py` 完成。
- Linux 手动修复包括加入 `panoCenterX/Y`、调用 `transformImgCoord()` 前转换为 Panorama Tools 中心坐标、使用 `px + 0.5 - center`、遍历 `pano.getNrOfImages()` 以避免边缘图缺失。
- 旧 `nona_pixel_map.csv` 和 sqlite 产物不可信。后续验证必须使用 Linux 重新编译后的 nona 生成的新 `.pmap`，短期 CSV 只能作为临时验证产物，不能继续扩展为正式链路。

## Optimization Modes

| 模式 | 用途 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| 边界中心补拍 | 全景采集与拼接输入优化 | 让扫描边界落在补拍照片的中心可信区，减少边缘畸变影响 | 增加少量拍摄点和拼接输入 | 待评审，当前阶段不实施 |
| Dense `.pmap` 查表 | 正式接口默认读取 | O(1) 查表，结构简单，生成和读取快，文件体积可控 | 每个像素只保存一个当前 owner | 采用 |
| 重叠区候选诊断表 | 测试阶段验证重叠区 owner 策略 | 可查看多个候选原图位置和几何评分 | 额外生成测试文件 | 采用 |
| 完整多候选表 | 所有有效像素都保存多候选 | 信息最完整 | 文件大，查询复杂，不利于生产 | 不采用 |
| CSV/SQLite | 历史验证方式 | 人可读，调试直观 | 文件大、生成慢、查表慢 | 移除 |

## Panorama Stitching Capture Optimization

当前全景采集已经通过裁剪输入图保留中心可信区，并设置重叠区来降低镜头边缘畸变对拼接的影响。但在扫描范围边界处，例如 pan 范围为 `0..180` 时，如果只有常规网格拍摄点，`0` 度和 `180` 度边界容易落在某些来源图的边缘区域。镜头边缘畸变、暗角、有效分辨率下降和重投影误差都会放大边界处的拼接误差。

后续可评审 **左右边界中心补拍** 作为下一阶段拼接优化：在全景范围的左右边界增加以边界角度为中心的照片，例如 `0..180` 范围额外拍摄 pan=`0` 和 pan=`180` 的中心对准照片。拼接时只使用这些照片朝向范围内部的一侧有效区域，裁掉范围外或不需要的一侧，只保留中心可信区和必要重叠区。

### 方案取舍

| 方案 | 做法 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| 不补拍边界 | 沿用当前网格点 | 拍摄点最少，耗时最低 | 范围边界可能由镜头边缘覆盖，边界畸变更明显 | 不推荐 |
| 左右边界中心补拍 | 每个 tilt 行额外拍 pan_min 和 pan_max | 明显改善左右边界可信度，新增点较少 | 水平拍摄次数增加 | 默认采用 |
| 四边界中心补拍 | 额外覆盖 pan_min/pan_max 和 tilt_min/tilt_max | 边界最完整，适合大垂直范围 | 拍摄点更多，拼接复杂度更高 | 本阶段不采用 |
| 外扩 overscan 后裁切 | 拍摄范围向外扩展，再裁回目标范围 | 边缘质量最好，拼接更自然 | 可能超出硬件限位，也会拍摄用户范围外内容 | 本阶段不采用 |

### 实施规则

- 对于每个参与拼接的 tilt 行，确保存在 pan=`pan_min` 和 pan=`pan_max` 的补拍照片。
- 左边界补拍照片只使用中心可信区中朝向范围内部的右侧区域；右边界补拍照片只使用朝向范围内部的左侧区域。
- 裁剪后仍保留必要重叠区，用于 Hugin/nona 建立连续映射和后续 `.pmap` 查表。
- 若边界角度超出硬件限位，则只使用限位内可拍摄的边界点，并在 metadata 中记录降级。

### 与 Hugin PTO 输入图的关联

边界中心补拍不是只在拍摄点列表中增加 pan=`pan_min` 和 pan=`pan_max`。要让补拍真正改善拼接边界，必须在 Hugin PTO 输入图生成前完成 shot 角色标记和非对称裁剪：

- 每个 shot metadata 需要记录 `stitch_role`：
  - `normal`：普通网格拍摄图。
  - `left_boundary`：左边界中心补拍图。
  - `right_boundary`：右边界中心补拍图。
- `normal` 继续使用现有对称中心裁剪策略。
- `left_boundary` 使用非对称裁剪，只保留朝向目标范围内部的右侧区域，并保留必要重叠区。
- `right_boundary` 使用非对称裁剪，只保留朝向目标范围内部的左侧区域，并保留必要重叠区。
- 非对称裁剪后必须同步修正相机内参：`cx -= crop_x`、`cy -= crop_y`；并把裁剪偏移、裁剪尺寸、原图尺寸、`stitch_role` 写入 `session.json`。
- PTO 中引用的必须是裁剪后的图片。如果 Hugin 仍使用对称裁剪图，边界补拍虽然存在，但边界区域仍可能主要来自镜头边缘，优化效果会明显下降。

因此，边界中心补拍的完整实施条件是：**拍摄点补充 + shot role 标记 + 非对称裁剪 + PTO 使用裁剪图**，四者缺一不可。

### 预期收益

- 扫描范围边缘不再主要依赖来源图边缘像素。
- 边界目标点击时更可能落在某张来源图的中心可信区域，后续 `.pmap` 查表更稳定。
- 减少边界区域因镜头边缘畸变导致的接缝错位、目标拉伸或像素映射偏差。
- 对 `0..180`、`90..270`、接近完整水平范围等场景尤其有价值。

## `.pmap` 字段定义

主 `.pmap` 采用单文件二进制格式，坐标系对齐最终显示的全景图像素。每个全景像素固定保存 4 个字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `owner` | `int16` | 当前全景像素最终选用的来源图编号，即 Hugin source image index。无来源时为 `-32768`。 |
| `source_x` | `int16` | 该全景像素映射到 `owner` 来源图中的 x 坐标，整数像素。无来源时为 `-32768`。 |
| `source_y` | `int16` | 该全景像素映射到 `owner` 来源图中的 y 坐标，整数像素。无来源时为 `-32768`。 |
| `coverage` | `uint16` | 覆盖该全景像素的来源图数量。`0` 表示无来源，`1` 表示非重叠区，`>1` 表示重叠区。 |

这里的 `source_x/source_y` 指 **Hugin/nona 实际参与拼接的输入图坐标**，也就是当前流程里的去畸变、可能裁剪后的 source image 坐标。若后续需要还原到完整去畸变图或原始 raw 图，需要在 metadata/header 中记录裁剪偏移和原图尺寸。

## Storage Layout

`.pmap` 使用 planar 二进制数组，而不是每像素 struct：

```text
PMAP header
owner[height][width]      int16
source_x[height][width]   int16
source_y[height][width]   int16
coverage[height][width]   uint16
```

选择 planar layout 的原因：

- C++ 侧可在内存中填充连续数组，最后一次性写入文件系统。
- Python 侧可用 `np.memmap` 或 `np.fromfile` 快速读取。
- 单字段分析简单，例如直接读取 `coverage` 判断重叠区。
- 文件大小固定为 `width * height * 8` 字节，不压缩也可控。

以 `3008 x 943` 全景为例，主 `.pmap` 原始体积约 21.6MB，远小于 CSV。

## Candidate Diagnostics

测试阶段额外生成重叠区候选诊断文件，只记录 `coverage > 1` 的像素候选关系。

候选诊断文件用于：

- 输入一个全景像素，查看它在多张原图中的候选来源。
- 展示每个候选的原图编号、原图像素坐标、拍摄 pan/tilt。
- 展示几何评分，例如中心距离、`center_norm`、rank、是否为中心最近候选。
- 辅助判断最终生产阶段应该使用哪种 owner 选择策略。

测试阶段主 `.pmap` 的临时 owner 先使用“离原图中心最近”策略，但这不作为最终生产结论。最终 owner 策略需要通过候选诊断工具和实测点确认后再确定。

## Implementation Decision

后续主路线采用 `.pmap` 查表，不再继续扩展 CSV/SQLite 路线。PTO 数学计算只作为诊断和大致候选参考，不作为最终像素级真值。

角度转像素的第一阶段采用进程内 PMAP 分桶反查索引，而不是 JSONL 点缓存或完整落盘反向索引：

- 索引生命周期：跟随 `_HuginPmapBatchContext` 进程内缓存，按 `pmap_path + session_json + mtime/size` 失效，最多保留少量最近使用的全景图。
- 索引结构：按 owner 分组，再按 source 坐标 bucket（默认 16 像素）保存 PMAP 有效像素的 `source_x/source_y -> pano_x/pano_y` 紧凑 NumPy 数组切片。
- 查询流程：`pan/tilt -> source 浮点像素 -> 当前 bucket/相邻 bucket 近邻查找`，窗口半径逐步扩大；仍无命中时使用最近 bucket 兜底并标记低可靠。
- 排序策略：多 owner 命中时 `source_error_px` 最小优先，`center_norm` 次之，`trusted_core` 作为辅助，不因边缘/接缝误差略大就拒绝返回。
- 权威边界：分桶索引只是 PMAP 的进程内加速结构，不是新的权威映射；`pixel_map.pmap` 始终是 Hugin 全景坐标转换的权威来源。

推荐实现顺序：

1. 先实现 `.pmap` 主文件和可选 `.pcand` 候选诊断文件的生成。
2. 再改后端读取 `.pmap`，保持现有 HTTP 接口不变。
3. 最后移除或默认关闭 CSV/SQLite 生成。

**已选定路线：C++ `nona` 直出 `.pmap`**，已完成编译部署（见本文末"nona C++ 实现现状"章节）。Python coordinate-layer 路线已放弃，不再作为选项。

## nona Transform Coordinate Notes

如果 `.pmap` 由修改后的 `nona` 直接生成，新的二进制写入逻辑必须继承这条修正：`SpaceTransform::transformImgCoord()` 接收的是 Panorama Tools 坐标，原点在全景画布中心，不是左上角像素坐标。生成映射时不能直接传 `px,py`。

`pixel_center_offset` 需要在实现后验证，不能把 `+0.5` 未经测试地写死为最终生产值。实现时至少支持测试 `0.0` 和 `0.5` 两种像素中心约定：

```cpp
const double pixelCenterOffset = 0.5; // 实测后确定，也必须支持测试 0.0
const double panoX = static_cast<double>(fullX) + pixelCenterOffset - fullWidth / 2.0;
const double panoY = static_cast<double>(fullY) + pixelCenterOffset - fullHeight / 2.0;
transform.transformImgCoord(srcX, srcY, panoX, panoY);
```

其中 `fullX/fullY` 是 PTO full canvas 坐标。若 `.pmap` 坐标系对齐最终显示图，则应遍历 crop 后的输出像素 `(x,y)`，先换回 full canvas：

```cpp
const int fullX = cropLeft + x;
const int fullY = cropTop + y;
```

再按上面的中心原点公式调用 transform。这样 `.pmap` 的数组下标就是最终全景图像素，后端查询时不再需要 CSV 路线里的 full canvas 转换。

`pixel_center_offset` 验证要求：

- 用同一 PTO 分别生成 offset=`0.0` 和 offset=`0.5` 的映射。
- 抽样非重叠区、重叠区、左边界、右边界像素。
- 对比 Hugin/nona 坐标层输出、候选诊断工具标注结果或人工可确认的原图目标点。
- 记录 source 像素残差的中位数、最大值和边界点误差。
- 选择整体残差更小、边界无系统偏移的一版作为生产默认。
- `.pmap` header/metadata 必须记录最终采用的 `pixel_center_offset`。

`.pmap` 生成时应遍历 PTO 中的 Hugin source image index，并在 metadata 中记录：

- `coordinate_system`: `final_cropped_panorama_pixels`
- `full_canvas`: `[fullWidth, fullHeight]`
- `crop`: `[cropLeft, cropRight, cropTop, cropBottom]`
- `pixel_center_offset`: `0.0` 或 `0.5`，按实测结果记录
- `pixel_center_convention`: `x+pixel_center_offset-center`
- `source_index_base`: `zero_based_hugin_image_index`
- `owner_policy`: 例如 `nearest_source_center`

**当前采用 C++ 直接输出 `.pmap` 路线**，已完成 nona.cpp 修改和编译。Python coordinate-layer 路线（用未修改的 nona 重投影再转换）已放弃，不再使用。

## Assumptions

- `source_x/source_y` 使用整数像素，不保存亚像素小数。实际场景最大分辨率 1920×1080，最多 20 张来源图，均在 `int16` 范围（−32767~32767）内，无溢出风险。
- `.pmap` 不压缩，以生成速度和读取速度优先。
- 无效 `int16` 值统一为 `-32768`（`PMAP_INVALID_INT16`）。有效 `source_x/source_y` clip 至 −32767~32767，不会与无效值混淆。
- `coverage=0` 表示无来源。
- 对外接口保持原样，不新增公开参数。
- CSV/SQLite 路线完全移除，不再作为后续验证路径。

### 实现方式

选用 **C++ nona 直出 `.pmap`** 主路线（不走 Python coordinate-layer 过渡方案）。

在 Linux 设备 `ultiwill@yuntai` 上，对 Hugin 2024.0.1 源码（`~/src/hugin-nona-map/hugin-2024.0.1+dfsg/src/tools/nona.cpp`）进行了以下改动：

1. 新增 `#include <vector>` / `#include <limits>` / `#include <cstdint>`
2. 添加 `--pixel-map FILE` 命令行参数（enum、longOptions、switch case、变量声明）
3. 在 `.run()` 之后，将原 CSV 逐行写入块替换为 **单遍内存数组 + 一次 fwrite 二进制写入块**

### 核心生成逻辑（已写入 nona.cpp，nona.cpp L615–L680）

```cpp
if (!pixelMapFile.empty()) {
    const int panoW = opts.getWidth();
    const int panoH = opts.getHeight();
    const int pixelCount = panoW * panoH;
    const double panoCenterX = static_cast<double>(panoW) / 2.0;
    const double panoCenterY = static_cast<double>(panoH) / 2.0;
    const int16_t PMAP_INV = (int16_t)(-32768);

    // 内存中分配四个 planar 数组
    std::vector<int16_t>  owner_arr(pixelCount, PMAP_INV);
    std::vector<int16_t>  src_x_arr(pixelCount, PMAP_INV);
    std::vector<int16_t>  src_y_arr(pixelCount, PMAP_INV);
    std::vector<uint16_t> cov_arr(pixelCount, 0);
    std::vector<float>    best_dist(pixelCount, std::numeric_limits<float>::max());

    // 单遍遍历：coverage 累加与 owner 决策同时完成，不需要两遍遍历
    for (unsigned int imgNr = 0; imgNr < pano.getNrOfImages(); ++imgNr) {
        const HuginBase::SrcPanoImage srcPanoImg = pano.getImage(imgNr);
        const vigra::Size2D srcSize = srcPanoImg.getSize();
        const float cx = static_cast<float>(srcSize.width())  / 2.0f;
        const float cy = static_cast<float>(srcSize.height()) / 2.0f;
        HuginBase::Nona::SpaceTransform transf;
        transf.createInvTransform(srcPanoImg, opts);
        for (int py = 0; py < panoH; ++py) {
            for (int px = 0; px < panoW; ++px) {
                double sx, sy;
                // 关键：转为以全景画布中心为原点的 Panorama Tools 坐标
                const double panoX = static_cast<double>(px) + 0.5 - panoCenterX;
                const double panoY = static_cast<double>(py) + 0.5 - panoCenterY;
                if (transf.transformImgCoord(sx, sy, panoX, panoY)) {
                    if (sx >= 0.0 && sy >= 0.0 &&
                        sx < srcSize.width() && sy < srcSize.height()) {
                        const int idx = py * panoW + px;
                        cov_arr[idx]++;                          // coverage 单遍内准确累加
                        const float dx = static_cast<float>(sx) - cx;
                        const float dy = static_cast<float>(sy) - cy;
                        const float dist = dx*dx + dy*dy;        // 平方距离，避免开方
                        if (dist < best_dist[idx]) {             // 离来源图中心最近策略
                            best_dist[idx] = dist;
                            owner_arr[idx] = static_cast<int16_t>(imgNr);
                            src_x_arr[idx] = static_cast<int16_t>(std::lround(sx));
                            src_y_arr[idx] = static_cast<int16_t>(std::lround(sy));
                        }
                    }
                }
            }
        }
    }

    // 构造 JSON metadata（含 width/height 运行时值）
    std::string meta =
        std::string(R"({"schema":"pmap_dense_planar_v1","width":)") +
        std::to_string(panoW) + R"(,"height":)" + std::to_string(panoH) +
        R"(,"invalid_int16":-32768,"layout":["owner:int16","source_x:int16","source_y:int16","coverage:uint16"],"coordinate_system":"full_canvas_pixels","pixel_center_convention":"x+0.5-center","source_index_base":"zero_based_hugin_image_index","owner_policy":"nearest_source_center"})";

    // 一次性 fwrite 写入 .pmap（header + metadata + 4 个 planar 数组）
    const uint64_t metaLen = static_cast<uint64_t>(meta.size());
    std::ofstream pmapFile(pixelMapFile, std::ios::binary);
    const char pmap_magic[8] = {'P','M','A','P','0','0','0','1'};
    const uint16_t pmap_ver=1, pmap_flags=0, pmap_res=0;
    const uint32_t pmap_w=static_cast<uint32_t>(panoW), pmap_h=static_cast<uint32_t>(panoH);
    const int16_t  pmap_inv=PMAP_INV;
    pmapFile.write(pmap_magic, 8);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_ver),   2);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_flags), 2);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_w),     4);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_h),     4);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_inv),   2);
    pmapFile.write(reinterpret_cast<const char*>(&pmap_res),   2);
    pmapFile.write(reinterpret_cast<const char*>(&metaLen),    8);
    pmapFile.write(meta.c_str(), static_cast<std::streamsize>(metaLen));
    pmapFile.write(reinterpret_cast<const char*>(owner_arr.data()),  pixelCount*2);
    pmapFile.write(reinterpret_cast<const char*>(src_x_arr.data()),  pixelCount*2);
    pmapFile.write(reinterpret_cast<const char*>(src_y_arr.data()),  pixelCount*2);
    pmapFile.write(reinterpret_cast<const char*>(cov_arr.data()),    pixelCount*2);
}
```

### 关键设计说明

| 要点 | 说明 |
|---|---|
| 单遍遍历 | `cov_arr[idx]++` 与 `best_dist` 比较在同一循环内完成，不需要两遍，不增加时间 |
| 坐标原点 | `px + 0.5 - panoCenterX` 对齐 Panorama Tools 约定，继承自已验证的 CSV 坐标修正 |
| owner 策略 | 离来源图中心最近（`dx²+dy²` 最小），用平方避免开方，性能更优 |
| 内存用量 | 运行时：4个数组(8字节/像素)+best_dist辅助(4字节/像素) ≈ **32 MB**；写入文件仅 8字节/像素 ≈ **21.6 MB**，生成完毕后释放 |
| 写入方式 | 遍历结束后一次性 `fwrite`，无中间 I/O 压力 |
| 格式兼容 | header 与 `pmap_utils.py` 的 `PMAP_HEADER` struct 对齐，Python 侧可直接用 `load_pmap()` 读取 |

### ⚠️ 仍需验证：pixel_center_offset 是否为 0.5

当前写死为 `+0.5`（像素中心约定）。如果 Hugin 内部 `transformImgCoord` 已经内置了 0.5 偏移，则实际等效于 `+1.0`，会导致所有 source 坐标整体偏移半像素。

**验证方法：**

```bash
# 用同一 PTO 生成两份 .pmap，offset 分别为 0.0 和 0.5
# 在 nona.cpp 里临时把 0.5 改为 0.0，重新编译，生成第二份
python3 - <<'PY'
from pmap_utils import load_pmap
p0 = load_pmap("pixel_map_offset0.pmap")
p5 = load_pmap("pixel_map_offset05.pmap")
# 对比非重叠区抽样点的 source_x/source_y 与人工可确认目标点的差异
PY
```

验证通过后，在 `metadata` 的 `pixel_center_convention` 字段中记录最终采用值（`x+0.5-center` 或 `x+0.0-center`）。
