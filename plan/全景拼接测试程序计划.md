> **运行位置**：云台设备端 Linux 环境，不是在当前 Windows 工作区直接运行。

## 一、背景与核心问题

当前全景拼接在多张图组合时容易出现重影、接缝错位或局部偏移。这个问题会直接影响前端点选和坐标映射：

```text
如果全景图里同时出现真实位置和旧位置虚影，
即使 pan/tilt 映射公式本身正确，
用户点到虚影时也会得到错误方向。
```

因此本阶段的核心目标不是先追求全景图观感最平滑，而是：

```text
优先消灭重影和不存在的旧位置，让全景图成为可点击、可定位的工程底图。
```

---

## 二、重要物理约束

本设备的云台水平转动不是理想的“相机绕镜头光心原地旋转”。

实际结构更接近：

```text
相机/天线绕云台圆心做带半径的圆周运动。
```

这意味着每张图片之间不仅有旋转，还有平移：

```text
相邻图片关系 = rotation + translation
```

该结构会带来视差问题：

1. 近处物体和远处背景无法同时完美对齐。
2. 对齐近景时远景可能重影。
3. 对齐远景时近景可能重影。
4. OpenCV Stitcher 只能选择一个相对合理的对齐层，无法从根本上消除所有深度的视差。
5. 张数越多，局部误差越容易累积。

所以本计划不把问题简单归因于“OpenCV 不够好”，而是按设备真实几何条件设计实验。

## 六、采集流程

测试程序负责采图，但调用原程序移动能力：

1. 根据命令行参数生成拍摄点位。
2. 对每个点位：
   - 向 Redis `ptz:command_queue` 写入 `move_absolute`。
   - 轮询 `ptz:current_status` 等待云台到位。
   - 到位后额外等待稳定时间，例如 2 到 4 秒。
   - 调用现有摄像头截图地址抓图。
   - 保存原图到 `raw/`。
   - 使用现有标定文件去畸变后保存到 `undistorted/`。
   - 记录目标角度和实际状态角度。

注意：

1. 采集程序不应直接 import 或调用 `ptz_control.py` 内部函数驱动硬件。
2. 采集程序只通过 Redis 与原程序通信。
3. 如果 Redis 或摄像头不可用，应失败退出并保留已有 session 文件。
4. 不要在采集失败时删除已有图片。

## 七、拼接实验路线

### 7.1 基线：当前几何投影

先复现当前正式逻辑的思想：

```text
按 pan/tilt + 相机内参，把每张去畸变图投到角度画布。
```

输出：

```text
outputs/geometry_baseline.jpg
```

用途：

1. 判断当前几何模型在这批图片上的真实表现。
2. 与后续单图所有权、接缝优化结果对比。

### 7.2 诊断：相邻图重叠区偏移

对每对相邻图片生成诊断：

1. 红/青叠加图。
2. 重叠区差异热力图。
3. 相位相关估计 `dx/dy`。
4. 判断偏移是否接近整块平移。

输出：

```text
diagnostics/red_cyan_overlay_001_002.jpg
diagnostics/overlap_shift.json
```

判断规则：

```text
如果重叠区整体偏移一致：
  优先怀疑 pan/tilt offset、FOV、到位误差。

如果重叠区局部偏移不一致：
  优先判断为视差问题，不能用一个全局平移完美修复。
```

### 7.3 主路线：single-owner mosaic

这是本计划的重点。

重叠区不要大范围融合两张图，而是为每个像素选择一个来源图：

```text
每个像素只来自一张原始图片。
```

目的：

1. 避免两个位置同时出现。
2. 尽量消灭重影。
3. 允许接缝略明显，但不要产生虚影。

输出：

```text
outputs/single_owner_mosaic.jpg
diagnostics/ownership_map.png
```

初版所有权策略可简单一些：

1. 对每张图投影后计算权重。
2. 每个像素选择权重最大的那张图。
3. 不做大范围多图融合。
4. 接缝附近只做很窄的羽化，避免重影。

### 7.4 接缝优化：避开高差异和强边缘区域

在 single-owner 基础上优化重叠区切线。

接缝成本可以包含：

```text
cost = 图像差异 + 边缘强度惩罚 + 接缝弯折惩罚
```

目标：

1. 接缝尽量走天空、墙面、地面等低纹理区域。
2. 避开柱子、窗框、设备边缘、文字等强结构。
3. 避免把同一个目标切成两半。

输出：

```text
outputs/seam_owner_mosaic.jpg
diagnostics/seam_debug.jpg
```

### 7.5 OpenCV Stitcher 只作为参考

可以输出 OpenCV Stitcher 结果，但不作为主路线。

原因：

1. OpenCV Stitcher 资源消耗大，多张 4K 图容易爆内存或 CPU。
2. 它会自由估计图像关系，可能让视觉更顺，但破坏 pan/tilt 角度可解释性。
3. 对本项目而言，坐标可信度比观赏性更重要。

输出：

```text
outputs/opencv_stitcher_reference.jpg
```

建议限制：

1. 默认使用降采样图测试。
2. 默认可关闭该输出。
3. 不把 OpenCV Stitcher 的结果直接用于坐标映射。

第一阶段先只做图像拼接实验，不接入正式坐标接口。

如果 single-owner mosaic 效果可用，后续可以考虑保存像素来源图：


1. 先查点击像素属于哪张原始图。
2. 用该原始图的拍摄 pan/tilt 和相机内参做单图球面反算。
3. 得到更贴近该局部图像来源的 pan/tilt。

这比全景大图统一线性映射更适合存在接缝和视差的场景。

但这属于第二阶段，不在本次测试程序里强制实现。

## 九、实验对比维度

每次实验至少记录以下观察：

| 维度 | 关注点 |
|---|---|
| 重影 | 是否还出现同一物体双影 |
| 接缝 | 接缝是否明显，是否切过重要目标 |
| 坐标可信度 | 图上目标是否只有一个可点击位置 |
| 资源消耗 | 是否出现内存爆、CPU 长时间占满 |
| 可回放性 | 同一批原图是否能反复生成不同拼接结果 |
| 稳定性 | 多次采集结果是否一致 |

优先级：

```text
消灭重影 > 坐标可信 > 接缝美观 > 高清输出
```

## 十二、Hugin 等距矩形坐标映射实测流程

本节用于现场验证 Hugin 等距矩形拼接结果是否能满足“点全景图像素 -> 转 PTZ 角度 -> 云台移动后目标进入画面中心附近”的闭环要求。

当前测试重点：

1. 使用 Hugin `equirectangular` 等距矩形投影。
2. 使用去畸变图 `undistorted`。
3. 第一轮不限制预览宽度，`--preview-width 0`，尽量使用 Hugin 原始输出尺寸，减少缩放带来的排查变量。
4. 测试误差目标：1° 以内可接受。
5. 不接入正式 API，不改正式全景图坐标接口；先用命令行工具验证坐标准确度。

### 12.1 生成采集 session

先在 Linux 设备上生成一次全景采集 session。示例路径统一使用：

```bash
cd /home/ultiwill/pacture
PY=/home/ultiwill/ptz_capture_system/ptz_env/bin/python
export PTZ_PROJECT_ROOT=/home/ultiwill/ptz_capture_system
```

#### 水平单行基础测试

适合先验证水平 pan 是否准确。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range 0 0 \
  --cols 6 \
  --rows 1 \
  --capture-only
```

输出会生成类似目录：

```text
/home/ultiwill/pacture/pano_exp_YYYYMMDD_HHMMSS
```

下面命令中的 `SESSION_DIR` 都替换成这个真实目录。

#### 两行垂直基础测试

用于验证 pan + tilt 的完整映射。设备端优先最多两行，减少拼接压力；建议从温和范围开始，不要一上来扫满垂直范围。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range -10 10 \
  --cols 6 \
  --rows 2 \
  --serpentine \
  --capture-only
```

说明：

1. `--cols 6` 表示每行 6 个水平拍摄点。
2. `--rows 2` 表示垂直方向 2 行，理论上已经足够拟合 Hugin yaw/pitch 到 PTZ pan/tilt。
3. `--serpentine` 表示蛇形采集，减少云台来回大幅移动时间。
4. 图片文件名会带 `panXX_tiltYY`，后续坐标拟合依赖这些真实 PTZ 角度。
5. 只有当两行测试不稳定时，再考虑临时增加到 3 行。

### 12.2 建议测试范围

当前云台范围：

```text
pan: 0 到 347
tilt: -85 到 20
```

推荐按从小到大的顺序测试：

#### A. 水平中等范围，单行

用于快速判断 Hugin 水平坐标是否可用。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range 0 0 \
  --cols 6 \
  --rows 1 \
  --capture-only
```

#### B. 水平中等范围，两行

用于第一轮 pan + tilt 验证，推荐优先做。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range -10 10 \
  --cols 6 \
  --rows 2 \
  --serpentine \
  --capture-only
```

#### C. 水平较宽范围，两行

用于测试左右边缘是否仍然准确。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 40 320 \
  --tilt-range -10 10 \
  --cols 8 \
  --rows 2 \
  --serpentine \
  --capture-only
```

#### D. 垂直较大范围，两行

用于测试垂直映射稳定性。注意避开极限边界，先不要直接扫到 `-85` 或 `20`。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range -30 10 \
  --cols 6 \
  --rows 2 \
  --serpentine \
  --capture-only
```

#### E. 接近完整水平范围，两行

用于压力测试。该范围更容易暴露边缘、视差、接缝问题。

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 10 337 \
  --tilt-range -10 10 \
  --cols 10 \
  --rows 2 \
  --serpentine \
  --capture-only
```

#### F. 更大垂直范围，不作为默认测试

镜头参数约为 `H:85° / V:48°`。两行采集时，`tilt_range` 的跨度就是两行中心点间距；为了让两行之间仍有可拼接重叠，垂直跨度建议控制在 35° 到 40° 左右，最多不要超过约 45°。

因此不建议用两行直接测试接近完整垂直范围，例如 `-70..15` 的跨度是 85°，两行之间没有有效重叠，理论上就很难拼接可靠。

如确实要扩大垂直范围，优先只把跨度略微扩大到约 45°：

```bash
$PY panorama_mosaic_experiment.py \
  --output-root /home/ultiwill/pacture \
  --pan-range 90 270 \
  --tilt-range -35 10 \
  --cols 6 \
  --rows 2 \
  --serpentine \
  --capture-only
```

如确实需要覆盖 `-70..15` 这类大垂直范围，理论上需要 3 行以上，但这对设备拼接压力较大，不作为默认现场流程。

### 12.3 Hugin 等距矩形拼接

用刚生成的 session 跑 Hugin 等距矩形拼接。第一轮建议 `--preview-width 0`：

```bash
SESSION_DIR=/home/ultiwill/pacture/pano_exp_YYYYMMDD_HHMMSS

$PY hugin_panorama_experiment.py \
  --session "$SESSION_DIR" \
  --image-set undistorted \
  --projection equirectangular \
  --preview-width 0
```

参数说明：

1. `--session`：输入采集 session 目录。
2. `--image-set undistorted`：使用去畸变图拼接。
3. `--projection equirectangular`：只生成等距矩形投影。
4. `--preview-width 0`：不强制缩放到 3000 宽，尽量保留 Hugin 原始输出尺寸。

拼接完成后重点看两个文件：

```text
$SESSION_DIR/hugin_outputs/最终图片/等距矩形投影_去畸变图_预览宽0.jpg
$SESSION_DIR/hugin_outputs/最终图片/等距矩形投影_去畸变图_预览宽0_coordinate_meta.json
```

如果现场发现 `预览宽0` 的文件名因为脚本版本或历史输出不同而不一致，以终端输出的 `final_image` 和 `coordinate_metadata` 为准。

如果后续需要生成方便传输/查看的 3000 宽图片，可以改用：

```bash
$PY hugin_panorama_experiment.py \
  --session "$SESSION_DIR" \
  --image-set undistorted \
  --projection equirectangular \
  --preview-width 3000
```

但坐标准度排查阶段优先用 `--preview-width 0`。

### 12.4 在电脑上取像素

把最终 JPG 打开，在图上选择明显目标，记录原图像素坐标：

```text
px,py
```

注意：

1. 必须记录图片自身像素坐标，不要记录屏幕显示坐标。
2. 如果看图软件缩放显示，仍要确认读取的是原图坐标。
3. 第一轮尽量选清晰、非重影、非接缝、非边缘目标。
4. 建议至少选 5 到 10 个点，覆盖左、中、右、上、中、下。

### 12.5 像素转 PTZ 角度

优先使用 `coordinate_map.npz` 走源图像素映射。该路线会让 Hugin/nona 把源图坐标编码图重投影到全景画布，然后点击像素时反查 `source_px/source_py`，再复用原来的单张图球面反算逻辑。

先生成 map：

```bash
MAP="${IMG%.jpg}_coordinate_map.npz"

$PY hugin_coordinate_mapper.py build-map \
  --meta "$META" \
  --session "$SESSION_DIR" \
  --out "$MAP"
```

然后转换像素：

```bash
$PY hugin_coordinate_mapper.py convert-map \
  --map "$MAP" \
  --point "1500,257"
```

一次转换多个点：

```bash
$PY hugin_coordinate_mapper.py convert-map \
  --map "$MAP" \
  --point "100,250 1500,257 2900,260"
```

输出中重点看：

```json
{
  "method": "coordinate_map_source_pixel",
  "pan": 176.59,
  "tilt": -2.1,
  "source": {
    "shot_file": "...shot_003_pan162.00_tilt0.00.jpg",
    "source_px": 1040.2,
    "source_py": 612.8
  }
}
```

旧的 Hugin yaw/pitch 拟合转换只作为调试参考：

```bash
META="$SESSION_DIR/hugin_outputs/最终图片/等距矩形投影_去畸变图_预览宽0_coordinate_meta.json"

$PY hugin_coordinate_mapper.py convert \
  --meta "$META" \
  --point "1500,257"
```

旧拟合路线一次转换多个点：

```bash
$PY hugin_coordinate_mapper.py convert \
  --meta "$META" \
  --point "100,250 1500,257 2900,260"
```

输出中重点看：

```json
{
  "pan": 176.59,
  "tilt": 0.0,
  "tilt_mapping_reliable": true
}
```

判断规则：

1. `pan` / `tilt` 是要移动云台的目标角度。
2. `in_frame=false` 表示输入像素超出最终图范围。
3. `tilt_mapping_reliable=false` 表示拍摄数据没有足够垂直行，不能把 tilt 当作正式验证结果。
4. 多行 tilt 测试时，期望 `tilt_mapping_reliable=true`。

### 12.6 移动云台验证

如果后端服务运行在本机 5000 端口，可以用：

```bash
curl -s -X POST http://127.0.0.1:5000/api/v1/ptz/move \
  -H "Content-Type: application/json" \
  -d '{"pan":176.59,"tilt":0.0,"pan_step_size":20.0,"tilt_step_size":10.0}'
```

查看当前云台状态：

```bash
curl -s http://127.0.0.1:5000/api/v1/ptz/status
```

如果服务端口不是 5000，替换命令中的端口。

移动后观察摄像头画面：

1. 目标是否进入当前画面。
2. 目标是否接近画面中心。
3. 偏差方向：偏左、偏右、偏上、偏下。
4. 偏差大小：约多少度，或约占画面宽/高的比例。

### 12.8 结果判断

#### 可接受

```text
大多数点误差 <= 1°
非接缝、非重影区域稳定
左、中、右没有明显系统性发散
```

可以继续推进 Hugin 等距矩形方案接入正式流程。

#### 只存在整体固定偏移

表现：

```text
所有点都偏左或都偏右
所有点都偏上或都偏下
偏差大小接近
```

处理方向：

```text
增加 pan_offset / tilt_offset 修正
```

#### 中间准，边缘偏

表现：

```text
中间点误差小
越靠左/右/上/下误差越大
```

处理方向：

```text
检查 Hugin crop/FOV 解释
检查 yaw/pitch -> PTZ 拟合模型
增加拍摄点数量或缩小单次拼接范围
```

#### 局部点偏，其他点准

表现：

```text
只有接缝附近、重影附近、近距离视差区域偏差大
```

处理方向：

```text
该区域视为坐标低可信区域
优化拼接或避开这些点
必要时记录像素可信 mask
```

#### 垂直不准

表现：

```text
pan 基本准
tilt 明显偏
metadata 中 tilt_mapping_reliable=false
```

处理方向：

```text
必须使用多行 tilt 重新采集
建议先用 tilt -10 到 10、rows 2
通过后再扩大到 -30 到 10、rows 2
```

### 12.9 现场测试优先级

建议按以下顺序执行：

1. `90..270 / tilt=0 / 6x1`：先确认水平 pan。
2. `90..270 / tilt=-10..10 / 6x2`：确认基础 pan + tilt。
3. `40..320 / tilt=-10..10 / 8x2`：确认宽水平范围。
4. `90..270 / tilt=-30..10 / 6x2`：确认较大垂直范围。
5. `10..337 / tilt=-10..10 / 10x2`：接近完整水平范围压力测试。
6. `90..270 / tilt=-35..10 / 6x2`：两行条件下的较大垂直范围压力测试。

只要前两项大多数点能控制在 1° 以内，就说明 Hugin 等距矩形坐标映射路线值得继续推进。
