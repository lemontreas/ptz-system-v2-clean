# 全面扫描数据选择优化方案

> **状态**：方案已确认，待交付给 AI 执行  
> **影响文件**：仅 `capture_worker.py`  
> **相关架构文档**：`idea/ARCHITECTURE.md` § 全面扫描代码地图  

---

## 一、背景与问题

全面扫描每个点位执行**跳频扫描**：逐个切换所有 `(channel, bandwidth)` 组合并抓包，最终把各信道收到的同一 MAC 数据汇聚成一条结果。

### 当前两个问题

**问题 A：AP 信道污染**

AP 只在自己工作信道（由 Beacon 的 `DS Parameter Set` IE 宣告，`Dot11Elt ID=3`）发送 Beacon。当天线跳到其他信道时，仍可能侦听到该 AP 的客户端数据帧或广播管理帧，从而积累不属于 AP 工作信道的 RSSI 样本，污染了真实工作信道下的定位结果。

**问题 B：所有设备 RSSI 被均值稀释**

当前对每个信道的多帧取均值，把突发的强信号包稀释掉，丢失了"该点位能接到设备的最强证据"。对 Client 尤其明显：Probe Request 等广播管理帧在多个信道均可抓到，均值比较会选错信道。

---

## 二、优化策略（已确认）

| 设备类型 | 当前行为 | 优化后行为 |
|---------|---------|-----------| 
| **AP（有 Beacon，有宣告信道）** | **先过滤**：只保留 Beacon 宣告信道号相同的包；**再取 max**：`rssi_avg` 存该信道内单包最大 RSSI；`channel` / `bandwidth` 来自 Beacon |
| **AP（有 Beacon，无信道信息）** | 同上 | 退化为 Client 策略 |
| **Client / 未识别** | 跨信道保留**单包最大 RSSI** 最高的那个信道，`rssi_avg` 存 `max(rssi)`，`channel` / `bandwidth` 存该信道扫描参数 |

> **核心统一原则**：所有设备在每个点位只保留最强的单包 RSSI（`max`），不再对多帧求均值。AP 额外的区别仅在于：先用 Beacon 信道过滤，再取信道内 max；`channel` / `bandwidth` 来自 Beacon 而非扫描参数。

### 关键设计决定

1. **AP 和 Client 都取 max**：每个点位的最强接收信号才是定位的有效依据，均值会稀释突发强信号，不再使用。

2. **AP 信道只比较信道号**：AP 宣告信道是主信道（Primary Channel），信道号相同即可，带宽从 Beacon 独立提取。

3. **AP 带宽从 Beacon 提取**：解析 HT Operation IE（`ID=61`）和 VHT Operation IE（`ID=192`），覆盖 HT20 / HT40+ / HT40- / VHT80 四种常见情况。解析失败时退化为扫描时的带宽。

4. **不新增字段**：`rssi_avg` 字段语义统一为"该信道单包最大值"（AP 和 Client 一致）。对前端、白名单筛选、`rebuild_full_scan_whitelist.py` 均无影响，字段语义在此文档说明即可。

5. **白名单筛选不变**：`_build_full_scan_whitelist_payload()` 继续读 `rssi_avg` 字段，不需要修改。用 max 代替 mean 只会让强信号设备更容易命中，不会漏掉。

## 三、代码改动（仅 `capture_worker.py`）

_global_ap_bandwidths = {}

---

### 改动 ②：`_extract_beacon_info()` 增加带宽解析（约 L58-86）

在现有遍历 `Dot11Elt` 的循环里，新增对 HT Operation IE（ID=61）和 VHT Operation IE（ID=192）的解析。**在一次遍历中同时处理 ID=0 / 3 / 61 / 192，不增加额外循环。**

**当前代码（循环内，简化）：**
```python
for elt in pkt[Dot11Elt:]:
    if elt.ID == 0 and ssid == '':
        ssid = _decode_ssid(elt.info)
    elif elt.ID == 3 and len(elt.info) >= 1:
        ap_ch = str(elt.info[0])
```

**替换为（在同一循环内追加 ID=61 / 192）：**
```python
ap_bw = None   # 新增：工作带宽，None 表示未解析到
for elt in pkt[Dot11Elt:]:
    if not hasattr(elt, 'ID'):
        break
    if elt.ID == 0 and ssid == '':
        ssid = _decode_ssid(elt.info)
    elif elt.ID == 3 and len(elt.info) >= 1:
        ap_ch = str(elt.info[0])
    elif elt.ID == 61 and len(elt.info) >= 2:
        # HT Operation IE：byte[1] bit0=STA_CH_WIDTH, bits1-2=secondary channel offset
        sta_ch_width = elt.info[1] & 0x01
        sec_ch_offset = (elt.info[1] >> 1) & 0x03
        if sta_ch_width == 0:
            ap_bw = 'HT20'
        elif sec_ch_offset == 1:
            ap_bw = 'HT40+'
        elif sec_ch_offset == 3:
            ap_bw = 'HT40-'
        else:
            ap_bw = 'HT20'   # 宣告支持 40MHz 但无有效 secondary offset，保守当 HT20
    elif elt.ID == 192 and len(elt.info) >= 1:
        # VHT Operation IE：byte[0]=channel_width（0=20/40 由 HT 决定，1+=80MHz及以上）
        if elt.info[0] >= 1:
            ap_bw = 'VHT80'  # 覆盖 HT Operation 结果，VHT 优先级更高
```

同时更新 `return` 语句，加入 `bandwidth` 字段：
```python
return {'bssid': bssid, 'ssid': ssid, 'channel': ap_ch, 'bandwidth': ap_bw}
```

### 改动 ⑤：聚合逻辑替换（约 L2587-2631）

将当前的 `rssi_avg` 比较逻辑替换为以下新逻辑：

**当前代码（L2588 附近，简化）：**
```python
rssi_avg = round(sum(rssi_list) / len(rssi_list), 2)
mac_entry = {
    'channel': channel, 'bandwidth': bandwidth,
    'rssi_avg': rssi_avg, 'rssi_samples': len(rssi_list),
    ...
}
if mac in discovered_macs:
    if rssi_avg > discovered_macs[mac]['rssi_avg']:
        discovered_macs[mac] = mac_entry
else:
    discovered_macs[mac] = mac_entry
```

**替换为：**
```python
bssid_lower = mac.lower()

# ── 判断是否为有宣告信道的 AP ──────────────────────────────
is_ap_with_declared_ch = (
    bssid_lower in _global_ap_macs and
    _global_ap_channels.get(bssid_lower) is not None
)

# ── AP 信道过滤（只处理有宣告信道的 AP）─────────────────────
if is_ap_with_declared_ch:
    declared_ch = _global_ap_channels[bssid_lower]
    if int(channel) != declared_ch:
        # 当前跳频信道号不等于 Beacon 宣告信道号 → 丢弃
        logger.debug(
            f"[{point_id}] AP {mac} 跳过非工作信道 {channel}"
            f"（Beacon 宣告信道={declared_ch}）"
        )
        continue

# ── 确定最终 channel / bandwidth ─────────────────────────
if is_ap_with_declared_ch:
    # AP：信道来自 Beacon DS Parameter Set，带宽来自 Beacon HT/VHT Operation IE
    # 若带宽未解析到，退化为本次跳频带宽
    final_channel = _global_ap_channels[bssid_lower]
    final_bandwidth = _global_ap_bandwidths.get(bssid_lower) or bandwidth
else:
    # Client / 未识别：使用本次跳频的 channel / bandwidth
    final_channel = channel
    final_bandwidth = bandwidth

# ── 计算本信道的代表 RSSI（AP 和 Client 统一取 max）─────────────────
# AP：只有通过信道过滤的包才到达这里，取该宣告信道内单包最大值
# Client / 未识别：取该跳频信道内单包最大值
# 兜底：rssi_max_by_mac 为空时退化为均值（理论上不会，但防御性处理）
representative_rssi = rssi_max_by_mac.get(mac) or round(sum(rssi_list) / len(rssi_list), 2)

# ── 构建 mac_entry ─────────────────────────────────────────
# rssi_avg 字段语义：统一存该信道单包最大值，AP 和 Client 一致（见文档说明）
mac_entry = {
    'channel': final_channel,
    'bandwidth': final_bandwidth,
    'rssi_avg': representative_rssi,
    'rssi_samples': len(rssi_list),
    'first_seen_at': first_seen_by_mac.get(mac),
    'last_seen_at': last_seen_by_mac.get(mac),
    # 全向字段由后续代码填充，此处不变
}

# ── 与已有记录合并 ────────────────────────────────────────
if mac in discovered_macs:
    existing = discovered_macs[mac]
    # 保留最早 first_seen / 最晚 last_seen
    existing['first_seen_at'] = (
        existing.get('first_seen_at') or first_seen_by_mac.get(mac)
    )
    existing['last_seen_at'] = (
        last_seen_by_mac.get(mac) or existing.get('last_seen_at')
    )
    # 统一用 rssi_avg 比较（AP 和 Client 都是 max 比 max，逻辑一致）
    if representative_rssi > existing['rssi_avg']:
        mac_entry['first_seen_at'] = (
            existing.get('first_seen_at') or mac_entry.get('first_seen_at')
        )
        discovered_macs[mac] = mac_entry
        logger.info(
            f"[{point_id}] 🔄 更新 {mac} -> ch{final_channel} {final_bandwidth},"
            f" rssi_avg={representative_rssi}"
        )
else:
    discovered_macs[mac] = mac_entry
    logger.info(
        f"[{point_id}] 🆕 新 MAC {mac} -> ch{final_channel} {final_bandwidth},"
        f" rssi_avg={representative_rssi}"
    )
```

## 四、字段语义变更说明

| 字段 | AP（有宣告信道） | Client / 未识别 / AP（无 Beacon） |
|------|----------------|----------------------------------|
| `channel` | **Beacon DS Parameter Set IE 宣告的主信道号** | max RSSI 最高的跳频信道号 |
| `bandwidth` | **Beacon HT/VHT Operation IE 解析的真实工作带宽**；解析失败时退化为扫描带宽 | 同上，对应跳频带宽 |
| `rssi_avg` | 该宣告信道内所有包的**单包最大值** | 该信道内所有包的**单包最大值** |
| `rssi_samples` | 不变（该信道抓到的包数） | 不变 |
| `omni_rssi_avg` | 不变 | 不变 |

> **注意**：`rssi_avg` 字段名保留不变，但语义统一为"该信道单包最大 RSSI"，AP 和 Client 一致。对前端和白名单筛选无影响，用 max 代替 mean 只会让强信号设备的 RSSI 表征更准确、更不易被噪声包拉低。

### Beacon 带宽解析覆盖范围

| Beacon IE | 解析结果 | 典型场景 |
|-----------|---------|---------|
| 无 HT Operation IE（ID=61） | `None`，退化为扫描带宽 | 旧式 802.11b/g 设备 |
| HT Operation, STA_CH_WIDTH=0 | `HT20` | 2.4GHz 只用 20MHz 的 AP |
| HT Operation, STA_CH_WIDTH=1, offset=above | `HT40+` | 2.4GHz / 5GHz HT40 AP（副信道在上） |
| HT Operation, STA_CH_WIDTH=1, offset=below | `HT40-` | 同上（副信道在下） |
| VHT Operation（ID=192）, channel_width≥1 | `VHT80` | 5GHz 802.11ac AP（覆盖 80MHz 及以上） |

## 六、验证要点

1. **AP channel 验证**：已知 AP 的 `channel` 字段应与其 Beacon 宣告信道一致，而非扫描时的跳频信道。
2. **AP bandwidth 验证**：已知 HT40 AP 的 `bandwidth` 字段应为 `HT40+` 或 `HT40-`，而非 `HT20`；已知 HT20 AP 应为 `HT20`；802.11ac 5GHz AP 应为 `VHT80`。
3. **AP 无 Beacon 退化**：`channel` 未解析到时，应退化为 Client 策略（跨信道 max）。
4. **AP bandwidth 解析失败退化**：`bandwidth` 未解析到时，应退化为扫描时的带宽，不应出现 `None`。
5. **Client 验证**：同一 Client 在结果中只出现一条，`rssi_avg` 等于该信道内实际收到的最强单包值。
6. **无越界**：结果中不应出现 AP 被分配到非工作信道的情况。
7. **白名单不退化**：优化后已知强信号 AP 的命中率不应下降。
