# 天线偏差优化方案

> **状态**：讨论确认，待实现  
> **目标**：修正定向天线 RF 最强响应方向与前端视觉真实方向之间的稳定偏差  
> **影响范围**：全面扫描、定位扫描、结果位置字段、白名单/目标区判断、Redis/请求体距离字段  
> **相关规则**：涉及扫描结果和状态时，仍必须遵守 `rule/状态管理规则.md`

---

## 一、背景与问题

当前定位扫描中发现：使用定向天线作为信源时，后端定位出的 RF 最强点与前端全景图上的实际目标位置存在偏差。

已测样本：

| 项 | 值 |
|---|---|
| 信源距离 | 220m |
| RF 定位最强点像素 | `px=4687, py=635` |
| 实际目标像素 | `px=4710, py=584` |
| 像素偏差 | `dx=+23, dy=-51` |
| 当前全景像素密度 | `36 px/deg` |

含义：

```text
实际目标在 RF 最强点的右上方。
```

如果不修正该偏差，后端会把 RF 扫描点直接当作视觉真实点使用，导致：

1. 定位结果点位在前端图上偏离真实目标。
2. 目标位于工作区/目标区边缘时，可能被错误判断为区内或区外。
3. 全面扫描白名单筛选、定位扫描目标区判断可能受到影响。

---

## 二、核心建模

本方案把问题建模为：

```text
RF 扫描坐标  <->  前端视觉坐标
```

其中：

```text
visual_position = rf_position + antenna_bias(target_distance)
rf_position     = visual_position - antenna_bias(target_distance)
```

前端传入的工作区/目标区属于 **visual 坐标系**。  
云台实际移动和抓包使用的是 **RF/PTZ 坐标系**。

### 2.1 第一版改为 distance_scaled_pixel 模型

当前实现曾采用 `lateral_offset` 模型：用 220m 标定样本反推等效物理横向偏移，再根据运行距离换算角度。

实测 5m 场景后确认该模型不适合作为第一版默认模型：如果把 220m 下的像素偏差反推成米级横向偏移，距离越近角度偏差会越大，5m 时会被 `最大补偿角度` 直接 clamp 到 5°，与当前业务直觉不符。

第一版应改为 `distance_scaled_pixel` 模型：把 220m 标定样本中的像素偏差按距离比例缩放，再换算成角度偏差。

核心假设：

```text
同一套前端全景/视觉坐标下，偏差像素随目标距离按比例缩放。
距离越近，偏差像素越小；距离越远，偏差像素越大。
```

计算流程：

```text
1. 标定输入使用像素点：
   rf_pixel = [4687, 635]
   actual_pixel = [4710, 584]
   calibration_distance_m = 220

2. 像素差换算为标定距离下的角度差：
   pan_bias_deg_220  = dx / px_per_deg
   tilt_bias_deg_220 = -dy / px_per_deg

3. 按运行距离 D 缩放像素偏差：
   scale = D / calibration_distance_m
   dx_D = dx * scale
   dy_D = dy * scale

4. 缩放后的像素偏差换算为角度：
   pan_bias_deg(D)  = dx_D / px_per_deg
   tilt_bias_deg(D) = -dy_D / px_per_deg

等价写法：
   pan_bias_deg(D)  = pan_bias_deg_220  * D / calibration_distance_m
   tilt_bias_deg(D) = tilt_bias_deg_220 * D / calibration_distance_m
```

按当前样本和 `36 px/deg` 估算：

```text
pan_bias_deg_220  ~= +0.64°
tilt_bias_deg_220 ~= +1.42°

D = 220m -> pan ~= +0.6°, tilt ~= +1.4°
D = 100m -> pan ~= +0.3°, tilt ~= +0.6°
D = 5m   -> pan ~= +0.0°, tilt ~= +0.0°（按 0.1° 精度处理后）
```

> 说明：`py` 越小表示全景图上越靠上，项目全景转换中 tilt 随 `py` 减小而增大，所以 `dy=-51` 对应 `tilt_bias_deg=+51/36`。

### 2.2 后续模型升级预留

第一版当前应实现 `distance_scaled_pixel`，并保留 `mode` 分发。后续拿到其他距离的偏差样本后，可以升级模型而不改扫描主流程。

后续新增样本格式：

```json
{
  "距离米": 500,
  "RF最强点像素": [0, 0],
  "实际目标像素": [0, 0]
}
```

推荐升级路线：

1. 继续用第一版 `distance_scaled_pixel` 预测新距离偏差。
2. 将预测偏差和新样本真实偏差比较。
3. 如果误差很小，继续使用 `distance_scaled_pixel`。
4. 如果不同距离明显对不上，升级为 `distance_lut`。
5. 如果既有固定角度偏差，又有物理横向偏移特征，再考虑 `hybrid`。

预留模式：

| mode | 第一版是否实现 | 说明 |
|---|---|---|
| `distance_scaled_pixel` | 是 | 用标定像素偏差按 `D / 标定距离` 缩放，再换算角度 |
| `lateral_offset` | 暂不作为默认；可作为旧配置兼容别名 | 旧模型：反推等效米级偏移，近距离会放大角度，当前不适合作为默认 |
| `distance_lut` | 否，预留 | 多距离标定表；命中标定点直接使用，中间距离插值，超出范围取最近点或有限外推 |
| `hybrid` | 否，预留 | 混合固定角度、比例缩放或物理偏移项，至少需要多个距离样本拟合 |

实现要求：

```text
build_antenna_bias(target_distance)
  -> 根据 config["天线偏差补偿"]["模式"] 分发
  -> 第一版支持 distance_scaled_pixel
  -> 为兼容当前已提交配置，可临时把 lateral_offset 当作 distance_scaled_pixel 处理，并在返回中标明 configured_mode
  -> 遇到未实现 mode 时禁用补偿并返回 reason，不要静默套错公式
```

## 三、数据约定

### 3.1 扫描类任务的 Redis 距离字段

全面扫描和定位扫描启动前，上层业务链路需要确保目标/信源距离已写入 Redis `gimbal:default_config`。可以由前端传给业务后端，再由业务后端写入 Redis；云台程序只关心运行时能否从 Redis 读到该字段。

新增字段：

```json
{
  "target_distance": 220
}
```

字段语义：

| 字段 | 类型 | 说明 |
|---|---|---|
| `target_distance` | number | 目标/信源与设备的直线距离，单位米，存放在 Redis `gimbal:default_config` 中 |

距离可以由前端按实际 UI 做档位输入，但写入 Redis 时建议仍为米数，方便云台程序统一计算。

如果 Redis `gimbal:default_config.target_distance` 缺失，则本次任务不启用天线偏差补偿。不要把配置里的 `标定距离米` 当作默认运行距离使用。

注意：本文中“全面扫描 / 定位扫描启动接口不新增 `target_distance` 参数”指的是云台程序接口。上层业务后端可以在自己的扫描启动接口中接收 `target_distance`，并在启动云台扫描前写入 Redis。

### 3.2 配置新增天线偏差标定

建议在 `config.json` 增加：

```json
{
  "天线偏差补偿": {
    "启用": true,
    "模式": "distance_scaled_pixel",
    "标定距离米": 220,
    "RF最强点像素": [4687, 635],
    "实际目标像素": [4710, 584],
    "最小距离米": 1,
    "最大补偿角度": 5.0
  }
}
```

说明：

| 字段 | 说明 |
|---|---|
| `启用` | 是否启用偏差补偿 |
| `模式` | 第一版使用 `distance_scaled_pixel`；后续预留 `distance_lut` / `hybrid`；已提交旧配置里的 `lateral_offset` 只能作为兼容别名处理 |
| `标定距离米` | 标定样本对应距离，只用于按距离比例缩放像素偏差，不是前端未传距离时的默认运行距离 |
| `RF最强点像素` | 未修正前定位出的 RF 最强点 |
| `实际目标像素` | 前端全景图上真实目标点 |
| `最小距离米` | 防止误填 0、负数或无意义的极小距离 |
| `最大补偿角度` | 防御性上限，避免错误配置导致大幅偏移 |

### 3.3 返回/Redis 字段约定

为了减少前端改动，原有 `position` 字段保持给前端使用，但内容改为 **偏差修正后的视觉坐标**。

新增调试字段保存转换前坐标：

```json
{
  "position": {
    "pan": 43.84,
    "tilt": 7.42
  },
  "rf_position": {
    "pan": 43.2,
    "tilt": 6.0
  },
  "antenna_bias": {
    "enabled": true,
    "mode": "distance_scaled_pixel",
    "target_distance": 220,
    "calibration_distance_m": 220,
    "pan_bias_deg": 0.64,
    "tilt_bias_deg": 1.42,
    "scale": 1.0,
    "dx_px_scaled": 23.0,
    "dy_px_scaled": -51.0,
    "direction": "rf_to_visual"
  }
}
```

字段语义：

| 字段 | 说明 |
|---|---|
| `position` | 修正后的视觉坐标，前端继续读这个字段 |
| `rf_position` | 原始 RF/PTZ 扫描坐标，仅用于调试 |
| `antenna_bias` | 本次任务使用的偏差参数，方便排查转换是否正确 |

## 四、修正规则

### 4.1 前端视觉点位转 RF 扫描点位

前端传入的工作区/目标区都是视觉坐标：

```text
visual_pan_range
visual_tilt_range
```

后端根据 Redis `gimbal:default_config.target_distance` 算出本次偏差：

```text
pan_bias_deg
tilt_bias_deg
```

实现时优先保留 **visual 点位表**，再为每个 visual 点位生成对应的 RF 移动点：

```text
rf_pan  = visual_pan  - pan_bias_deg
rf_tilt = visual_tilt - tilt_bias_deg
```

等价理解为：实际云台扫描范围相对前端目标范围整体反向偏移了一段偏差。但结果归属仍然使用原 visual 点位。

已有的扩边/guard 逻辑继续复用，不新增一套 guard。扩边、guard、点位去重、阶段划分应优先在 visual 坐标中完成，然后再映射出 RF 移动点。

### 4.2 RF 扫描结果转视觉结果

扫描中云台真实到达的点为：

```text
rf_position
```

写结果和返回前端时，主字段使用对应的 visual 点位：

```text
position.pan  = visual_position.pan
position.tilt = visual_position.tilt
```

同时保留：

```text
rf_position = 实际移动/抓包坐标
```

RSSI 是在 `rf_position` 上采到的，但归属到对应的 `position` 视觉点位上。这样前端看到的点位仍然落在它定义的目标范围/工作区坐标体系中。

实现注意：

```text
capture_worker 收到的 pan/tilt 应是 RF 实际移动点。
capture_worker notify/result 中如果带回 pan/tilt，也应视为 RF 坐标。
ptz_control.py 写 full_scan:results / location_scan:result 时，必须把最终业务字段归属回 visual 点位：
  position / best_position = visual 点位
  rf_position / rf_best_position = capture_worker 或本地记录的 RF 点位
不能把 capture_worker 回传的 RF pan/tilt 直接写成最终 position。
```

### 4.3 区域判断

是否在工作区/目标区内，统一用结果主字段 `position` / `best_position` 这种 visual 坐标判断。

如果某段旧逻辑手里只有 `rf_position`，必须先转回 visual 坐标再判断：

```text
rf_position
  -> 加 antenna_bias 得到 position
  -> 用 position 判断是否落在前端 visual 工作区/目标区内
```

这样可以修正边缘误判。例如：

```text
RF 点看起来在区内
但加偏差后的视觉点在区外
=> 应按区外处理，可被白名单或定位结果筛掉
```

## 五、全面扫描接入点

全面扫描需要接入两处：

### 5.1 启动参数与距离透传

API 层接收前端 `work_ranges` / `pan_range` / `tilt_range` 后，只负责准备任务参数，不在 API 层生成 RF 扫描点，也不改变原始范围字段的语义：

1. 保留前端传入的 visual 范围，原样放入扫描命令，供 `ptz_control.py` 生成 visual 点位。
2. 从 Redis `gimbal:default_config.target_distance` 读取距离。
3. 如果距离有效，计算 `antenna_bias` 并放入扫描命令 / 启动响应；如果距离缺失或无效，则标记补偿未启用。
4. scan plan / result 中可附带 `antenna_bias` 和原始 visual ranges，用于前端调试和预估。
5. API 层不要把 `work_ranges` / `pan_range` / `tilt_range` 预先改成 RF 范围。

真实执行阶段必须放在 `ptz_control.py`：

1. 按原 visual 范围生成 visual 点位。
2. 每个 visual 点位执行前映射为 RF 点位。
3. 云台移动 / 抓包使用 RF 点位。
4. 写结果时，RSSI 归属回对应的 visual `position`。

### 5.2 白名单筛选

全面扫描点位结果里：

```text
position    = 前端 visual 点位
rf_position = 实际移动/抓包点位 = position - antenna_bias
rssi        = 在 rf_position 上采集，但归属到 position
```

白名单判断工作区内外时，应使用修正后的 `position`。

需要重点检查：

| 函数/逻辑 | 要求 |
|---|---|
| `_build_full_scan_whitelist_payload()` | 工作区内外判断使用修正后的视觉坐标 |
| 结果点位写入 | 保留 `position` 和 `rf_position` |
| 失败/淘汰原因 | `metrics` 中尽量保留偏差参数，方便解释边界排除 |

## 六、定位扫描接入点

定位扫描需要接入三处：

### 6.1 启动目标区转换

前端传入目标区范围：

```text
pan_ranges / tilt_ranges
```

这些范围是 visual 坐标。定位扫描不从请求体读取 `target_distance`；后端需读取 Redis `gimbal:default_config.target_distance`，计算偏差，并把每个 visual 扫描点映射为 RF 移动点：

```text
rf_pan  = visual_pan  - pan_bias_deg
rf_tilt = visual_tilt - tilt_bias_deg
```

定位扫描已有的扩圈、候选框收缩、目标区保底点逻辑继续复用。

### 6.2 最强点和最终定位结果

每个 MAC 的原始最强点：

```text
rf_position
```

返回和写 Redis 的主字段：

```text
best_position = 对应的 visual 最强点
```

同时新增：

```text
rf_best_position
```

建议结果结构：

```json
{
  "best_position": {"pan": 43.84, "tilt": 7.42},
  "rf_best_position": {"pan": 43.2, "tilt": 6.0},
  "antenna_bias": {
    "enabled": true,
    "target_distance": 220,
    "pan_bias_deg": 0.64,
    "tilt_bias_deg": 1.42
  }
}
```

### 6.3 结果是否在目标区内

定位扫描中如果需要判断最强点是否落在前端目标区，应统一使用修正后的 `best_position`。

## 七、用户指定点位类接口

用户指定点位类接口包括：

| 接口/功能 | 说明 |
|---|---|
| `capture_at_best` | 前端显式传入 `pan/tilt/mac/channel/bandwidth`，要求云台到指定点位抓包 |
| WiFi 连接尝试 | 前端显式传入目标连接信息，可附带 `best_position` 接收角度，要求连接前先让天线朝向指定点位 |

这类接口不使用 Redis `gimbal:default_config.target_distance`。Redis 的 `target_distance` 只用于全面扫描和定位扫描。

### 7.1 距离来源

用户指定点位类接口的距离来源是请求体。抓包接口直接使用顶层 `pan/tilt`，WiFi 连接尝试优先使用 `best_position.pan/best_position.tilt`：

```json
{
  "pan": 43.84,
  "tilt": 7.42,
  "target_distance": 220
}
```

```json
{
  "best_position": {
    "pan": 43.84,
    "tilt": 7.42
  },
  "target_distance": 220
}
```

规则：

```text
请求体有 target_distance：
  按该距离计算 antenna_bias
  将前端传入的 visual 点位反向纠偏为 RF pan/tilt
  云台移动到 RF pan/tilt 后执行抓包/连接

请求体没有 target_distance：
  不启用偏差补偿
  后端按前端传入点位直接移动/执行
```

### 7.2 capture_at_best

实现要求：

```text
前端传 pan/tilt
如果请求体带 target_distance：
  rf_pan  = pan  - pan_bias_deg
  rf_tilt = tilt - tilt_bias_deg
  后端移动到 rf_pan/rf_tilt 抓包
  返回 target_position = 前端传入 pan/tilt
  返回 rf_target_position = 实际移动 pan/tilt
  返回 antenna_bias
否则：
  后端按现有逻辑直接移动到 pan/tilt
  不计算 antenna_bias
  不新增 rf_target_position
```

注意：如果调用方传入的是定位扫描结果中的 `best_position`，它是 visual 坐标；当请求体同时传入该目标的 `target_distance` 时，后端需要反向纠偏到 RF 接收角度后抓包。这不是二次偏移，而是“visual 指定点 -> RF 实际接收点”的一次转换。

### 7.3 WiFi 连接尝试

WiFi 连接尝试如需要让天线先朝向目标，应在请求体中新增可选字段：

```json
{
  "ssid": "TargetNetwork",
  "bssid": "aa:bb:cc:dd:ee:ff",
  "password": "password",
  "best_position": {
    "pan": 43.84,
    "tilt": 7.42
  },
  "target_distance": 220
}
```

`best_position` 是扫描结果里的最佳信号位置，属于 visual 角度坐标；它已经是云台程序使用的角度单位，不是像素，但还不是反向纠偏后的 RF 接收角度。

规则：

```text
请求体有 best_position.pan/best_position.tilt/target_distance：
  先按 target_distance 计算 antenna_bias
  反向纠偏得到 rf_pan/rf_tilt
  连接尝试前先移动云台到 rf_pan/rf_tilt
  再进入原有 WiFi 连接流程

请求体没有 best_position：
  不移动云台，按原有 WiFi 连接流程执行

请求体有 best_position 但没有 target_distance：
  不启用偏差补偿
  如实现支持连接前移动，则直接移动到 best_position.pan/best_position.tilt
```

兼容建议：如果现有云台程序已经支持顶层 `pan/tilt`，可以继续兼容；但本次与前端对齐的主格式是 `best_position`，实现 AI 不能只支持顶层 `pan/tilt` 而忽略 `best_position`。

WiFi 连接尝试仍然必须遵守既有资源互斥和网卡模式切换规则。本方案只增加“连接前可选云台指向”和“可选反向纠偏”，不改变连接状态机、stop、monitor/managed 切换语义。

实现边界：

```text
capture_worker.py 不能直接控制云台，也不能 import ptz_control.py。
如 WiFi 连接尝试需要连接前移动云台，必须通过 Redis / ptz:command_queue 交给 ptz_control.py 执行。
web_server.py 或现有协调流程负责发起“先移动到 rf_pan/rf_tilt，再开始 WiFi 连接”的编排。
```


需要定位：


4. Redis 结果主字段 `position` 使用对应 visual 点位，RSSI 归属到这个 visual 点位。

### 10.4 第四步：接入结果字段归属

## 十一、字段不变量

这些规则是兼容性红线，不能破坏。

### 11.1 顶层 PTZ 状态不参与本次偏差转换

`ptz:current_status.position` 表示云台当前真实位置，用于运动状态展示和硬件调试。不要把顶层 `ptz:current_status.position` 改成视觉修正坐标。

本方案中的 `position` 指的是扫描结果里的点位字段，例如：

```text
full_scan results point.position
location_scan result best_position
capture_at_best target_position（请求体带 target_distance 时会同时返回 rf_target_position）
```

不是顶层 PTZ 实时状态：

```text
/api/v1/ptz/status.position
```

### 11.2 扫描结果字段语义

| 字段 | 实现后语义 | 前端是否需要改 |
|---|---|---|
| `position` | 修正后的视觉坐标 | 不需要 |
| `best_position` | 修正后的视觉最强点 | 不需要 |
| `target_position` | `capture_at_best` 前端显式传入的 visual 移动点 | 不需要 |
| `rf_position` | 原始 RF/PTZ 扫描坐标 | 仅调试 |
| `rf_best_position` | 原始 RF/PTZ 最强点 | 仅调试 |
| `rf_target_position` | 用户指定点位接口实际移动点，仅在请求体带 `target_distance` 时返回 | 仅调试 |
| `antenna_bias` | 本次补偿参数 | 调试/展示可选 |

### 11.3 方向不变量

永远记住：

```text
visual -> rf：减偏差
rf -> visual：加偏差
```

以当前样本为例：

```text
RF 最强点 px=4687,py=635
实际目标 px=4710,py=584
实际目标在右上

所以：
visual = rf + pan(+0.64), tilt(+1.42)
rf     = visual - pan(+0.64), tilt(+1.42)
```

如果实现后前端点位朝左下修正，方向就是反了。

### 11.4 未启用补偿时必须完全兼容旧逻辑

以下情况必须等价于没有本功能：

```text
天线偏差补偿.启用 = false
扫描类任务：Redis target_distance 缺失
用户指定点位类任务：请求体 target_distance 缺失
配置非法导致 bias 构建失败
```

此时：

```text
position == 原始扫描坐标
不强制新增 rf_position
不改变工作区/目标区范围
用户指定点位类接口不新增 rf_target_position
```

### 16.2 不同距离验证

```text
距离 target_distance

```

用现有 `distance_scaled_pixel` 模型预测该距离下偏差，比较预测偏差与真实偏差：

```text
scale = target_distance / calibration_distance_m
pan_bias_deg(D)  = (actual_px - rf_px) * scale / px_per_deg
tilt_bias_deg(D) = -(actual_py - rf_py) * scale / px_per_deg
```

如果多个距离明显对不上，再升级模型：

```text
distance_lut：距离标定表 + 插值
```

只有当多距离实测明确表现为“距离越近角度偏差越大”时，才重新评估旧的 `lateral_offset` 或 `hybrid`：

```text
hybrid：固定角度项 + 距离比例项 / 物理横移项
```

## 十七、风险与注意事项

1. **偏差来源可能不只一种**  
   当前第一版按“像素偏差随距离等比例缩放”建模，可能混合了天线极化、主瓣偏斜、安装角度、多径等因素。后续需要用不同距离样本验证；如果实测不符合比例关系，再升级为 `distance_lut` 或混合模型。

2. **不要让前端承担坐标系复杂度**  
   原 `position` 字段继续给前端使用，新增 `rf_position` 仅调试。

3. **精度按云台能力收敛**  
   当前偏差计算最终保留到小数点后一位，避免输出云台无法执行的无效小数。中间计算可用浮点数，但写入命令、RF 点位换算和 Redis 调试字段前要统一到 0.1°。

4. **距离缺失时要有明确策略**  
   扫描类任务如果 Redis `gimbal:default_config.target_distance` 缺失，必须禁用天线偏差补偿。用户指定点位类接口如果请求体 `target_distance` 缺失，也必须禁用本次点位纠偏。不要使用 `标定距离米` 或其他默认值替代运行距离，避免在未知距离下误修正。

5. **范围转换和结果转换方向不能混淆**

```text
前端 visual -> 后端 RF 扫描：减偏差
后端 RF 结果 -> 前端 visual 展示：加偏差
```

### 18.3 必须实现的模型

```text

dy = -51

```

第一版模型必须按距离比例缩放像素偏差：

```text
scale = target_distance / calibration_distance_m
dx_scaled = dx * scale
dy_scaled = dy * scale

tilt_bias_deg = -dy_scaled / px_per_deg
```

等价写法：

```text
pan_bias_deg(D)  = pan_bias_deg_220  * D / calibration_distance_m
tilt_bias_deg(D) = tilt_bias_deg_220 * D / calibration_distance_m
```

最终输出按当前约定保留到小数点后一位：

```text
D = 220m -> pan_bias_deg = +0.6, tilt_bias_deg = +1.4
D = 5m   -> pan_bias_deg = +0.0, tilt_bias_deg = +0.0
```

`最大补偿角度` 可以继续作为防御性 clamp，但在 5m 场景下绝不能因为公式导致偏差被 clamp 到 5°。
