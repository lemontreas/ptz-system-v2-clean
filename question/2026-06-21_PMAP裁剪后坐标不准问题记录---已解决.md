## 当前结论

后端和设备端已经从旧 CSV `pixel_map.pmap` 切到了二进制 PMAP 文件格式，文件头验证为 `PMAP0001`。

2026-06-21 最终验证已定案：问题已经不再是“CSV 伪装成 `.pmap`”，也不是 PTO、人工参考点、最终 crop 尺寸或 `.pmap` 二进制布局问题，而是 Linux 设备上 `nona.cpp` 中新增的 PMAP 生成逻辑没有对齐 nona 实际 remap 时使用的 transform 方向。

旧 PMAP 生成逻辑是在 `nona.run()` 完成后，额外遍历 panorama 像素并调用：

```cpp
HuginBase::Nona::SpaceTransform transf;
transf.createInvTransform(srcPanoImg, opts);
transf.transformImgCoord(sx, sy, panoX, panoY)
```

这套外挂重算逻辑使用了错误的 `createInvTransform()`，并手动传入 `x+0.5-center` 坐标，导致左右边缘来源图错配和 `coverage_count=0`。

已验证通过的修复是对齐 nona 正常 remap 路径：

```cpp
HuginBase::Nona::SpaceTransform transf;
transf.createTransform(pano, imgNr, opts, vigra::Diff2D(srcSize.width(), srcSize.height()));

const double panoX = static_cast<double>(fullPx);
const double panoY = static_cast<double>(fullPy);
transf.transformImgCoord(sx, sy, panoX, panoY);
```

关键原因：`RemappedPanoImage.h` 中 nona 正常 remap 使用 `createTransform(...)`，并把 full canvas 像素坐标 `x/y` 直接传给 `transformImgCoord()`；`transformImgCoord()` 内部已经做 `+0.5/-0.5` 像素中心处理，不应在 PMAP 分支外部再减全景中心。

正式验证结果：

- `createInvTransform + x+0.5-center`：失败，左侧 `shot_001` 被映射到 `shot_005`，右侧 `shot_005` 大量 `coverage_count=0`。
- `createInvTransform + x+0.5`：失败，owner 仍明显左右错配。
- `createTransform + fullPx/fullPy`：通过，7 组人工参考点全部 `owner_match=True`，source 坐标误差约 `0.1~1.1px`。

## 为什么“nona 自己拼接肯定知道”，但现在 PMAP 还会错

nona 真正拼接时，大致流程是：

```text
源图 -> remap 到全景坐标下的 layer -> enblend 融合 layer -> 最终图
```

nona 在 remap 每张源图时，内部确实已经知道：

```text
目标全景像素 -> 源图 sx/sy -> 采样颜色
```

但是当前 PMAP 改法没有在这个内部采样位置直接记录，而是在 `nona.cpp` 的高层流程结束后，另外写循环重新计算。这个重算逻辑并不保证和真实 remap loop 的坐标输入完全一致。

因此现在的问题不是“nona 不知道”，而是“我们没有在 nona 知道的那个位置直接把信息捞出来”。

## 当前 C++ PMAP 生成逻辑的风险点

裁剪版里大致逻辑是：

```cpp
const vigra::Rect2D roi = opts.getROI();
int cropLeft = roi.left();
int cropRight = roi.right();
int cropTop = roi.top();
int cropBottom = roi.bottom();

for (int py = 0; py < pmapH; ++py) {
    for (int px = 0; px < pmapW; ++px) {
        const int fullPx = cropLeft + px;
        const int fullPy = cropTop + py;

        const double panoX = static_cast<double>(fullPx) + 0.5 - panoCenterX;
        const double panoY = static_cast<double>(fullPy) + 0.5 - panoCenterY;

        transf.transformImgCoord(sx, sy, panoX, panoY);
    }
}
```

高度怀疑 `transformImgCoord()` 在当前上下文期待的坐标不是 `fullPx + 0.5 - center`。

之前 full canvas 版本出现 `source_y` 跑到顶部；裁剪版本出现很多有效视觉区域 `coverage_count=0`，都指向这个坐标系问题。

### 当前锁定的问题与旧假设修正

结合 `hugin/nona_pixelmap/实际操作记录_nona改造完整流程.md` 与 `hugin/nona_pixelmap/nona_pixelmap_source_notes.md` 可确认：此前手动改造 nona 时，把以下判断写成了已验证结论：

```text
transformImgCoord() 不能直接传左上角像素坐标，必须传 px + 0.5 - panoCenterX / py + 0.5 - panoCenterY。
```

但 2026-06-21 的 7 组人工参考点测试已经反证该结论：

- PTO math 与人工参考点几乎完全一致；
- 当前 PMAP 左侧 `shot_001` 参考点被错误映射到 `shot_005`；
- 当前 PMAP 右侧 `shot_005` 参考点大量 `coverage_count=0`；
- 因此问题不在 `.pmap` 二进制格式、人工坐标、PTO、最终图坐标或源图裁剪，而在 `nona.cpp` 里外挂重算 PMAP 时传给 `SpaceTransform::transformImgCoord()` 的目标坐标系。

最终根因：

```text
PMAP 外挂分支误用了 createInvTransform(srcPanoImg, opts)，并手动传入 x+0.5-center。
nona 正常 RemappedPanoImage 路径使用 createTransform(...)，并把 full canvas 像素坐标直接传给 transformImgCoord()。
```

已通过修正项是：

```cpp
// 错误
transf.createInvTransform(srcPanoImg, opts);
panoX = fullPx + 0.5 - panoCenterX;
panoY = fullPy + 0.5 - panoCenterY;

// 正确
transf.createTransform(pano, imgNr, opts, vigra::Diff2D(srcSize.width(), srcSize.height()));
panoX = fullPx;
panoY = fullPy;
```
