## 问题概述

- 时间：2026-06-26 15:32
- 状态：排查中
- 影响范围：全景图点击定位、PMAP 查表后的 source 像素转云台角度、点击后移动到目标点的视觉对准验证。
- 简单描述：用户在全景图点击像素，经 PMAP 查到来源图像素后，用像素到角度公式计算 pan/tilt 并移动云台；实测发现同一来源图/画布中不同像素的偏差不是固定值，可能水平差 1°，也可能水平差 3-4°，垂直方向也有约 1° 偏差。
- 关联计划：`plan/全景拼接与像素查表优化方案.md`
- 关联 Todo：Hugin/PMAP 坐标闭环与真机点击移动验证。

## 现场现象

- 触发条件：全景图点击像素后，PMAP 已能查到对应来源图 owner/source 像素；后端再把 source 像素转换为云台角度并执行移动。
- 期望结果：如果 source 像素到角度的模型正确，移动后前端实时流中心十字应与目标物体对准；同一来源图不同像素的误差应符合相机模型，不应出现无规律的大幅变化。
- 实际结果：移动后目标与中心十字存在非固定偏差；不同像素点偏差量不同，部分点可达水平 3-4°。
- 相关命令 / 接口 / 日志：

```text
POST /api/v1/camera/coordinate_convert
mode=panorama 时优先走 PMAP：
  panorama pixel -> pmap owner/source_x/source_y -> source pixel -> pan/tilt

POST /api/v1/camera/capture
普通拍照保存的是 flip 后再 undistort 的图片，不是前端实时 raw 流。
```

## 排查记录

### 2026-06-26 15:32

- 操作：梳理 PMAP 与后续角度转换的边界。
- 观察：如果 PMAP 已保证“全景像素 -> 来源图/source 像素”正确，则后续主要应排查“source 像素 -> 云台角度”链路，而不是继续把问题归因于全景图像素本身。
- 结论：PMAP 先作为已解决前提处理；后续排查聚焦单张图坐标系、去畸变内参、拍摄姿态、云台物理移动验证。
- 下一步：先用单张图闭环排除坐标系问题，再回到全景链路。

### 2026-06-26 15:32

- 操作：核对当前代码中普通拍照与 Hugin source 图的图像处理流程。
- 观察：普通拍照接口保存流程为 `raw snapshot -> cv2.flip(im, -1) -> cv2.undistort(..., new_K) -> image_url`。Hugin 全景 source 图也基于去畸变图，并在拼接前做中心裁剪；裁剪函数会同步修正内参 `cx -= crop_x`、`cy -= crop_y`。
- 结论：前端实时流如果是 raw 图，不能直接拿 raw 流上的边缘像素坐标去套后端去畸变图的角度公式；必须用普通拍照返回的去畸变图上的同一真实目标像素，或对 raw 像素使用 `K + dist` 做等效去畸变。
- 下一步：确认用户当前边缘点输入来自去畸变图还是 raw 流；用户反馈当前使用的是去畸变图坐标，并且同一物体在 raw/undistorted 两张图上的位置差约 3-4 像素。

### 2026-06-26 15:32

- 操作：讨论前端中心十字与后端角度模型中心是否一致。
- 观察：用户已测试中心点，前端中心和实际后端模型中心基本一致，偏差不到 1-2 像素。
- 结论：中心十字/光轴中心错位不是当前主因，可以暂时排除。
- 下一步：进入边缘点排查，重点看去畸变/内参/标定残差是否导致非固定偏差。

### 2026-06-26 15:32

- 操作：梳理可能导致“非固定偏差”的假设树。
- 观察：
  - 去畸变/内参不一致会导致偏差随像素位置变化，尤其边缘和角落更明显。
  - 若 `source_x/source_y` 对应的是裁剪后的 source 图，角度公式必须使用裁剪后的 `new_K`。
  - 如果目标不够远，且相机光心与云台旋转中心不重合，移动后目标是否居中会依赖目标距离；这会影响真实移动验证，但不会影响纯像素射线角计算。
  - 当前 Hugin 默认 `HUGIN_GEOMETRY_MODE=optimized`，PMAP 基于优化后的 PTO 生成；后续 source 像素转角度时使用的是 session 中的拍摄 pan/tilt，而不是优化后 PTO 的每图 yaw/pitch/roll。若两套姿态不一致，可能导致不同 owner/不同位置出现非固定偏差。
  - 云台到位误差也可能造成偏差，但中心已对齐且 PTZ 控制内部默认高精度阈值约 0.1°，单独解释 3-4° 不够充分。
- 结论：当前优先级建议为：单张图去畸变/内参一致性 > 真实移动的距离/视差影响 > Hugin optimized 姿态与 session 拍摄姿态不一致 > 云台短距离移动误差。
- 下一步：按“一个一个排查”的方式推进，不一次混测多个变量。

### 2026-06-26 16:06

- 操作：根据用户补充的现场测试结果，先排除三类可疑因素：硬件云台重复定位、接口多次像素/角度转换漂移、去畸变引入的大幅像素偏差。
- 观察：
  - 硬件设备：用户让云台在两个角度之间来回循环 20 次以上，最终移动位置相同。
  - 接口转换：用户使用同一张图片的多个像素，反复执行像素转角度、角度再反转像素，发现数值一直不变。
  - 去畸变：用户对比去畸变单张图与未去畸变单张图中同一物体的像素位置，像素差约 2-3 px，量级很小，不足以解释当前 3-4° 级别的实测偏差。
- 结论：
  - 暂时排除云台硬件重复定位误差或累积漂移作为主因。
  - 暂时排除接口多次转换导致像素和角度数值漂移作为主因。
  - 暂时排除去畸变本身造成大幅偏差作为主因。
- 下一步：继续聚焦 source 像素到云台角度模型、拍摄姿态来源、Hugin optimized 姿态与 session 记录姿态是否一致，以及真实目标距离/相机光心与云台旋转中心偏移等因素。

## 当前卡点

- 卡在哪里：硬件云台重复定位、接口多次转换数值漂移、去畸变造成大幅像素偏差已先排除；剩余重点仍在 source 像素到云台角度模型、拍摄姿态来源和真实几何验证。
- 缺少什么信息：

  - 测试目标大概距离设备多远。
  - 大偏差点对应的 PMAP `owner_index/source_px/source_py/center_norm/coverage_count`。
- 推荐下一步：
  1. 只用普通拍照返回的去畸变图做 `mode=single` 边缘点测试，移动后看目标是否进中心。
  2. 若单图边缘仍有明显非固定偏差，重点查 source 像素到角度公式使用的内参、FOV、中心点和拍摄姿态。
  3. 检查同一偏差点对应的 PMAP `owner_index/source_px/source_py/center_norm/coverage_count`，确认 source 像素确实落在预期来源图和预期目标上。
  4. 若单图准确但全景 PMAP 点不准，再对比 `HUGIN_GEOMETRY_MODE=ptz_locked` 与默认 `optimized` 生成的全景/PMAP。
  5. 若纯角度模型准确但移动后仍偏，继续验证目标距离、相机光心与云台旋转中心偏移造成的视差影响。
