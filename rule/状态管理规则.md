# 状态管理规则

**适用范围**：`web_server.py`、`ptz_control.py`、`capture_worker.py`、前端对接文档、状态流测试。

这份规则用于后续 AI 修改和审核状态相关代码。凡是涉及全面扫描、定位扫描、抓包、stop、`/api/v1/ptz/status` 的修改，都必须先读本文件。

---

## 1\. 核心原则

1.  `/api/v1/ptz/status` 是前端状态主入口。
2.  顶层 `status.state` 是云台运动状态，不是扫描任务生命周期。
3.  全面扫描生命周期只看 `status.full_scan.state` / `status.full_scan.terminal`。
4.  定位扫描生命周期只看 `status.location_scan.state` / `status.location_scan.terminal`。
5.  抓包生命周期只看 `status.capture.running` / `status.capture.active`，不要用顶层 `state="IDLE"` 判断抓包结束。
6.  `capture_worker.py` 只维护 Redis `capture:running` 与 `capture:status`；`/api/v1/ptz/status` 由 `web_server.py` 聚合这些抓包字段后返回前端。不要在 `capture_worker.py` 里直接写 `ptz:current_status`。
7.  `phase` 只表示执行阶段，不表示任务是否仍在运行。
8.  `active` 是前端运行展示兼容字段；全面扫描收到 stop 后立即为 `false`，生命周期和启动门禁仍必须使用 `state` / `terminal` / `active_scan_id`。
9.  stop 是"请求已接收 -> 后端收尾 -> 终态"的两阶段过程，不是 API 返回即完全停止。
10. 全面扫描手动 stop 从 API 接收到 worker 发布终态采用统一的 2 秒端到端硬预算，不能给每个清理步骤分别累计 2 秒。
11. 全面扫描最终轮次结果写入 Redis 后，必须先基于该结果发布一致的终态与计数，再执行测试摘要、SQLite 历史快照等非关键后处理；这些旁路产物失败或阻塞不得让 `full_scan` 残留在 `running`。
12. Web stop 与 PTZ worker 对 `ptz:current_status` 的局部更新必须使用 Redis `WATCH/MULTI/EXEC` 乐观事务重试，禁止用无锁的“读取整份 JSON -> 修改 -> 整份写回”，否则旧 worker 快照可能偶发覆盖 `stopping/stopped`。状态查询发现 stop key 与状态 `scan_id` 匹配且 `active_scan_id` 已消失时，应幂等自愈为 `stopped/terminal=true` 并释放遗留项目门禁；如果没有匹配 stop key、`active_scan_id` 也不存在且 `history:current_project` 不再是同 scan_id 的 RUNNING/STOPPING 项目，`/api/v1/ptz/status` 必须把残留的 `full_scan.active=true/state=running` 投影为正常终态 `done/terminal=true`。
13. stop key 的换代权只属于新任务启动事务：必须原子执行“清理旧 stop -> 发布新 `active_scan_id` -> 命令入队”。所有 stop 检查函数只能读取并忽略不匹配信号，禁止删除；worker 接手后也不得再次清理 stop。任务首次观察到 stop 后必须在内存中锁存 reason，终态发布不得依赖 Redis key 最后仍存在。HTTP 对 `history:current_project=STOPPING` 超过2秒的同 scan_id 历史脏态必须自愈并按值条件释放 active ID。
14. PTZ 等待全面扫描点位通知时不得使用1秒级阻塞 BRPOP；必须使用不超过50ms的非阻塞轮询并在每轮优先检查 stop。HTTP 2秒 auto-heal 只作为异常兜底，正常 worker 收口必须先于兜底完成。

## 2\. 统一生命周期模型

扫描子对象必须遵守以下状态语义：

```
queued    等待 worker 接手
running   worker 正在执行
stopping  stop API 已收到，worker 或 capture_worker 正在收尾
stopped   手动停止后的终态
done      正常完成后的终态
error     异常终态
idle      未运行或空闲兼容态
```

推荐判断函数：

```typescript
function getScanLifecycle(scan?: {
  state?: string
  phase?: string
  scan_id?: string
  active?: boolean
  stop_requested?: boolean
  terminal?: boolean
}) {
  const state = scan?.state || (scan?.active ? "running" : "idle")
  const terminal =
    scan?.terminal === true ||
    ["stopped", "done", "error", "idle"].includes(state)

  return {
    state,
    phase: scan?.phase,
    scanId: scan?.scan_id,
    isActive: ["queued", "running", "stopping"].includes(state),
    isStopping: state === "stopping" || scan?.stop_requested === true,
    isTerminal: terminal,
    canStop: state === "queued" || state === "running",
    canStart: terminal,
  }
}
```

后端写状态时必须保证：

```
state in queued/running           -> active=true
full_scan.state=stopping          -> active=false（前端立即退出运行展示）
其他扫描 state=stopping           -> active 按各自兼容契约
state in stopped/done/error/idle  -> active=false
terminal=true                     -> state 必须是 stopped/done/error/idle 之一
stop_requested=true               -> 通常只出现在 stopping 收尾期
终态后 stop_requested             -> false 或不存在
```

## 3\. `/api/v1/ptz/status` 字段契约

`full_scan` 子对象至少应支持：

```
{
  "active": true,
  "state": "running",
  "phase": "coarse",
  "scan_id": "full_1710000000000",
  "stop_requested": false,
  "stop_requested_at": null,
  "terminal": false,
  "reason": null,
  "round": 1,
  "round_id": "full_1710000000",
  "current_point": 1,
  "total_points": 10,
  "remaining_seconds": 120
}
```

`location_scan` 子对象至少应支持：

```
{
  "active": true,
  "state": "running",
  "phase": "channel_probe",
  "scan_id": "location_1710000000000",
  "stop_requested": false,
  "stop_requested_at": null,
  "terminal": false,
  "reason": null,
  "current_point": 1,
  "total_points": 10,
  "remaining_macs": ["aa:bb:cc:dd:ee:ff"]
}
```

`capture` 子对象至少应支持：

```
{
  "active": true,
  "running": true,
  "phase": "capturing",
  "status": "running"
}
```

不要让前端依赖 Redis 内部 key。前端只能依赖 HTTP 状态字段。

## 4\. stop API 语义

stop API 返回成功只代表"停止请求已接收"，不代表任务已完全停止。

### 4.1 全面扫描 stop

接口：

```
POST /api/v1/ptz/full_area_scan/stop
```

API 层必须立即：

```
full_scan.state = "stopping"
full_scan.stop_requested = true
full_scan.stop_requested_at = now
full_scan.active = false
full_scan.terminal = false
```

worker 收尾完成后才写：

```
full_scan.state = "stopped"
full_scan.active = false
full_scan.terminal = true
full_scan.reason = "manual_stop"
full_scan.stop_requested = false
```

### 4.2 定位扫描 stop

接口：

```
POST /api/v1/ptz/location_scan/stop
```

API 层必须立即：

```
location_scan.state = "stopping"
location_scan.stop_requested = true
location_scan.stop_requested_at = now
location_scan.active = true
location_scan.terminal = false
```

worker 收尾完成后才写：

```
location_scan.state = "stopped"
location_scan.active = false
location_scan.terminal = true
location_scan.reason = "manual_stop"
location_scan.stop_requested = false
```

### 4.3 禁止事项

不要在 stop API 中直接把任务写成完全终态，除非 worker 已经确认没有任何收尾任务。全面扫描的 `active=false` 仅表示前端立即退出运行展示，不等于终态。

禁止：

```
stop API 直接 terminal=true
stop API 直接删除 active_scan_id 后不保留 stop 标志 TTL
前端看到 stop 返回就当已停止
```

## 5\. scan\_id 隔离规则

每个任务启动时生成唯一 `scan_id`：

```
全面扫描：full_{timestamp_ms}
定位扫描：location_{timestamp_ms}
```

启动时写：

```
full_scan:active_scan_id = scan_id
location_scan:active_scan_id = scan_id
```

所有 stop key 和 capture 命令必须携带或匹配 `scan_id`。

### 5.1 stop key 格式

全面扫描：

```
{
  "scan_id": "full_1710000000000",
  "reason": "manual_stop",
  "ts": 1710000000.0
}
```

定位扫描：

```
{
  "scan_id": "location_1710000000000",
  "ts": 1710000000.0
}
```

### 5.2 审核要求

1.  worker 只响应 scan\_id 匹配的 stop。
2.  scan\_id 不匹配的旧 stop 必须忽略或清理，不能误伤新任务。
3.  capture\_worker 收到旧命令时，如果 `active_scan_id` 不存在或不匹配，必须跳过或返回 `status="stopped"`。
4.  `scan_id` 由后端生成，前端不要生成或传入。

## 6\. capture\_worker 任务级取消规则

扫描任务 stop 后，已经派发给 `capture_worker` 的旧抓包/探测命令也必须尽快停止。

### 6.1 全面扫描 capture 命令

`discover_macs_for_full_scan` 必须检查：

```
capture:stop
multi_scan:stop_full_area_scan
full_scan:stop JSON，且 scan_id 匹配
full_scan:active_scan_id 消失或不匹配
```

必须在以下位置检查取消：

```
收到命令后
setup_monitor_mode_once 前
setup_monitor_mode_once 后
每个信道配置循环开始前
信道切换前
信道切换后
sniff 前/中/后
解析数据包前
发送 notify 前
```

取消时必须写 notify：

```
{
  "status": "stopped",
  "reason": "manual_stop",
  "point_id": "...",
  "scan_id": "...",
  "mac_count": 0,
  "macs": {}
}
```

### 6.2 定位扫描 capture 命令

`detect_channels` 和 `scan_at_point` 必须使用统一取消判断，检查：

```
capture:stop
location_scan:stop JSON，且 scan_id 匹配
命令自带 stop_key
location_scan:active_scan_id 消失或不匹配
```

必须在以下位置检查取消：

```
收到命令后
setup_monitor_mode_once 前
setup_monitor_mode_once 后
每个信道/配置循环前
信道切换前
信道切换后
sniff 前/中/后
```

### 6.3 sniff 规则

不要用长时间阻塞的 `sniff(timeout=long_time)` 且只依赖 `stop_filter`。

应使用短片轮询：

```
sniff slice timeout <= 0.5s
每个 slice 前后都检查 stop
没有包时也必须能及时退出
```

全面扫描手动停止时，不再等待当前点位的抓包结果通知；迟到结果依靠 `scan_id` 隔离并丢弃。配置会话清理必须共享 API stop 写入时间计算出的 2 秒截止时间。清理未按期确认时，请求 `run_manager.py` 只重启 `capture_worker.py`，不得连带重启 PTZ 和 Web 服务。

### 6.4 stop key 生命周期

worker 结束任务时：

```
可以删除 active_scan_id，让新任务能启动
不要立刻删除 stop key
stop key 应保留 TTL 约 120s，给 capture_worker 消化旧队列
```

正确：

```python
r.delete("full_scan:active_scan_id")
if r.get("full_scan:stop") and r.ttl("full_scan:stop") == -1:
    r.expire("full_scan:stop", 120)
```

错误：

```python
r.delete("full_scan:active_scan_id", "full_scan:stop")
```

## 7\. capture 状态规则

`capture` 是独立生命周期，和扫描生命周期分开。

前端判断抓包是否运行：

```
status.capture.running == true
或 status.capture.active == true
```

不要用：

```
status.state == "IDLE"
```

因为云台可以空闲，但抓包仍在运行。

### 7.1 扫描 stop 和 `/capture/stop`

全面扫描 stop：

```
前端不要为了停止全扫内部抓包额外调用 /api/v1/capture/stop
后端通过 full_scan stop key 和任务级取消通知 capture_worker
capture_worker 通过 full_scan:stop / multi_scan:stop_full_area_scan / active_scan_id 校验退出旧任务
```

定位扫描 stop：

```
前端不要为了停止定位内部抓包额外调用 /api/v1/capture/stop
定位 stop API 会设置 location_scan:stop 和 capture:stop
capture_worker 通过 location_scan:stop / capture:stop / active_scan_id 校验退出旧任务
```

独立抓包或未设置 `capture_time_limit` 的 `capture_at_best`：

```
前端需要调用 POST /api/v1/capture/stop?timeout=15
停止依据是 capture.running=false
```

### 7.2 启动扫描前

前端或 API 层应避免在独立抓包仍运行时启动扫描：

```typescript
const canStartScan =
  getScanLifecycle(status.full_scan).isTerminal &&
  getScanLifecycle(status.location_scan).isTerminal &&
  status.capture?.running !== true
```

如果是独立抓包或持续 `capture_at_best`，先走 `/api/v1/capture/stop?timeout=15`。

## 8\. 阶段 phase 规则

`phase` 只做进度展示，不做生命周期判断。

### 8.1 全面扫描常见 phase

```
queued
coarse
fine_1
fine_2
deviation_a
deviation_b
waiting_interval
completed
stopped
```

注意：

```
启动接口 scan_plan.phase 仍可能是 precheck/guard/work
实时运行阶段必须看 status.full_scan.phase
不要用 scan_plan.phase 判断实时阶段
```

### 8.2 定位扫描常见 phase

```
queued
channel_probe
fast_verify
coarse
mid
fine
locating
capturing
waiting
idle
stopped
```

注意：

```
phase="idle" 不等于生命周期运行中
生命周期仍看 location_scan.state / terminal
```

## 9\. 前端契约

前端必须：

1.  轮询 `/api/v1/ptz/status` 作为主入口。
2.  全面扫描生命周期读 `full_scan.state` / `full_scan.terminal`。
3.  定位扫描生命周期读 `location_scan.state` / `location_scan.terminal`。
4.  抓包生命周期读 `capture.running` / `capture.active`。
5.  stop 后显示"正在停止"，禁用 stop 按钮。
6.  stop 后不要立即开放 start。
7.  只有对应子对象终态后才允许重新开始。
8.  phase 只做阶段展示。
9.  不生成、不传入 `scan_id`。
10.  不根据 `active=false` 单独判断能否开始。

前端禁止：

```
用顶层 status.state 判断扫描生命周期
用 phase 判断任务是否结束
stop API 返回后立即显示已停止
stop API 返回后立即允许新任务
为了停止扫描内部抓包额外调用 /capture/stop
```

## 10\. 后端修改审核清单

状态相关改动必须逐项检查：

*   是否仍以 `/api/v1/ptz/status` 为前端主入口？
*   是否保留 `full_scan.state` / `location_scan.state` 生命周期语义？
*   是否没有把顶层 `state` 当扫描生命周期？
*   stop API 是否立即写 `stopping`，但不直接写终态？
*   全面扫描 stop API 是否立即写 `active=false`，且迟到状态 patch 不会复活为 `running/active=true`？
*   worker 终态是否在 API stop 后 2 秒内写 `terminal=true`、`active=false`？
*   stop key 是否带 `scan_id`？
*   worker 是否只响应 scan\_id 匹配的 stop？
*   capture 命令是否携带 scan\_id？
*   capture\_worker 是否在 setup 前后、切信道前后、sniff 前后检查 stop？
*   capture\_worker 取消时是否写 `status="stopped"` notify？
*   capture\_worker 是否只写 `capture:running` / `capture:status`，并由 `/api/v1/ptz/status` 合并展示？
*   任务结束时是否保留 stop key TTL，避免 capture\_worker 读不到？
*   新任务启动时是否清理会误伤新任务的旧 `capture:stop`？
*   是否避免独立抓包运行时直接启动扫描？
*   文档是否同步更新？
*   状态流测试是否覆盖 start、stop、phase、terminal、no resurrection？

## 11\. 常见错误

### 错误 1：stop 后直接写 stopped

错误：

```python
scan["state"] = "stopped"
scan["active"] = False
scan["terminal"] = True
```

正确：

```python
# API 收到 stop
scan["state"] = "stopping"
scan["stop_requested"] = True
scan["active"] = False
scan["terminal"] = False

# worker 收尾完成
scan["state"] = "stopped"
scan["active"] = False
scan["terminal"] = True
```

### 错误 2：删除 stop key 太早

错误：

```python
r.delete("full_scan:active_scan_id", "full_scan:stop")
```

正确：

```python
r.delete("full_scan:active_scan_id")
if r.get("full_scan:stop") and r.ttl("full_scan:stop") == -1:
    r.expire("full_scan:stop", 120)
```

### 错误 3：capture\_worker 只在信道循环里检查 stop

错误：

```
setup_monitor_mode_once 很耗时，但 setup 前后不检查 stop
```

正确：

```
收到命令后检查
setup 前检查
setup 后检查
每个信道前后检查
sniff 前中后检查
```

### 错误 4：把 capture stop 当扫描状态

错误：

```
status.state="IDLE" -> 认为抓包停止
```

正确：

```
status.capture.running=false -> 抓包停止
```

### 错误 5：前端传 scan\_id

错误：

```
前端生成 scan_id 并传给后端
```

正确：

```
后端生成 scan_id
前端只读 scan_id 用于展示或调试，不参与控制
```

## 12\. 推荐测试场景

状态相关修改后至少测试：

1.  全面扫描启动后进入 `queued/running`。
2.  全面扫描 `coarse` 阶段 stop。
3.  全面扫描抓包/信道扫描中 stop，确认 capture\_worker 不再继续完整信道列表。
4.  全面扫描 `waiting_interval` 阶段 stop。
5.  定位扫描启动后进入 `queued/running`。
6.  定位扫描 `channel_probe` 阶段 stop。
7.  定位扫描 `coarse/mid/fine` 阶段 stop。
8.  定位扫描 `capturing` 阶段 stop。
9.  stop 后不出现 `stopped -> running` 复活。
10.  stop 后马上启动另一类扫描，确认旧命令不会误伤新任务。
11.  独立 `capture_at_best` 持续抓包时，`capture.running=true`，顶层 `state=IDLE` 不应误判为空闲。
12.  `/api/v1/capture/stop?timeout=15` 后 `capture.running=false`。
13.  全面扫描 stop 后立即观察到 `active=false/state=stopping`，并在 2 秒内到达 `terminal=true`。
14.  抓包会话停止超时时，只重启 `capture_worker`，Web/PTZ 不退出。

## 13\. 给后续 AI 的硬性要求

如果要改状态相关代码，必须在回复或提交说明中回答：

1.  改动影响哪个生命周期：`full_scan`、`location_scan`、`capture`、顶层 PTZ？
2.  是否改变 `/api/v1/ptz/status` 字段？
3.  stop API 是否仍然先进入 `stopping`？
4.  worker 是否仍然负责最终 `terminal=true`？
5.  scan\_id 隔离是否仍然成立？
6.  capture\_worker 是否仍能取消已派发旧任务？
7.  前端文档是否需要同步？
8.  状态流测试是否需要同步？

如果回答不了，不要改。
