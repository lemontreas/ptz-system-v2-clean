# 设备测试与部署指南（测试/部署 AI 专用）

> **何时读本文件**：收到"测试/部署"类任务时读取。开发 AI 做纯代码修改时不需要读。
> **更新规则**：设备环境变化（IP、路径、服务名等）时必须同步更新本文件。

---

## 1. 角色边界与权限

### 1.1 职责范围

测试/部署 AI 负责：

- 部署代码到测试设备
- 真实设备环境验证
- Redis 状态检查
- 日志检查
- API 检查
- scan_id 隔离验证
- 两阶段停止语义验证
- 旧流程兼容性验证
- 测试通过后的精确 Git 保存
- 测试失败后的现场保留和失败分析

测试/部署 AI **不负责**：

- 设计新功能
- 修改业务代码
- 顺手修复代码
- 扩展当前 Todo
- 提前实现后续 Todo
- Git push
- 使用 `git add .`
- 覆盖无关工作区改动

### 1.2 禁止执行的命令

无论任何情况，未经用户单独明确确认，禁止执行：

- `git reset` / `git checkout` / `git clean` / `git restore`
- `git add .` / `git push`
- `del /s` / `rmdir /s` / `Remove-Item -Recurse` / `rm -rf`
- `scp` / `rsync` / `ssh 192.168.8.239`（未经确认的部署命令）
- 服务重启命令（未经确认）
- 修改 Redis 的命令（未经确认）
- 清空日志、清空缓存、清空数据库的命令

### 1.3 允许执行的安全命令

允许执行：

- `git status` / `git diff` / `git diff -- 文件名` / `git log --oneline -n 5`
- `dir` / `ls` / `type` / `Get-Content` / `Select-String`
- 只读搜索命令、只读查看代码
- 项目内必要的静态检查命令

### 1.4 高风险操作停止规则

遇到以下操作，必须停止并询问用户：

- 部署
- 连接目标设备
- 修改 Git 历史
- 删除/覆盖文件
- 批量格式化、大范围重构
- 修改 Redis
- 修改前端/API 契约
- 修改测试设备配置

### 1.5 工作区保护规则

必须保留工作区原有改动。禁止：

- reset / checkout / clean / restore
- 覆盖用户未提交文件
- 把无关文件一起提交

如果发现工作区已有无关 diff，必须只区分和报告，不得处理它们。

## 7. 部署流程（标准步骤）

### 7.1 部署前检查（10 项）

1. 当前 `git status`
2. 当前 `git diff`（逐文件）
3. 本次改动文件是否与开发 AI 报告一致
4. 是否存在无关修改
5. 是否存在未提交但不属于本次测试的改动
6. agent.md / AGENT.md 是否允许当前部署方式
7. 当前服务部署方式
8. 当前回退方式
9. 当前测试是否需要保护 Redis / scan_id / stop 语义
10. 是否存在会影响测试可信度的环境问题

## 8. 必须覆盖的验证项（10 项）

每次测试必须覆盖以下所有项目。如果某项与当前原子步骤无直接关系，也要做基本回归或说明"不涉及专项变更，已做基础确认"。

### 8.1 服务启动验证

- 服务能正常启动
- 无启动异常、无明显 Traceback
- 相关端口/API 可访问
- 服务状态正常

### 8.2 当前功能正向验证

- 按开发 AI 给出的当前原子步骤目标执行
- 验证新行为是否生效
- 验证返回结果是否符合成功标准
- 验证关键状态是否符合预期

### 8.3 当前功能边界验证

- 验证空值、缺省值
- 验证旧参数
- 验证异常输入
- 验证边界状态
- 只测当前步骤相关边界，不扩展到后续 Todo

### 8.4 API 验证

- 记录关键请求/响应
- 确认字段兼容
- 确认没有破坏前端/API 契约
- 确认旧调用方式仍可用（除非当前 Todo 明确要求改变）

### 8.5 Redis 验证

- 检查当前步骤涉及的 Redis key
- 检查 patch 更新是否正常
- 检查状态字段是否符合预期
- 检查是否存在脏状态或状态覆盖
- 不涉及 Redis 专项变更时，也要说明"已做基本状态回归"

### 8.6 scan_id 隔离验证

- 验证不同 scan_id 不互相污染
- 验证旧 scan_id 数据不会错误影响新扫描
- 验证当前 scan_id 状态更新正确
- 不涉及 scan_id 专项变更时，也要说明"已做基本回归"

### 8.7 两阶段停止语义验证

- 验证停止请求 → 停止中 → 最终停止状态
- 验证停止不会造成脏状态或服务卡死
- 验证停止后状态仍可查询
- 不涉及停止语义专项变更时，也要说明"已做基本回归"

### 8.8 日志验证

- 查看服务日志和错误日志
- 记录关键日志片段
- 确认无新增严重异常
- 确认异常路径日志可定位
- 确认没有明显刷屏或循环报错

### 8.9 旧流程兼容性验证

- 验证当前 Todo 之前已经可用的主流程没有被破坏
- 验证旧入口路径没有断
- 验证旧参数行为仍兼容
- 只做必要回归，不测试后续未实现功能

### 8.10 异常场景验证

- 验证当前步骤最容易出错的路径
- 验证异常不会导致服务崩溃或状态卡死
- 验证 Redis 不会留下明显错误状态
- 验证日志能反映失败原因

## 14. 全面扫描验证技巧（2026-06-30 实测总结）

### 14.1 验证实际下发配置（最重要）

**正确方法**：读取扫描摘要文件中 `stages[0].configs`。

```bash
# 扫描停止或轮次完成后，查看最新摘要
ssh ultiwill@192.168.8.239 'ls -t /tmp/logs/full_scan_*summary*.json | head -1 | xargs /home/ultiwill/ptz_capture_system/ptz_env/bin/python -c "
import sys, json
with open(sys.argv[1]) as f:
    d = json.load(f)
for s in d.get(\"stages\", []):
    cc = s.get(\"config_count\")
    configs = s.get(\"configs\", [])
    chs = sorted(set(c.get(\"channel\") for c in configs))
    bws = sorted(set(c.get(\"bandwidth\") for c in configs))
    print(f\"config_count={cc}, channels={chs}, bandwidths={bws}\")
"'
```

**错误方法（不要用）**：
- ~~`redis-cli lpop capture:command_queue`~~ — 命令被 capture_worker 消费，竞争失败
- ~~`redis-cli monitor`~~ — 正则解析不可靠，且有性能影响
- ~~扫描结果中的 `macs` 信道~~ — 只显示**扫到 MAC 的信道**，不是**扫描的全部信道**

**摘要文件命名**：`/tmp/logs/full_scan_{round_id}_summary.json`

**注意**：摘要文件在轮次完成或扫描停止后才写入，运行中不存在。

### 14.2 验证 stop 终态（running → stopping → stopped）

**关键发现**：`ptz:current_status` 是 **JSON 字符串**（`redis-cli get`），不是 hash（不要用 `hget`）。

**验证步骤**：
1. 启动扫描，等进入 `running` + 至少一个点完成
2. 记录 stop 前完整状态
3. POST stop
4. 每 0.3-0.5 秒同时采集 HTTP 和 Redis 状态
5. 必须观察到中间态 `stopping/active=true/terminal=false`
6. 必须观察到终态 `stopped/active=false/terminal=true`
7. 终态到达后继续轮询 5+ 秒，确认不消失、不复活
8. 检查 `active_scan_id` 已清理

### 14.3 Redis 结构速查

| Key | 类型 | 说明 |
|-----|------|------|
| `ptz:current_status` | **JSON 字符串** | 包含 position/state/full_scan/location_scan/capture |
| `full_scan:active_scan_id` | string | 当前运行中的扫描 ID，终态后清空 |
| `full_scan:stop` | JSON string | stop 信号，含 scan_id/reason/ts，有 TTL ~120s |
| `full_scan:results` | JSON string | 当前轮次结果，含 results 子对象（逐点数据） |
| `capture:command_queue` | list | capture_worker 消费，不要试图 lpop 竞争 |

### 14.4 scan_id 隔离验证

1. 启动任务 A，记录 scan_id_A
2. 停止任务 A，确认终态
3. 启动任务 B（不同 wifi_mode 或范围），记录 scan_id_B
4. 确认 scan_id_A ≠ scan_id_B
5. 停止任务 B，确认旧 stop key 未误伤

## 15. 多次扫描测试的数据陷阱（2026-06-30 实测总结）

### 15.1 核心规则

**Redis `full_scan:results` 是单实例覆盖写，不保留历史。** 每次新扫描启动或 stop 后的结果写入都会覆盖前一次数据。

因此：**同一会话中执行多次扫描时，禁止从 Redis 读取前一次扫描的结果做判断。**

### 15.2 可靠数据源优先级

| 优先级 | 数据源 | 说明 |
|--------|--------|------|
| 1 | `/tmp/logs/full_scan_{round_id}_summary.json` | 轮次完成/停止后写入，包含 stages、configs、points，**不会被后续扫描覆盖** |
| 2 | `journalctl -u ptz_capture.service` | 实时日志，包含 MAC 发现、信道切换、偏差区执行等全过程 |
| 3 | `full_scan:results`（Redis） | **仅当次扫描有效**，下一次扫描启动后立即失效 |

### 15.3 典型错误场景

```text
❌ 错误流程：
1. 启动扫描 A，等待完成
2. 启动扫描 B（例如 stop 终态测试）
3. 从 Redis full_scan:results 读取扫描 A 的数据 → 读到的是扫描 B 的空/不完整数据
4. 得出"扫描 A 没有 MAC"的错误结论

✅ 正确流程：
1. 启动扫描 A，等待完成
2. 从 /tmp/logs/full_scan_{A_round_id}_summary.json 读取结果
3. 如需 MAC 详情，从 journalctl 按 scan_id 过滤
4. 再启动扫描 B 做其他验证
```

### 15.4 从日志提取 MAC 的命令

```bash
# 按 scan_id 过滤 MAC 发现日志
ssh ultiwill@192.168.8.239 "journalctl -u ptz_capture.service --since '2026-06-30 16:10:00' --until '2026-06-30 16:50:00' --no-pager" \
  | grep 'full_1782807014' | grep -E '🆕 新 MAC|🔄 更新' | head -20

# 统计唯一 MAC 数
ssh ultiwill@192.168.8.239 "journalctl -u ptz_capture.service --no-pager" \
  | grep 'full_1782807014' | grep '新 MAC' \
  | grep -oP '[0-9a-f]{2}(:[0-9a-f]{2}){5}' | sort -u | wc -l
```

### 15.5 summary 文件中偏差区字段说明

summary 文件的 `stages[].phase == "deviation_a"` 包含：

| 字段 | 说明 |
|------|------|
| `config_source` | `"target_seen_macs"` 表示配置来自目标区真实 MAC |
| `config_count` | 信道配置数（如 13 = 全部 5GHz HT20） |
| `configs` | 具体信道列表 `[{channel, bandwidth}, ...]` |
| `points[].layer` | 偏差层 1/2/3 |
| `points[].side` | 边方向 left/right/top/bottom |
| `points[].narrow_side` | 是否窄边（布尔值） |
| `skip_reason` | 非空表示偏差区被跳过及原因 |

## 16. 单信道快速测试模式（2026-06-30 实测确认）

### 16.1 适用场景

全面扫描正式任务需要遍历 13（单频）或 26（双频）个 HT20 配置，一轮耗时 10-20 分钟。**如果本次测试不涉及信道配置验证、MAC 发现完整性或白名单召回率**，应使用单信道测试模式将每点扫描时间从 10+ 秒降到约 1 秒，整轮扫描从 10-20 分钟降到约 1 分钟。

适用于：
- 路径生成验证（粗扫/细扫/偏差点位数量与分布）
- 状态机验证（running → stopping → stopped、terminal、scan_id 隔离）
- 新字段验证（evidence_role、sampling_complete、completion_reason）
- stop 响应验证
- 云台移动行为验证（直接移动、中继回退）

**不适用于**：
- 信道配置数量验证（需用正式模式确认 13/26 个 HT20）
- MAC 发现完整性验收
- 白名单最终效果验收

### 16.2 开启方式

设备端临时修改 `config.json`：

```bash
ssh ultiwill@192.168.8.239 'cd /home/ultiwill/ptz_capture_system && /home/ultiwill/ptz_capture_system/ptz_env/bin/python -c "

with open(\"config.json\", \"r\", encoding=\"utf-8\") as f:
    cfg = json.load(f)
cfg[\"全面扫描\"][\"允许单信道内部测试\"] = True
with open(\"config.json\", \"w\", encoding=\"utf-8\") as f:
    json.dump(cfg, f, ensure_ascii=False, indent=2)
print(\"允许单信道内部测试 =\", cfg[\"全面扫描\"][\"允许单信道内部测试\"])
"'
```

然后重启服务：

```bash
ssh ultiwill@192.168.8.239 "sudo systemctl restart ptz_capture.service"
```

### 16.3 请求方式

在正常全面扫描请求中加入 `test_channel`：


- `test_channel` 必须是正式频段列表中的信道（2.4GHz: 1-13, 5GHz: 36/40/44/48/52/56/60/64/149/153/157/161/165）
- `wifi_mode` 此时不决定配置列表，后端按 `test_channel` 自动识别频段
- 返回中包含 `internal_test: {single_channel: true, channel: xxx, bandwidth: "HT20"}`
- 每个点位只扫描 1 个信道，粗扫 29 点 + 细扫 21 点 + 偏差区 24 点 ≈ 1 分钟完成

## 17. 最小点位快速测试模式（2026-06-30 实测确认）
