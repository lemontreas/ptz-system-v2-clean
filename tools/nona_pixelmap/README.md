# Pure nona pixel-map workflow

This note is for the Linux runtime device. The current Windows workspace does
not contain Hugin source code, so the source change must be applied on the
device where Hugin can be built and tested.

Goal:

```text
panorama pixel (X, Y) -> source image index + source pixel (x, y)
```

This deliberately ignores enblend/enfuse color blending. The result is a
geometric mapping table produced at the nona remapping stage.

## 1. Prepare Hugin source on Linux

Run these commands on the Linux device:

```bash
which nona
nona --help | head -40
hugin_executor --version || true
```

Install build dependencies. On Debian/Ubuntu, start with:

```bash
sudo apt update
sudo apt build-dep hugin
sudo apt install build-essential cmake ninja-build mercurial git
```

Get source matching the runtime as closely as possible. If the device uses an
apt package, this is usually the safest first attempt:

```bash
mkdir -p ~/src/hugin-nona-map
cd ~/src/hugin-nona-map
apt source hugin
```

If `apt source hugin` is unavailable, fetch upstream source:

```bash
cd ~/src/hugin-nona-map
hg clone https://hg.code.sf.net/p/hugin/hugin hugin
```

Then enter the source directory:

```bash
cd ~/src/hugin-nona-map/hugin
```

If `apt source` created a versioned directory such as `hugin-2024.0.1`, enter
that directory instead.

## 2. Locate the nona remap code

Find the executable entry point and remap loop:

```bash
find . -iname '*nona*' -o -iname '*Stitcher*' -o -iname '*Remap*'
grep -R "createInvTransform\\|transformImgCoord\\|stitchPanorama\\|nona" -n src | head -80
```

Common places in Hugin source trees are:

```text
src/tools/nona.cpp
src/hugin_base/algorithms/nona/
src/hugin_base/nona/
```

The exact file varies by Hugin version. The place we need is the code that has
all of these available:

```text
panorama options / output size
current source image index
inverse transform from panorama pixel to source pixel
the per-pixel remap loop or an equivalent remap worker
```

## 3. Add a pixel-map option to nona

Add a command-line option:

```text
--pixel-map FILE
```

When present, nona should write a CSV file with this header:

```csv
pano_x,pano_y,image_index,source_x,source_y
```

Recommended behavior:

```text
one row per valid source candidate
coordinates are in the same cropped/final panorama space that the user sees
source_x/source_y are floating point source image coordinates
image_index is the zero-based Hugin image index
```

If the easiest insertion point sees the full uncropped canvas, also write this
metadata as comment lines before the CSV header:

```csv
# full_width=3046
# full_height=1325
# crop_left=128
# crop_top=0
# crop_right=2548
# crop_bottom=795
pano_x,pano_y,image_index,source_x,source_y
```

The test program in this folder accepts both with and without comment metadata.

## 4. Minimal C++ logic to add

Use `nona_pixelmap_source_notes.md` as the source-edit reference. The important
logic is:

```cpp
for each source image:
    create inverse transform for this image
    for each panorama pixel:
        if transform maps panorama pixel to source pixel:
            if source pixel is inside the source image:
                write pano_x,pano_y,image_index,source_x,source_y
```

If your insertion point already iterates destination pixels for each source
image, do not create a second loop. Write the row next to the existing sampling
decision instead.

## 5. Build only the needed tools

Example CMake build:

```bash
mkdir -p build-nona-map
cd build-nona-map
cmake .. -DCMAKE_BUILD_TYPE=Release -GNinja
ninja nona hugin_executor
```

The produced binary is usually one of:

```bash
./src/tools/nona --help
./bin/nona --help
```

Verify the new option exists:

```bash
./src/tools/nona --help | grep pixel-map
```

## 6. Run nona with pixel-map output

Use the final PTO generated by this backend, usually `05_output.pto`:

```bash
PTO=/home/ultiwill/ptz_capture_system/data/captures/<session>/hugin/pto/05_output.pto
OUT=/tmp/nona_map_test

mkdir -p "$OUT"
./src/tools/nona \
  --pixel-map "$OUT/pixel_map.csv" \
  -m TIFF_m \
  -o "$OUT/layer" \
  "$PTO"
```

If your modified nona outputs full-canvas coordinates, crop/resize must be
handled before comparing with the final displayed panorama. Prefer writing
cropped final coordinates directly if your source insertion point makes that
available.

## 7. Inspect and verify

Copy `nona_pixelmap_verify.py` to the Linux device or run it from this project:

```bash
python tools/nona_pixelmap/nona_pixelmap_verify.py info "$OUT/pixel_map.csv"
python tools/nona_pixelmap/nona_pixelmap_verify.py pano "$OUT/pixel_map.csv" 1200,300
python tools/nona_pixelmap/nona_pixelmap_verify.py source "$OUT/pixel_map.csv" 1 928.4,431.8
python tools/nona_pixelmap/nona_pixelmap_verify.py sample "$OUT/pixel_map.csv" --limit 20
```

Accuracy expectation:

```text
non-overlap area: one candidate, mapping should be stable
overlap area: multiple candidates can exist for the same panorama pixel
blend seam area: geometry is still valid, final RGB ownership is not claimed
```

## 8. Integrate after it passes

Once Linux verification passes, update this backend to call your custom nona
binary instead of system nona. Keep it explicit, for example:

```bash
export HUGIN_NONA_BIN=/home/ultiwill/bin/nona-pixelmap
```

Then make the backend call `${HUGIN_NONA_BIN:-nona}` when generating the
panorama and coordinate map.
